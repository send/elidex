## §0 Scope

B1.3-i wires the **CharacterData data-mutation methods** (`setData`/`appendData`/
`insertData`/`deleteData`/`replaceData`) to produce `MutationKind::CharacterData`
MutationRecords. Axes A1 (replaceData algorithm) / A2 (oldValue capture-timing) /
A3 (Comment branch) from the umbrella §1.

**OUT of scope (→ B1.3-ii)**: `splitText` records, Range text-splice records
(`deleteContents`/`extractContents`/`insertNode` step 7). B1.3-ii depends on the
`apply_replace_data` + `empty_record`-pub + `splice_utf16`-move + `get_char_data`-
re-export primitives this slice lands.

This is a **terminal per-PR slice** under the approved umbrella (CLAUDE.md base-case
clause) — single PR, this plan-review gates it.

## §1 Ideal anchor

CharacterData data writes decompose onto ONE canonical record-producing primitive
`apply_replace_data` (mirroring `apply_append_child` etc.), delivered through the
existing B1 chokepoint (`push_notify_record` → `drain_notify_records` at the
`invoke_dom_api` boundary → §4.3 microtask). All five methods are spec-defined as
`replace data(offset, count, data)` invocations (DOM §4.10), so they collapse to a
single producer with no per-method record logic. `oldValue` is captured by the
caller (the full pre-mutation data string) BEFORE the primitive mutates — faithful
to DOM §4.10 step 4 (queue record passing node's data) running before steps 5-7
(insert+remove).

### §2.1 `splice_utf16` → `elidex-ecs` (F3)

`splice_utf16` (`elidex-dom-api/char_data/char_data_handlers.rs:193`, `pub(crate)`)
is a pure `&str + offset + count + Option<&str> → String` function (encode_utf16 /
from_utf16_lossy only, no DOM/ECS dependency). `apply_replace_data` (in
`elidex-script-session`) needs the identical splice for the Comment branch. To
avoid a second copy (CLAUDE.md One-issue-one-way), **move** it to `elidex-ecs`
(dependency of both `elidex-script-session` and `elidex-dom-api`, no cycle).

**Full unification (F1 — not just a move)**: `EcsDom::replace_text_data`
(`dom/mod.rs:379`) ALREADY inlines the same `encode_utf16 → splice → from_utf16_lossy`
over `TextContent` (its doc at `:369` literally says it "mirrors `splice_utf16`").
A bare move would leave TWO splice impls in the same `elidex-ecs` file = strangler
residue, not One-issue-one-way. So after the move, **refactor `replace_text_data`
to delegate its string-building to `splice_utf16`**: `replace_text_data` reads the
current `TextContent`, computes `clamped_count = (offset+count).min(len) - offset`
locally (still needed for the `MutationEvent::ReplaceData` payload), calls
`splice_utf16(&tc.0, offset, count, Some(replacement))` for the new string, writes
it back, then fires the event. This collapses the duplicated `Vec<u16>` splice into
the single canonical helper — the actual ideal, not a relocation.

- New home: `crates/core/elidex-ecs/src/dom/mod.rs` (1147 LoC — already >1000; see
  §9 note) or a small `text_util` submodule there, as `pub fn splice_utf16` (NEW
  location — moved from elidex-dom-api). `dom::splice_utf16` (NEW) is acceptable.
- **Unit coverage**: there is NO `splice_utf16` unit test in elidex-dom-api today
  (the `*surrogate_pair*` tests are JS-integration tests in
  `elidex-js/.../tests_character_data.rs`, exercising substringData/splitText
  end-to-end, NOT `splice_utf16` directly — F6). The relocated fn would have no
  direct coverage in its new home, so **author a fresh `splice_utf16` unit test in
  elidex-ecs** (coverage-with-code): mid-splice, append-shape, delete-shape,
  surrogate-pair lossy → U+FFFD. The end-to-end JS tests continue to cover the
  Text/Comment splice paths transitively.
- `elidex-dom-api` callers (`splice_char_data` Comment fallback) re-point to
  `elidex_ecs::...::splice_utf16`. The `debug_assert!(offset <= len)` precondition
  is preserved.

## §3 `apply_replace_data` (new primitive)

Location: `crates/script/elidex-script-session/src/mutation/mod.rs`, beside
`apply_set_text` (the existing single CharacterData record producer for
`textContent`).

```rust
/// DOM §4.10 "replace data" — record-producing primitive.
///
/// `old_value` MUST be the entity's FULL data string captured by the caller
/// BEFORE this call. DOM §4.10 "replace data" step 4 queues the characterData
/// mutation record passing node's data as the oldValue argument BEFORE steps 5-7
/// mutate the data; "queue a mutation record" (§4.3.2 step 3.3) then exposes that
/// oldValue only to observers with characterDataOldValue:true.
///
/// Precondition: `offset <= utf16_len(entity's data)` (caller validates and raises
/// IndexSizeError; this fn `debug_assert!`s via `replace_text_data` / the splice).
///
/// Returns a single `CharacterData` record. Text / CDATASection route through
/// `EcsDom::replace_text_data` (fires `MutationEvent::ReplaceData` for live-range
/// fixup); Comment routes through `CommentData` + `splice_utf16` + `rev_version`
/// (no live-range event — §4.10 boundary steps 8-11 apply to all CharacterData,
/// but elidex's range-fixup hook is Text/CDATASection-only by construction).
pub fn apply_replace_data(
    dom: &mut EcsDom,
    entity: Entity,
    offset: usize,
    count: usize,
    replacement: &str,
    old_value: String,
) -> MutationRecord {
    // Text / CDATASection branch: replace_text_data returns Some(new_len) iff the
    // entity has a TextContent component, doubling as the branch discriminator.
    if dom.replace_text_data(entity, offset, count, replacement).is_some() {
        return MutationRecord {
            old_value: Some(old_value),
            ..empty_record(MutationKind::CharacterData, entity)
        };
    }
    // Comment branch: splice CommentData in place via the shared helper.
    if let Ok(mut cd) = dom.world_mut().get::<&mut CommentData>(entity) {
        let spliced = splice_utf16(&cd.0, offset, count, Some(replacement)); // moved to elidex-ecs
        spliced.clone_into(&mut cd.0);
        drop(cd);
        dom.rev_version(entity);
        return MutationRecord {
            old_value: Some(old_value),
            ..empty_record(MutationKind::CharacterData, entity)
        };
    }
    // Neither branch: entity is not CharacterData. Callers brand-check + validate
    // before invoking, so this is a defensive no-op record (target only). It will
    // not be reached from the wired handlers.
    empty_record(MutationKind::CharacterData, entity)
}
```

Notes:
- `CommentData` is `elidex_ecs::CommentData` (already imported in mutation/mod.rs? —
  verify; `apply_set_text` imports `TextContent`. Add `CommentData` to the use).
- The borrow: `world_mut().get::<&mut CommentData>` then `drop(cd)` before
  `rev_version` (which needs `&mut self` on a different path) — mirrors the existing
  `set_char_data` Comment branch in `char_data_handlers.rs:59-72`.
- Returns `MutationRecord` (not `Option`) — One record per call; the 5 handlers map
  1:1.

## §4 Handler wiring (`char_data_handlers.rs`)

Each of the 5 handlers: capture `old_data` once (via `get_char_data`, the existing
Text+Comment dual helper), validate offset (existing IndexSizeError gates), call
`apply_replace_data`, push the record via `session.push_notify_record`.

`DomApiHandler::invoke` receives `session: &mut SessionCore` (the handlers currently
bind it as `_session`). Change `_session` → `session` and push the record.

| Handler | old_data source | apply_replace_data(entity, offset, count, replacement, old_data) |
|---------|-----------------|------------------------------------------------------------------|
| `SetData(new)` | `get_char_data` | `(this, 0, utf16_len(&old_data), &new, old_data)` |
| `AppendData(s)` | `get_char_data` | `(this, utf16_len(&old_data), 0, &s, old_data)` |
| `InsertData(off, s)` | `get_char_data` | `(this, off, 0, &s, old_data)` |
| `DeleteData(off, cnt)` | `get_char_data` | `(this, off, cnt, "", old_data)` |
| `ReplaceData(off, cnt, s)` | `get_char_data` | `(this, off, cnt, &s, old_data)` |

Pattern (ReplaceData example):
```rust
fn invoke(&self, this, args, session: &mut SessionCore, dom: &mut EcsDom)
    -> Result<JsValue, DomApiError>
{
    let offset = require_usize_arg(args, 0)?;
    let count = require_usize_arg(args, 1)?;
    let replace_str = require_string_arg(args, 2)?;
    let old_data = get_char_data(this, dom)?;     // dual Text/Comment read
    if offset > utf16_len(&old_data) {
        return Err(index_size_error("offset exceeds data length"));
    }
    let record = apply_replace_data(dom, this, offset, count, &replace_str, old_data);
    session.push_notify_record(record);
    Ok(JsValue::Undefined)
}
```

### §4.1 SetData behavioral fix (TextChange → ReplaceData)

Current `SetData` → `set_char_data` → `dom.set_text_data` fires
`MutationEvent::TextChange` (whole-string). Routing through `apply_replace_data(0,
len, new)` fires `MutationEvent::ReplaceData` instead — a **live-range fidelity
improvement** (per-boundary collapse to offset, not clamp-to-new-length). DOM
§4.10 defines `setData` as `replace data with offset 0, count = length, new data`,
so ReplaceData is spec-correct. Existing tests assert data VALUES, not range
events; no breakage expected. Flag in the PR description as a behavioral fix.

### §4.2 `splice_char_data` deletion (F13 — resolved now)

`splice_char_data` has exactly 4 callers (verified 2026-06-28 via
`grep -rn "splice_char_data" crates/ --include="*.rs"`):
`char_data_handlers.rs:344/371/398/425` = the `AppendData`/`InsertData`/
`DeleteData`/`ReplaceData` handler bodies that §4 rewrites. No boa native, no other
crate references it. So after wiring it is **fully dead → DELETE** (CLAUDE.md
"デッドコードや shim は残さず削除"). (`SetData` already uses `set_char_data`, not
`splice_char_data` — but it too moves to `apply_replace_data`.)

Note: `splice_char_data` (the deleted Comment-fallback wrapper) is distinct from
`splice_utf16` (the pure splice, moved to elidex-ecs in §2.1, which STAYS alive and
is reused by `apply_replace_data` + the unified `replace_text_data`). Also confirm
`set_char_data`'s remaining callers after `SetData` is rewritten — if it too becomes
dead, delete it (impl-time grep; `set_char_data` may still serve other code).

## §5 Delivery path (unchanged B1 mechanism)

`SetData`/etc. are dispatched via `invoke_dom_api` (`dom_bridge.rs:458`). After the
handler returns, `invoke_dom_api` calls `ctx.vm.drain_notify_records()`
(`dom_bridge.rs:510`) → `take_notify_records` → `queue_mutation_record` → `notify`
(ancestor walk + characterData gating via `MutationObservedBy`) → §4.3 microtask →
callback. The `flush()` leak-guard (`session.rs:120` `notify_records.clear()`)
already covers any un-drained scratch. NO new delivery wiring — this slice only
adds record PRODUCTION at the handler seam.

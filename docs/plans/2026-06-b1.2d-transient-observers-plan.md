## ¬ß1 Problem

DOM ¬ß4.3: each node has a **registered observer list** (defined in ¬ß4.3 "Mutation observers"; ¬ß4.3.1 is the `MutationObserver` interface). A **transient registered observer** is appended to a *removed* node so an ancestor's `subtree:true` observer keeps receiving records for mutations in the removed subtree until the next microtask delivery, after which it is cleared. elidex implements neither half:

- **Creation gap (DOM ¬ß4.2.3 "remove" step 15):** *"For each inclusive ancestor inclusiveAncestor of **parent**, and then for each registered of inclusiveAncestor's registered observer list, if registered's options['subtree'] is true, then append a new transient registered observer whose observer is registered's observer, options is registered's options, and source is registered to **node's** registered observer list."* ‚Äî **not** gated by `suppressObservers` (only step 16's record is). elidex's removal paths (`apply_remove_child` + the move-adopt source-removal in `apply_append_child`/`apply_insert_before`/`apply_replace_child` + `apply_replace_all`) do none of this.
- **Clearing gap (DOM ¬ß4.3 "notify mutation observers" step 6.3):** *"For each node of mo's node list: remove all transient registered observers whose observer is mo from node's registered observer list."* ‚Äî elidex's microtask delivery (`vm_api.rs` `clear_pending_records` / the `NotifyMutationObservers` microtask) drains record queues but has no transient housekeeping.

**Observable effect today:** `obs.observe(ancestor, {childList:true, subtree:true})`, then `ancestor.removeChild(mid)` where `mid` has a subtree, then `mid.appendChild(x)` (mutating the now-detached subtree before microtask) ‚Üí the spec delivers the `mid.appendChild` childList record to `obs` (via the transient observer on `mid`); elidex delivers **nothing** (no transient on `mid`, and `mid`'s ancestor walk hits null).

## ¬ß2 Coupled invariants at this corner (the design surface / edge matrix)

1. **Creation timing: eager per-mutation `notify` (spec-aligned with "during remove"), NOT a deferred microtask drain.** Spec runs step-15 creation *during* `remove` (synchronous). **Verified mechanism (plan-review R1, corrected):** elidex's ancestor-walk `notify` runs **eagerly/synchronously per record** ‚Äî `queue_mutation_record` (`mutation_observer.rs:459`) calls `notify_one` (`:521`) immediately, drained at each `invoke_dom_api` boundary in JS-execution order. The `NotifyMutationObservers` microtask (`deliver_pending_mutation_records:469`) does **no walk** ‚Äî it only flushes already-queued per-observer record queues to callbacks. (`natives_promise.rs:279` "FIFO drain" is *callback* delivery order, irrelevant to creation.) So hooking creation inside `notify_one` runs it at the mutation, before `node` can be re-mutated ‚Äî exactly the spec's "during remove", correct **by construction** (no deferred-drain race; the original "microtask-FIFO" framing here was a mis-read of the timing model). The only ordering that matters is *within one call emitting multiple records* (move-adopt ¬ß2.5): the records `Vec` iterates source-removal-first (`move_record_list`), so the removal's transient precedes the destination record's walk. **Invariant to verify:** `notify_one` is the single eager per-record walk site (both callers `:437` batch loop + `:459` route through it); the per-call records `Vec` is source-removal-first.
2. **Layering: creation cannot live in `apply_*`.** `elidex-api-observers` **depends on** `elidex-script-session` (Cargo.toml), so `apply_remove_child` (script-session) **must not** reach `MutationObservedBy` (api-observers) ‚Äî that would be a dependency cycle. ‚Üí transient creation is owned by **`elidex-api-observers`** (a new registry method) and **orchestrated by the VM** at the drain (VM depends on both). This is also why B1.2d-i and B1.2d-ii are layer-disjoint.
3. **Walk PARENT's inclusive ancestors; append to the removed NODE.** Not node's own observers (those stay on node, still observing it directly). Source ‚âî the ancestor's registered observer.
4. **`suppressObservers` does NOT gate creation ‚Äî and every removal surfaces its removed set on a `ChildList` record (plan-review R1 correction).** Spec step 15 (creation) has no `suppressObservers` guard (only step 16's *record* does). **Verified:** every elidex removal path carries the removed set on a `ChildList` record reaching `notify_one` ‚Äî standalone (`apply_remove_child` = 1 removal record; move-adopt = a source-removal record) **or coalesced** (replace = `removed_nodes=[old_child]`; replace-all = removed children in one record, queued iff added‚à™removed non-empty). So the ¬ß5.2 **record-keyed** hook (`record.kind==ChildList && !removed_nodes.is_empty()`, keyed off `record.target` + `record.removed_nodes`) covers **all** of them ‚Äî there is **no** record-less removal path that loses the removed set (the earlier "suppressed records ‚Üí need a non-record path" framing was inverted). Invariant to confirm at impl: no `apply_*` path detaches a non-empty removed set while emitting zero `ChildList` record.
5. **Move-adopt removals create transients too.** `el.appendChild(alreadyParented)` adopts ‚Üí source-parent removal (¬ß4.5 step 2 / remove). That removed-from-old-parent step is a `remove` ‚Üí step 15 applies. So the move-adopt source-removal record drain must also create transients on the moved node (w.r.t. its *old* parent's ancestors). Edge: the node is immediately re-inserted elsewhere ‚Äî but the transient (keyed to the old-ancestor observer) still belongs until microtask.
6. **Clearing needs "mo's node list"; elidex has no observer‚Üínodes index.** ECS-native resolution: transient observers ARE `MutationObservedBy` entries (component), so clearing = **one ECS query** over all `MutationObservedBy` removing entries where `transient && observer ‚àà notifySet` (the cleared observers). No reverse index, no side-store (mirrors the existing `disconnect()` retain-query `mutation.rs:150`). Preserves the side-store‚Üícomponent rule + Sub-check 2b (no new SoT to reconcile).
7. **`subtree:true` preserved on the transient ‚áí deep-subtree mutations caught by the upward walk.** Spec appends the transient to `node` only (not descendants), carrying the *original options* (`subtree:true`). A mutation deep in the removed subtree produces a record whose `notify` walk climbs target ‚Üí ‚Ä¶ ‚Üí `node` (transient, subtree:true) ‚Üí matches. So append-to-node-only is correct (no per-descendant transients). **Invariant:** the transient entry carries the original `MutationObserverInit` verbatim.
8. **ECS-native by construction (Sub-check 2a/2b).** Transient = a `MutationObservedBy` entry with a `transient` marker; it participates in the existing `notify` ancestor-walk with **zero** walk changes (the walk already reads `MutationObservedBy`). The ONLY new code = creation (append entries at drain) + clearing (query-remove at microtask end) + the marker field. No new registry data structure, no OO observer-list object.
9. **Dual-runtime / VM-scoped.** MO delivery is VM-side (B1); transient creation/clearing piggyback the VM drain. boa = known-differ, scope-out.

## ¬ß4 Design decision (ideal: transient = a marked component entry, VM-orchestrated, cleared by ECS query)

The ideal is **not** a new "transient observer list" structure (OO direct-translation, Axis 2a) nor an observer‚Üínodes reverse index (side-store, Axis 2b). A transient registered observer **is** a `MutationObservedBy` entry that happens to be marked transient ‚Äî it then participates in the existing notify-walk for free.

- **Data:** `MutationObservation` (`mutation.rs:76`) gains `transient: bool` (default false). The spec's `source` (a registered observer) collapses to the entry's `observer` id in elidex: a node has at most one registration per observer (observe() replaces), so both clearings ‚Äî notify step 6.3 ("observer is mo") and observe() step 7.1 ("source is registered") ‚Äî key on **observer id**. No `source` field needed; ¬ß4.3 fidelity preserved by construction (documented, not deferred).
- **Creation seam (api-observers, VM-invoked):** new `MutationObserverRegistry::add_transient_observers(&self, dom: &mut EcsDom, parent: Entity, removed: &[Entity])` = remove-step-15: walk `parent`'s inclusive ancestors (reuse the same bounded walk as `notify`, `MAX_ANCESTOR_DEPTH`), collect each `subtree:true` `MutationObservation`, append a `transient:true` copy to each `removed` node's `MutationObservedBy` (insert the component if absent; dedup if an identical (observer,transient) entry exists). Invoked by the VM **at the record-drain**, for each drained ChildList record with non-empty `removed_nodes`, using `record.target` (= old parent) + `record.removed_nodes`, **before** `notify`-ing subsequent records (FIFO order, invariant ¬ß2.1).
- **Clearing seam (api-observers, VM-invoked):** in the VM microtask delivery, after building the delivered-observer set (notifySet) and firing callbacks, run one ECS query over `MutationObservedBy` (mirror `disconnect()`'s `retain`-query `mutation.rs:150`) removing entries where `transient && notifySet.contains(observer)`; drop the component if it becomes empty. (Also: `observe()` on a node removes that node's existing `transient` entries for the same observer ‚Äî re-observe path.)
- **Why VM-orchestrated, not a ConsumerDispatcher consumer:** a `MutationEvent::Remove` consumer (like `LiveRangeBridge`) was considered ‚Äî rejected because (a) it splits transient lifecycle across two seams (creation at ConsumerDispatcher, clearing at microtask) on different timing models, and (b) the record-drain already iterates exactly the removal records in FIFO order with `parent`+`removed_nodes` in hand, so creation co-locates cleanly with the existing `notify` call (one orchestration point, single-writer). ECS-native + One-issue-one-way.

**Crate-API / layering mapping:** algorithm home = `elidex-api-observers/src/mutation.rs` (owns `MutationObservedBy` + registry); orchestration = VM drain (`vm/vm_api.rs` + `vm/host/mutation_observer.rs`/`natives_promise.rs`). No `elidex-script-session`/`elidex-dom-api` change. No `vm/host/` *algorithm* (the walk/append/clear live in api-observers; VM only calls them at the drain = marshalling/orchestration, Layering-mandate-clean).

## §1 Problem

`MutationObserver` does not observe the ChildNode/ParentNode direct tree ops invoked from the **VM** (`elidex-js`). Three coupled defects, one root:

1. **MO-silent (the B1.2 gap).** `before`/`after`/`replaceWith`/`remove`/`prepend`/`append`/`replaceChildren` mutate the tree but produce no `MutationRecord` — neither the VM natives (`crates/script/elidex-js/src/vm/host/childnode.rs`, `parentnode.rs`, call `EcsDom::insert_before`/`append_child`/`remove_child` directly) nor the engine-independent dom-api handlers (`crates/dom/elidex-dom-api/src/child_node/mutations.rs`, use record-free `insert_node_expanding_fragment`).
2. **Double implementation (One-issue-one-way violation).** The algorithm exists twice: dom-api registry handlers `Before`/`After`/… (used by **boa** via `invoke_dom_handler_void` and **wasm-runtime**) and a parallel VM native re-implementation (`native_child_node_*`/`native_parent_node_*` + recursive `flatten_into`).
3. **Layering violation (CLAUDE.md "VM host/ は engine-bound 責務のみ").** The VM natives re-implement the tree-mutation algorithm (`flatten_into`, viable-sibling, convert-to-fragment) instead of calling the engine-independent crate. `EcsDom::*` direct calls here are NOT marshalling — they are the algorithm.

**boa is the proof of the fix:** boa already routes every one of these seven methods through the dom-api registry handler (`crates/script/elidex-js-boa/src/globals/element/accessors/node_methods.rs:115` `register_child_parent_mixin_methods` → `invoke_dom_handler_void`). The VM is the laggard. The canonical `appendChild`/`insertBefore`/`replaceChild` already prove the record-producing path: VM `node_proto.rs` → `invoke_dom_api` → registry `AppendChild`/`InsertBefore`/`ReplaceChild` (`element/tree.rs`) → `apply_append_child`/… (record-producing, `elidex-script-session/src/mutation/mod.rs`) → drained at `invoke_dom_api` Phase 2.5 (`dom_bridge.rs:521`).

## §2 Coupled invariants at this corner (the design surface / edge matrix)

1. **Record shape & count** (inherited, B1/B1.2a/B1.2-fragment): fresh = 1 record; move (already-parented node) = 2 (source-removal §4.5 adopt, NOT suppressed + dest insertion); fragment insert = fragment record (removedNodes=children, addedNodes=«») + dest record; replace = 1 coalesced (+1 source-removal on move). `apply_append_child`/`apply_insert_before`/`apply_remove_child`/`apply_replace_child` already encode all of this — b-1 **reuses** them, does not re-derive.
2. **`replace all` coalescing** (§4.2.3 `#concept-node-replace-all`, verified): remove all children with `suppressObservers=true` (no per-child records) + insert with `suppressObservers=true` (no insert/fragment record) + **one** combined record `(addedNodes, removedNodes, null, null)`. This is a **new** record discipline distinct from the per-node `apply_*` — needs a new `apply_replace_all` primitive. `replaceChildren` is its sole b-1 consumer.
3. **`convert nodes into a node`** (§4.2.6 `#convert-nodes-into-a-node`): strings → Text; one node → that node; many → a fresh DocumentFragment with each node **appended via the DOM "append"** (= mutation append, which **expands** sub-fragments). The current `convert_nodes_into_node` (`child_node/mod.rs:317`) uses the *non-expanding* `EcsDom::append_child` primitive → a fragment arg is linked as a child of the temp fragment → **nested fragment** → `append(frag1, frag2)` inserts nested (latent bug, affects boa today; masked in the VM only because recursive `flatten_into` re-flattens). b-1 must make `convert_nodes_into_node` expand fragment args one level (spec-faithful; one level suffices because a fragment's children are never fragments — B1.2-fragment "never nests").
4. **Viable-sibling capture** (ChildNode `before`/`after`/`replaceWith`): the reference child must be computed **before** `convert_nodes_into_node` reparents the args (existing `viable_prev_sibling`/`viable_next_sibling`, `child_node/mutations.rs:47/93/162`). Preserved verbatim — the rework changes only the *insertion* call, not the sibling math.
5. **Single-level vs recursive fragment expansion**: `expand_fragment` (canonical, `apply_*`) is single-level; VM `flatten_into` is recursive. Once `convert_nodes_into_node` expands fragment args (inv. 3), the temp fragment is flat, so single-level `expand_fragment` is sufficient and `flatten_into`'s recursion is provably dead for JS-reachable states. ⇒ delete the recursive VM machinery.
6. **CE reactions** (HTML §4.13.6): driven by the `EcsDom` `ConsumerDispatcher` (Mechanism A) on the per-node relink inside `apply_*`/`expand_fragment` — unchanged by routing; the dom-api handlers already drive the same `EcsDom` primitives.
7. **Dual-runtime / leak-guard**: the dom-api handlers are shared by boa + wasm + VM. After the rework they `session.push_notify_record(...)`. The VM drains at `invoke_dom_api` Phase 2.5; **boa/wasm never drain** → records accumulate in `notify_records` until `SessionCore::flush`'s leak-guard drain discards them (existing B1 behavior, removed at S5). Net: boa/wasm behavior is unchanged (records produced but unobserved); only the VM newly delivers. Must confirm pushing records is side-effect-free for boa/wasm.
8. **VM arg marshalling for variadic (Node | string)**: the VM dispatcher must pass *string* args through `invoke_dom_api` so the dom-api `collect_nodes` converts them to Text (boa does this via `boa_args_to_elidex`). Today the VM's `prepare_arg`/`materialize` is exercised by `appendChild` (Node args only). **Verify** string passthrough is supported; extend if not (this is the one genuinely new VM-side code path).

## §4 Design decision: converge onto `invoke_dom_api → dom-api handler → apply_*`

The ideal (and only One-issue-one-way + layering-correct) resolution: **make the VM mirror boa.** The dom-api registry handler is the single algorithm home; `apply_*` is the single record-producing primitive home; the VM native becomes a thin marshalling dispatcher exactly like `appendChild`.

Rejected alternative (B): keep the VM natives, teach each to push records. Perpetuates the double-impl + layering violation; rejected on philosophy.

This collapses all three §1 defects at once: routing the VM through `invoke_dom_api` gives records for free (Phase 2.5 drain) **and** deletes the VM duplicate algorithm **and** restores layering.

## §5 Implementation

### 5.1 dom-api handler rework (`crates/dom/elidex-dom-api/src/child_node/mutations.rs`)
Replace each `insert_node_expanding_fragment(parent, node, ref, dom)?` with a record-producing call + push:
- `ref == None` → `apply_append_child(dom, parent, node)`; `ref == Some(r)` → `apply_insert_before(dom, parent, node, r)`.
- Push the returned `Vec<MutationRecord>` via `session.push_notify_record(rec)` for each. Empty list semantics: for these mixin methods an empty list from a non-fragment node is a genuine failure → map to `HierarchyRequestError` (mirror `element/tree.rs`); an empty-fragment no-op stays a no-op (the `node` here is either a single arg node or a temp fragment built by `convert_nodes_into_node` — if the temp fragment is empty the method was an empty-args no-op already short-circuited at `nodes.is_empty()`).
- `Before`/`After`/`ReplaceWith`: keep viable-sibling capture verbatim; only the insertion call changes.
- `ReplaceWith`: the same-parent replace branch (`mutations.rs:167`) currently does `remove_child` then `insert_node_expanding_fragment`. Re-express as: `apply_remove_child(parent, this)` for the removal record, then `apply_insert_before`/`apply_append_child` for the insertion. **Read-order invariant (F3)**: the viable reference sibling is captured **pre-convert** (inv. 4, unchanged); each `apply_*` captures its own `previousSibling`/move-source **pre-write** internally (`capture_move_source`/`prev_exposed_sibling`, `mutation/mod.rs:369/436/464`). Because `apply_remove_child(this)` runs first, the subsequent `apply_insert_before` reads the sibling chain *after* `this` is gone — so the reference sibling must be one captured pre-convert that is NOT `this` (the viable-sibling walk already excludes the to-be-moved args; it does not exclude `this`, so for `replaceWith` use `viable_next_sibling` which steps past `this`). Verify record ordering (removal-of-`this` THEN insertion) against §4.2.8 replaceWith steps + add a same-parent-move test.
- `ChildNodeRemove` → `apply_remove_child(dom, parent, this)` + push.
- `ReplaceChildren` → new `apply_replace_all` (5.3).

### 5.2 `convert_nodes_into_node` fragment-arg fix (`child_node/mod.rs:317`)
When appending an arg that is itself a DocumentFragment to the temp fragment, expand it (move its children in) rather than linking it — one level (a fragment's children are never fragments). Keeps the temp fragment flat = spec-faithful "append". **The expansion uses the record-free raw `EcsDom` write (`dom.append_child` per moved child), NOT `apply_append_child`** — the temp fragment is transient/unobserved, so it MUST NOT emit a fragment/dest/move-source record (the canonical `apply_*` would); the records for the *outer* insertion are produced once, later, when `apply_*` expands T into `parent` (§5.1). (F2) Removes the latent `append(frag1, frag2)` nesting bug for boa too.

### 5.3 New `apply_replace_all` primitive (`elidex-script-session/src/mutation/mod.rs`)
Per §4.2.3 `#concept-node-replace-all`: `removedNodes` = parent's current children, snapshotted via **`dom.child_list_uncapped(parent)`** — the same SoT primitive `expand_fragment` uses (`mutation/mod.rs:309`), NOT `dom.children()`/`children_iter` which truncate at `MAX_ANCESTOR_DEPTH` and would silently drop a >cap parent's `removedNodes` tail (the bug #387 fixed for `expand_fragment`). (F1) `addedNodes` = node's children if fragment else «node» (or «» if node is null); remove all children + insert node via the record-free raw `EcsDom` writes (no per-node records); emit **one** combined `ChildList` record `(target=parent, added, removed, prev=null, next=null)` **iff added∪removed non-empty** (step 7 — an empty parent + null node ⇒ zero records, no spurious empty record). Single One-issue-one-way home, reused by B1.2c (textContent).

### 5.4 VM dispatcher conversion (`childnode.rs`, `parentnode.rs`)
Convert `native_child_node_before`/`after`/`replace_with`/`remove` + `native_parent_node_prepend`/`append`/`replace_children` to thin dispatchers: receiver brand-check (`require_receiver`, kept) → `invoke_dom_api(ctx, "<method>", entity, &args)`. Model: `node_proto.rs:692` `native_node_append_child`. **Verify (inv. 8)** `invoke_dom_api`/`prepare_arg` passes string args through to `collect_nodes`; extend the arg-prep path if Node-only today.

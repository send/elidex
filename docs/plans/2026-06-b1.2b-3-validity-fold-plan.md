## ┬¦1 Problem

`element/tree.rs` `AppendChild` (:46) / `InsertBefore` (:94) / `ReplaceChild` (:~225) / `insertAdjacent*` (:~340) do, in effect:
```rust
let records = apply_append_child(dom, this, child_entity);   // NO validity gate
if records.is_empty() {
    if is_empty_fragment_noop(dom, this, child_entity) { return Ok(...); }
    return Err(hierarchy_error("cycle or invalid parent"));   // ONLY cycle/shadow/ref-child
}
```
`apply_append_child` (`elidex-script-session/src/mutation/mod.rs`) enforces only **step 2** (host-including-inclusive-ancestor cycle, atomic for fragments) + **step 3** (ref-child-is-child, `apply_insert_before`) + `rejects_shadow_root_insertion`. It does **NOT** enforce:
1. **Step 1** ŌĆö parent must be `Document`/`DocumentFragment`/`Element` (e.g. `textNode.appendChild(x)` should `HierarchyRequestError`; today the EcsDom relink may silently no-op or mis-link).
2. **Step 4** ŌĆö node must be `DocumentFragment`/`DocumentType`/`Element`/`CharacterData` (e.g. `el.appendChild(documentNode)` should throw; today not checked).
3. **Step 5** ŌĆö `Text` not under a `Document`; `doctype` only under a `Document` (e.g. `document.appendChild(textNode)` / `el.appendChild(doctype)` should throw).
4. **Step 6** ŌĆö `Document` element-child constraints (e.g. `document.appendChild(secondHtmlElement)` should throw "Document already has an element child"; `document.appendChild(doctypeWhenElementExists)` ordering).

So `appendChild`/`insertBefore`/`replaceChild`/`insertAdjacent*` are **spec-non-conformant** for these cases ŌĆö a class of `HierarchyRequestError`s that simply do not fire. The mixins (`before`/`after`/ŌĆ”) are conformant (they call the validator), producing a **split-surface inconsistency** (One-issue-one-way violation): two validity paths for the same DOM invariant, one complete, one partial.

## ┬¦2 Coupled invariants at this corner (the design surface / edge matrix)

1. **Validator-first ordering, on the PRE-mutation tree.** `ensure_*_validity` reads `children_iter`/sibling/ancestor state and must run **before** `apply_*` mutates. Insert it as the first step after entity resolution (mirrors the mixins: `ensure_pre_insertion_validity(...)?; route_insert(...)`). The validator is read-only (`&EcsDom`), so it composes cleanly before the `&mut` `apply_*`.
2. **No self-exclusion in the step-6 Document counting (pre-insert).** ŌÜĀ’ĖÅ **Corrected post-plan (Codex B1.2b-3 R1).** The pre-plan draft asserted `validate_document_element_constraint`'s `c != node` clause was a spec-correct "self-exclusion" so that `document.appendChild(document.documentElement)` would pass ŌĆö **this was wrong**. webref ┬¦4.2.3: pre-insert step 6 Element = "parent has an element child" (counts ALL element children, **no** exclusion), and pre-insertion validity (pre-insert step 1) runs **before** the self-reference adjustment (pre-insert step 3). The documentElement is therefore still an element child of the document when validity runs, so `document.appendChild(document.documentElement)` **throws HierarchyRequestError**. The "that is not child" exclusion exists **only in the *replace* algorithm** (carried by `validate_document_element_constraint`'s `exclude` slice = ┬½child┬╗; the spec: "the above statements differ from the pre-insert algorithm"). **Fix**: remove the spec-wrong `c != node` from all three branches (Element / DocumentFragment-element-subcase / DocumentType); `exclude` is the sole (replace-only) exclusion. Regression-locked at the validator level (`child_node/tests/validation.rs`: element + doctype re-append throw, replace-swap stays OK) and the handler level (`element/tests_tree.rs`: `append_child_existing_document_element_throws_hierarchy`).
3. **Replace uses the DISTINCT replace-validity (NOT pre-insert).** `replaceChild` ŌåÆ `ensure_replace_validity` (┬¦4.2.3 `#concept-node-replace` steps 1-6): step 3 = `child`'s parent is parent (the *old* child), step 6 counts **exclude `child`** ("parent has an element child **that is not child**"). webref-confirmed the replace statements "differ from the pre-insert algorithm". Do NOT route replace through `ensure_pre_insertion_validity`. The existing inline `replaceChild` old-child-is-child `NotFoundError` (tree.rs ~:247) is **subsumed** by replace step 3 ŌåÆ remove it (One-issue-one-way), let the validator throw it (same error, same order: step 3 NotFound precedes step 4 node-type).
3b. **Self-replace no-op ├Ś validator ordering (coupled corner).** `ReplaceChild` also carries a `new_entity == old_entity` no-op short-circuit (tree.rs ~:259, browser-parity, returns `old` without mutating). The spec runs replace validity **steps 1-6 BEFORE** the self-replace collapse (steps 7-8 referenceChild adjustment / step 11 "can only be false if child is node"). So after removing the inline :247 NotFound, `ensure_replace_validity` MUST run **before** the :259 short-circuit: `el.replaceChild(orphan, orphan)` (orphan not a child) ŌåÆ step-3 **NotFoundError** (validator), NOT a silent no-op; a valid `el.replaceChild(x, x)` (x is a child of el) ŌåÆ validator passes (x is a valid child; for a non-Document `el` step 6 does not run, and for a Document `el` the replace `exclude` = ┬½child┬╗ skips x in the element-child count) ŌåÆ then the no-op short-circuit returns. Keep the short-circuit; just order it after the validator.
4. **NotFoundError(identity-map) vs the validity throws ŌĆö ordering preserved.** Entity-resolution `NotFoundError` ("child not found" = the JS object isn't a live wrapper) is a *marshalling* failure distinct from ┬¦4.2.3 step-3 NotFound (entity exists but isn't a child). Order stays: resolve entities (identity-map NotFound) ŌåÆ validator (step 1-6 throws) ŌåÆ `apply_*`. No reordering of the two NotFound sources.
5. **Shadow-root rejection stays in `apply_*` (NOT folded into the validator).** A ShadowRoot is `NodeKind::DocumentFragment` ŌåÆ it *passes* ┬¦4.2.3 step 4. Its non-movability is an elidex engine invariant (┬¦4.8 host edge), enforced by `rejects_shadow_root_insertion` + (mixins) `collect_nodes`. The post-validator empty-list branch therefore narrows to **{empty-fragment no-op | shadow-root rejection}**: `is_empty_fragment_noop` distinguishes the no-op; otherwise the empty list is the shadow-root rejection ŌåÆ `HierarchyRequestError`. Keep the fallback (its message can be refined to "shadow root cannot be inserted"); its now-redundant ancestor re-check (the validator covers step 2) MAY be simplified but is harmless.
6. **insertAdjacent runs the validator on the RESOLVED site.** `resolve_adjacent_site` yields `(parent, refChild)` for the four positions (beforebegin/afterbegin/beforeend/afterend), with a **null-parent no-op** for beforebegin/afterend on a parentless `this` (returns success, no throw). The validator runs only when a parent is resolved, on `(resolved_parent, node, resolved_ref)`. Invariant: do not throw on the null-parent no-op path; do validate the real-parent path (an `insertAdjacentElement('beforebegin', doctype)` on a document child should now throw step-5/6).
7. **`select.add` is OUT (HTML ┬¦2.6.4.3 is authoritative).** Asserted, not assumed: node Ōłł {option, optgroup} ŌćÆ step 1 (parent=select=Element Ō£ō), step 4 (Element Ō£ō), step 5 (not Text/doctype Ō£ō), step 6 (select ŌēĀ Document Ō£ō) are all satisfied; cycle = ┬¦2.6.4.3 step 1 (`is_ancestor_or_self`), before-must-be-descendant = ┬¦2.6.4.3 step 2. The ┬¦2.6.4.3 algorithm **is** the spec validity for `options.add`; do not layer ┬¦4.2.3 on top.
8. **Dual-runtime / no VM change.** The validators live in engine-independent `elidex-dom-api`; both the VM (`invoke_dom_api`) and boa route `appendChild`/`insertBefore`/`replaceChild`/`insertAdjacent*` through the same handlers ŌåÆ both gain the validity by construction. No `vm/host` edit. (Boa is S5-deletion-bound; the fix is in the shared handler regardless.)

## ┬¦4 Design decision: reuse the canonical validators (no new code, no new layer)

The ideal (One-issue-one-way + layering-correct): the core Node methods call the **existing** `ensure_pre_insertion_validity`/`ensure_replace_validity` ŌĆö the exact functions the mixins use ŌĆö before `apply_*`. No new validator, no duplicated checks, no VM change. Rejected alternatives:
- *"Teach `apply_*` to run full validity"* ŌĆö pushes a DOM ┬¦4.2.3 algorithm down into the `elidex-script-session` mutation primitive layer (engine-internal), duplicating what the dom-api validator already owns + forcing every `apply_*` caller (incl. the mixins, which already validate) to re-pay it. Wrong layer (validity is a dom-api-handler concern; `apply_*` is the record-producing structural primitive).
- *"Add a third per-method inline check"* ŌĆö re-creates the split-surface this slice exists to remove.

**Crate-API / layering mapping:** algorithm home unchanged = `ensure_pre_insertion_validity`/`ensure_replace_validity` (`elidex-dom-api/src/child_node/mod.rs`, `pub(crate)`); the change is the call-sites in the sibling module `elidex-dom-api/src/element/tree.rs` (add `use crate::child_node::{ensure_pre_insertion_validity, ensure_replace_validity};`). VM (`vm/host/*`) untouched.

## ┬¦5 Implementation

### 5.1 `AppendChild` (`element/tree.rs`)
After resolving `child_entity`, before `apply_append_child`:
```rust
ensure_pre_insertion_validity(this, child_entity, None, dom)?;   // append = pre-insert before null
let records = apply_append_child(dom, this, child_entity);
// ... existing empty-list handling: now only {empty-fragment no-op | shadow-root rejection}
```

### 5.2 `InsertBefore` (`element/tree.rs`)
Both branches. Null-ref branch: `ensure_pre_insertion_validity(this, new_entity, None, dom)?` then `apply_append_child`. Ref branch: resolve `ref_entity`, then `ensure_pre_insertion_validity(this, new_entity, Some(ref_entity), dom)?` then `apply_insert_before`. (The validator's step 3 covers ref-child-is-child; keep the identity-map NotFound for ref-not-a-live-wrapper.)

### 5.3 `ReplaceChild` (`element/tree.rs`)
After resolving `new_entity`+`old_entity`, and **before** the `new_entity == old_entity` self-replace short-circuit (tree.rs ~:259, per ┬¦2.3b): `ensure_replace_validity(this, new_entity, old_entity, dom)?` ŌåÆ then the self-replace short-circuit ŌåÆ then `apply_replace_child`. **Remove** the inline old-child-is-child `NotFoundError` (~:247, subsumed by replace step 3). **Keep**: identity-map NotFound for missing wrappers; the self-replace short-circuit (re-ordered to *after* the validator).

### 5.4 `insertAdjacentElement` / `insertAdjacentText` (`element/tree.rs`)
After `resolve_adjacent_site` yields `Some((parent, ref))` (skip the null-parent no-op): `ensure_pre_insertion_validity(parent, node, ref, dom)?` before `apply_adjacent_insert`.
- **insertAdjacentElement**: `node` = the resolved element arg (already exists) ŌåÆ validator, then apply.
- **insertAdjacentText**: the Text node is **created before** the validator (the entity is the validator's `node` input). A step-5 throw (Text adjacent to a document-child at `beforebegin`/`afterend`, where the resolved parent = Document) would orphan the freshly-created Text. **Directive**: create the Text ŌåÆ `ensure_pre_insertion_validity(parent, text, ref, dom)` ŌåÆ on `Err`, `dom.destroy_entity(text)` **before** propagating the error (mirrors the existing empty-list cleanup at tree.rs ~:421 + the #399 resolve-before-alloc / no-leak discipline). Steps 1/4 trivially pass for a Text; step 5 is the live throw. (Do NOT pre-alloc-skip via a node-kind special-case ŌĆö that re-duplicates the validator's kind logic; create-validate-destroy reuses the canonical path.)

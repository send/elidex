## §2 Canonical seam + delivery architecture

The B1 mechanism (unchanged): record produced **synchronously** at the
engine-independent seam → `session.push_notify_record(record)` →
`drain_notify_records` at the `invoke_dom_api` boundary → `queue_mutation_record`
→ §4.3 microtask → callback.

For Range text-splice (not via `invoke_dom_api`): records join the existing
`Vec<MutationRecord>` returned by `delete_contents` / `extract_contents` /
`insert_node` → `commit_range_mutation_records` (B1.2d-ii chokepoint, unchanged).

### apply_replace_data location

`elidex-script-session/src/mutation/mod.rs` — same crate as `apply_append_child`,
`apply_remove_child`, etc. Required because `elidex-dom-api` CharacterData handlers
import `apply_*` from `elidex-script-session`; placing it in `elidex-dom-api` would
create a circular dependency.

**Implication**: the function must not call `splice_char_data` (in `elidex-dom-api`).
Instead it calls EcsDom primitives directly — identical to the existing pattern
where `apply_append_child` calls `dom.append_child` rather than going through a
dom-api helper.

### UTF-16 splice helper de-duplication (F3 — One-issue-one-way)

The Comment branch of `apply_replace_data` needs the same UTF-16 splice logic as
`splice_utf16` (currently a `pub(crate)` fn in
`elidex-dom-api/char_data/char_data_handlers.rs`). Inlining a copy in
`elidex-script-session` would leave the splice algorithm in **two** places — a
strangler middle-state CLAUDE.md "One-issue-one-way" forbids. **Resolution
(B1.3-i)**: move `splice_utf16` (a pure `&str + offset + count + Option<&str> →
String` function with **no DOM/ECS dependency**) to a crate both layers depend on
— `elidex-ecs` (the `EcsDom` crate, already a dependency of both
`elidex-script-session` and `elidex-dom-api`) or a small shared `util` module
there. Both the existing `elidex-dom-api` callers and the new
`apply_replace_data` Comment branch then call the single canonical helper. The
surrogate-pair lossy-decode contract (`from_utf16_lossy`, the
`*surrogate_pair*` tests) moves with it. This is a standalone prerequisite-style
de-dup folded into B1.3-i (the function is tiny and has no behavioral change).

## §3 B1.3-i — `apply_replace_data` + CharacterData handler wiring

### §3.1 `apply_replace_data` design

```rust
/// DOM §4.10 "replace data" — record-producing primitive.
///
/// `old_value` MUST be captured by the caller BEFORE this call. DOM §4.10
/// "replace data" step 4 queues the characterData mutation record passing
/// **node's data** as the oldValue argument BEFORE steps 5-7 mutate the data;
/// the "queue a mutation record" AO (§4.3.2 step 3.3) then exposes that oldValue
/// only to observers with characterDataOldValue:true.
pub fn apply_replace_data(
    dom: &mut EcsDom,
    entity: Entity,
    offset: usize,
    count: usize,
    replacement: &str,
    old_value: String,
) -> MutationRecord
```

The caller pre-captures `old_value` (the entity's full data string before mutation)
and passes it in. This keeps the function pure (no second read for old_value) and
makes the timing contract explicit in the signature.

**Text / CDATASection path**: `dom.replace_text_data(entity, offset, count, replacement)`.
This fires `MutationEvent::ReplaceData` for live-range boundary adjustment (correct
per §5.5 replace-data rule; §4.10 step 8 boundary steps). Returns `MutationRecord {
kind: CharacterData, old_value: Some(old_value), target: entity, .. }`.

**Comment path**: Read `CommentData`, apply the shared UTF-16 splice helper (F3:
`splice_utf16` moved to a shared crate), write back, call `dom.rev_version(entity)`.
Returns the same `MutationRecord` shape. Note: the DOM §4.10 "replace data"
algorithm (incl. the live-range boundary steps 8-11) applies to ALL CharacterData
nodes; elidex's `replace_text_data` / `MutationEvent::ReplaceData` engine hook —
which drives `LiveRangeBridge` boundary adjustment — is only invoked for
Text/CDATASection (the `CommentData` write path does not fire it). So Comment data
mutation produces a characterData MutationRecord but no live-range fixup event — a
correct elidex implementation detail, NOT a §5.5 restriction.

Returns `MutationRecord` (not `Option`) — callers already perform the CharacterData
brand check and offset validation before invoking.

**SetData alignment**: `SetData(data)` is equivalent to `replaceData(0, length, data)`
per DOM §4.10. The current `SetData` handler calls `set_char_data → dom.set_text_data`
which fires `MutationEvent::TextChange` (not `ReplaceData`). B1.3-i will change
`SetData` to route through `apply_replace_data(entity, 0, utf16_len(&old), &new, old)`
which fires `MutationEvent::ReplaceData`. This is a correctness fix: live-range
boundary adjustment for full-replace via TextChange gives a coarser result than
ReplaceData (clamp-only vs. correct per-boundary collapse to offset). Existing tests
check data values, not Range events — no test breakage expected. Flag in impl as
behavioral fix.

### §3.2 Handler wiring (char_data_handlers.rs)

Five handlers change pattern. For each: `_session` → `session`, read `old_data`
once for both (a) IndexSizeError validation and (b) old_value, call
`apply_replace_data`, push record to `session.push_notify_record`.

| Handler | apply_replace_data args |
|---------|------------------------|
| `SetData(new)` | `(entity, 0, utf16_len(&old_data), &new, old_data)` |
| `AppendData(s)` | `(entity, utf16_len(&old_data), 0, &s, old_data)` |
| `InsertData(off, s)` | `(entity, off, 0, &s, old_data)` |
| `DeleteData(off, cnt)` | `(entity, off, cnt, "", old_data)` |
| `ReplaceData(off, cnt, s)` | `(entity, off, cnt, &s, old_data)` |

`AppendData` currently reads length implicitly via `get_char_data` before
`splice_char_data`. The new pattern reads once and reuses the string for old_value
and length computation — removes the double-read.

`get_char_data` returns the entity's full data string (Text or Comment), so one
call covers both Text and Comment. The `apply_replace_data` dispatch to the right
EcsDom branch is internal.

### §3.3 Coupled invariants (A1–A3)

- **A1 (replaceData algorithm)**: all five handlers map to `apply_replace_data`
  uniformly — One-issue-one-way for characterData record production.
- **A2 (oldValue timing)**: `get_char_data(entity, dom)` runs first in every handler;
  `apply_replace_data` receives the already-captured string. The DOM mutation
  happens INSIDE `apply_replace_data` (after old_value is in scope).
- **A3 (Comment branch)**: covered by `apply_replace_data`'s internal dispatch
  (Text path via `replace_text_data` / Comment path via the shared `splice_utf16`
  helper [F3] + `CommentData` write + `rev_version`). Both branches return the
  same record shape.

## §4 B1.3-ii — splitText records + Range text-splice records

Depends on B1.3-i (`apply_replace_data` must be available).

### §4.1 splitText dual records

**SplitText handler** (`char_data_handlers.rs`) currently:
1. Brand check this = Text
2. `split_text_at_offset(this, offset, dom)` → `new_node`
3. Return new_node wrapper

B1.3-ii adds: capture pre-split state + assemble records post-split.

**Cannot** call `apply_insert_before`/`apply_append_child` after
`split_text_at_offset` — those apply ECS mutations that `split_text_at_offset`
already made (would double-mutate). Records must be assembled from pre-captured
state + post-split observation.

**Pre-split captures (before `split_text_at_offset` call)**:
- `old_data = get_char_data(this, dom)?` — for characterData record
- `parent_opt = dom.get_parent(this)` — for childList record target
- `next_sibling_opt = dom.get_next_sibling(this)` — for childList record nextSibling

**Post-split assembly (after `split_text_at_offset` call succeeds with `new_node`)**:

`split_text_at_offset` has ALREADY applied both ECS mutations (insert newNode
[split step 7.1] + truncate entity to head [split step 8]). So the records are
**assembled from pre-captured state**, NOT by calling `apply_*` (which would
double-mutate). This mirrors B1.2a's move-record pattern, where Range/childnode
handlers build records manually from `capture_move_source` + post-write sibling
reads because the ECS mutation already happened at the primitive layer.

**Design rationale (F14 — why not make `split_text_at_offset` return records?)**:
`split_text_at_offset` is a pre-B1 engine-independent DOM-algorithm function with
non-MO callers (boa-path, future Range-spec consumers) and a bespoke
`insert → fire_split_text → set_text_data` dispatch order that the Range
live-tracking events depend on. Injecting MutationRecord production into it would
(a) change its interface for all callers, and (b) couple record assembly to the
internal event-dispatch ordering. Keeping record assembly in the handler layer
(post-split, from captured state) is the ideal seam: `split_text_at_offset` stays
a pure DOM-mutation primitive; the MO-record responsibility lives at the
session-boundary handler.

**F1 — cross-crate `empty_record` constraint**: `empty_record` is `pub(super)` in
`elidex-script-session/src/mutation/mod.rs:218` and NOT visible from
`elidex-dom-api`. The SplitText handler + insertNode step-7 record construction
both live in `elidex-dom-api`, so the `..empty_record(..)` struct-update spread
will NOT compile cross-crate. **Resolution (decide in B1.3-ii per-slice plan)**:
the ideal is to **promote `empty_record` to `pub` and re-export it from the
`elidex-script-session` crate root** (One-issue-one-way — a single canonical
record-skeleton constructor shared by both the in-crate `apply_*` producers and
the `elidex-dom-api` manual-assembly sites), rather than open-coding full struct
literals at each `elidex-dom-api` site (which would duplicate the default-field
list and drift). B1.3-ii will apply this promotion.

childList record (only if `parent_opt.is_some()`):
```rust
MutationRecord {
    kind: MutationKind::ChildList,
    target: parent,
    added_nodes: vec![new_node],
    previous_sibling: Some(this),    // entity is new_node's prev sibling
    next_sibling: next_sibling_opt,  // pre-split next of `this`
    ..empty_record(MutationKind::ChildList, parent)  // empty_record promoted to pub (F1)
}
```

characterData record (always — splitText always mutates entity's data):
```rust
MutationRecord {
    kind: MutationKind::CharacterData,
    target: this,
    old_value: Some(old_data),   // pre-split full text (captured before split_text_at_offset)
    ..empty_record(MutationKind::CharacterData, this)
}
```

Push to session: `session.push_notify_record(char_data_record)` always, plus
`session.push_notify_record(childlist_record)` if `parent_opt.is_some()`. Record
order: childList (insertion) then characterData (head-truncation) to mirror the
`split_text_at_offset` internal order (insert at step 7.1 before truncate at step 8).

**Orphan splitText**: `split_text_at_offset` skips the insert step when entity has
no parent. Only the characterData record fires (entity's data changed). No childList
record (no parent to observe). An observer registered on entity itself with
`{characterData:true}` WOULD see the record — DOM §4.3.2 "queue a mutation record"
**step 2** sets `nodes` = the inclusive ancestors of target, which for an orphan
target is just `[target]`, so a registration on target itself is still iterated
(tree membership is not required).

### §4.2 Range::delete_contents boundary text-splice

**Current** (in `range/mutation.rs`, 3 sites verified 2026-06-28 lines 260/302/331):
- Same-container Text: `dom.replace_text_data(start_container, start_offset, count, "")` → no record
- Cross-container start text: `dom.replace_text_data(start_container, ...)` → no record
- Cross-container end text: `dom.replace_text_data(end_container, ...)` → no record

**B1.3-ii**: replace each `dom.replace_text_data` call with a record-producing pattern
(3 sites verified 2026-06-28: lines 162/189/200 in `range/mutation.rs` via
`grep -n "replace_text_data" crates/dom/elidex-dom-api/src/range/mutation.rs` →
12 total hits; 3 are in `delete_contents`, 3 in `extract_contents`, others in
`to_string`/`insert_node`/comments):
```rust
// F2: capture old_data via the dual-component helper (Text OR Comment), NOT a
// bare TextContent read — a Range boundary container can be a Comment node, and
// a TextContent-only read would silently yield old_value = "".
let old_data = get_char_data(node, dom).unwrap_or_default();
records.push(apply_replace_data(dom, node, offset, count, "", old_data));
```

These Range sites only reach `node`s that already passed an `is_text` /
`NodeKind::Text` guard in the surrounding code, so in practice the dual read
resolves to the Text branch — but routing through `get_char_data`
(`char_data_handlers.rs`, already TextContent+CommentData dispatching) keeps the
old-value capture correct-by-construction if a future Comment-boundary path is
added, and matches `apply_replace_data`'s own internal dual dispatch.

`apply_replace_data` is in `elidex-script-session` → importable from `elidex-dom-api` ✓
(same import path as `apply_append_child`, `apply_remove_child`). `get_char_data`
is a `pub(crate)` helper but lives in the **private** module `char_data_handlers`
(`mod char_data_handlers;` in `char_data/mod.rs`), so `range/mutation.rs` (a sibling
of `char_data`) **cannot** name `crate::char_data::char_data_handlers::get_char_data`
(E0603 module-private). **B1.3-ii must add a re-export**:
`pub(crate) use char_data_handlers::get_char_data;` in `char_data/mod.rs` (it already
`pub use`s the handler structs + `utf16_len` etc.). This is the same
private-visibility-reachability class as F1, one boundary down — surfaced by the
plan-review re-check, not at impl time.

Lock test `intra_single_text_delete_contents_delivers_no_record` must be UNLOCKED
and rewritten to assert ONE characterData record.

### §4.4 Range::insert_node step 7 ("insert a node into a live range", `#concept-range-insert` step 7)

**Current** (in `range/mutation.rs:insert_node`):
```rust
if let Ok(tail) = split_text_at_offset(start_container, start_offset, dom) {
    reference_node = Some(tail);
}
```
No record produced for the split.

**B1.3-ii**: capture pre-split state + produce records post-split, same as the
SplitText handler pattern (F1: `empty_record` promoted to `pub`; F2: `old_data`
via `get_char_data` dual read):
```rust
let pre_parent = dom.get_parent(start_container);
let pre_next = dom.get_next_sibling(start_container);
let old_data = get_char_data(start_container, dom).unwrap_or_default();
if let Ok(tail) = split_text_at_offset(start_container, start_offset, dom) {
    reference_node = Some(tail);
    // characterData record for head-truncation of start_container (split step 8)
    records.push(MutationRecord {
        kind: MutationKind::CharacterData,
        target: start_container,
        old_value: Some(old_data),
        ..empty_record(MutationKind::CharacterData, start_container)
    });
    // childList record for tail insertion (split step 7.1, only if parented)
    if let Some(parent) = pre_parent {
        records.push(MutationRecord {
            kind: MutationKind::ChildList,
            target: parent,
            added_nodes: vec![tail],
            previous_sibling: Some(start_container),
            next_sibling: pre_next,
            ..empty_record(MutationKind::ChildList, parent)
        });
    }
}
```

Note: `insert_node` already handles the step 12 insertion of `node` via
`apply_insert_before`/`apply_append_child` — that childList record is separate
from the step-7 split records above.

### §4.5 Record ordering (DOM §4.3 invariant)

DOM §4.3.2 "queue a mutation record" is called per-mutation in source order.
In "insert a node into a live range" (`#concept-range-insert`):
1. Step 7 split → push characterData (split step 8 head-truncation) + childList
   (split step 7.1 tail insertion) records
2. Step 12 pre-insert `node` → push childList record (node inserted), produced by
   the existing `apply_insert_before`/`apply_append_child` call

This ordering matches the spec's "source order" requirement since the split
(step 7) happens before the main insertion (step 12).

## ¬ß0 Scope decomposition (anchor: ideal, not incremental)

The `select`/`options` family is the last childList-mutating surface in B1.2b that is still VM-resident and MO-silent. It splits along **one coherent algorithm family** (the WHATWG `HTMLOptionsCollection` add/remove/length steps, which `HTMLSelectElement.add`/`remove(index)` delegate to verbatim per HTML `#dom-select-add` / `#dom-select-remove`):

- **B1.2b-2-select ‚Äî THIS slice.** `HTMLSelectElement.add(element, before?)` / `HTMLSelectElement.remove(index)` / `HTMLOptionsCollection.add(element, before?)` / `HTMLOptionsCollection.remove(index)` / `HTMLOptionsCollection.length`-setter ‚Üí records via the converged `invoke_dom_api ‚Üí dom-api handler ‚Üí apply_*` path. Plus `HTMLSelectElement.remove()` **no-arg** ‚Üí route to the existing `remove` (ChildNode) handler (it already records, #393) and **delete the VM inline duplicate** (One-issue-one-way).
- **Explicitly OUT (= B2 / other slices):** `selectedIndex` / `value` / `selectedness` setters mutate the `selected` **attribute**, not the child-list ‚Äî attribute-mutation = **B2** (`#11-tree-mutation-record-pipeline` attribute facet). `options` / `selectedOptions` **accessors** are reads (live collections), no mutation. `<datalist>`/`<optgroup>` have no add/remove tree-op API of their own.

**Why one slice (lens):** ideal-over-pragmatic + edge-dense decomposition. `select.add`/`options.add` already share the VM helper `select_add_impl`; `select.remove(idx)`/`options.remove` share `select_remove_option_at_impl`; the `length`-setter is a batch of the same primitives. They are ONE algorithm family with ONE engine-independent home ‚Äî splitting them would create a strangler (two record-paths for the same algorithm). This is the narrowly-scoped per-PR slice of the approved B1.2 umbrella = the base-case terminal unit (CLAUDE.md "Edge-dense ... base case").

## ¬ß1 Problem

`MutationObserver` does not observe any `select`/`options` tree mutation. Coupled defects, one root (identical shape to #393 / #399):

1. **MO-silent (the B1.2 gap).** `select_add_impl` (`html_select_proto.rs:790/797`) calls raw `EcsDom::insert_before`/`append_child`; `select_remove_option_at_impl` (`:849`) and the no-arg path (`:821`) call raw `EcsDom::remove_child`; `select_set_options_length_impl` (`:901/:907`) raw `append_child`/`remove_child`. **None** produce a `MutationRecord`.
2. **Layering violation (CLAUDE.md "VM host/ „ÅØ engine-bound Ë≤¨Âãô„ÅÆ„Åø").** The entire add/remove/length **algorithm** (ancestor-check, before-descendant validity, index‚Üínode resolution, batch grow/truncate) lives VM-side in `html_select_proto.rs`. No `select`/`options` dom-api handler exists (`registry.rs` has none; `elidex-form/select/mod.rs` holds only selectedness/value = **attribute** state, NOT tree ops). The algorithm is engine-independent and mis-placed.
3. **One-issue-one-way violation: `select.remove()` no-arg inline-duplicates `ChildNode.remove`.** `html_select_proto.rs:818-823` re-implements detach inline (`get_parent` + raw `remove_child`) instead of routing to the `remove` handler ‚Äî a second copy of an algorithm #393 already converged.
4. **Pre-existing spec bugs (found at plan-read, to fix in the canonical layer):**
   - **(a) `optgroup` wrongly rejected.** WebIDL `add((HTMLOptionElement or HTMLOptGroupElement) element, ...)` ‚Äî the union accepts `<optgroup>`. The VM (`:722-726`) throws `HierarchyRequestError` for any `tag != "option"`, rejecting a valid `<optgroup>`. (Also wrong **error type**: a non-option/optgroup arg is a WebIDL union-conversion failure = **TypeError**, not HierarchyRequestError.)
   - **(b) `length` cap deviation: engine 10,000 vs spec 100,000 ‚Äî ANCHORED DECISION = keep 10,000 as a documented engine constraint.** `select_set_options_length_impl` caps at `MAX_OPTIONS` (= `MAX_ANCESTOR_DEPTH` = 10,000); HTML `#dom-htmloptionscollection-length` setter step 2.1 caps at **100,000**. **This is NOT arbitrary**: the options `LiveCollection` walks children via `children_iter`, which truncates at `MAX_ANCESTOR_DEPTH` (10,000) ‚Äî options grown beyond that are unaddressable by `options.item`/`.length` (a deeper incoherence than a low cap). So the ideal-given-constraints decision is **keep 10,000**, preserve the spec's *over-cap ‚Üí return* shape (a value > the engine cap ‚Üí no-op return, mirroring the spec's >100,000 ‚Üí return), and register the deviation as an **unconditional** maintenance note (¬ß9). Raising to spec-100,000 is a separate cross-cutting effort (it must raise `MAX_ANCESTOR_DEPTH`-coupled walk limits together) ‚Äî out of scope here, NOT a silent cap.

**`appendChild`/`insertBefore`/`removeChild` prove the record path; #399 proves the marshalling/handler split.** Routing `select`/`options` through `invoke_dom_api` ‚Üí record-producing `apply_*` ‚Üí Phase 2.5 drain is the same mechanism, mirrored once more.

## ¬ß2 Coupled invariants at this corner (the design surface / edge matrix)

1. **WebIDL union `element` type** (WebIDL ¬ß3.2.25 Union types, `(HTMLOptionElement or HTMLOptGroupElement)`): option OR optgroup accepted; anything else ‚Üí **TypeError** (union-conversion failure). VM brand-check (resolve entity + detached distinction, like `require_element_arg`) + handler defensive tag-guard (`option`/`optgroup`, boa/wasm + DiD). Fixes inv-1.4(a).
2. **WebIDL union `before` `(HTMLElement or long)?`**: null/undefined ‚Üí append; HostObject wrapping an Element ‚Üí element reference; everything else ‚Üí `ToInt32` ‚Üí index. VM resolves the **union** (binding/marshalling, incl. `ToInt32`) ‚Üí passes the handler EITHER an element-ObjectRef OR a `JsValue::Number` (index) OR Null; the **handler** does spec step 5 (node ‚Üí reference, integer ‚Üí `index`th node via the options live-collection). Mirrors insertAdjacent (`where` coerced VM-side, algorithm in handler).
3. **`add` validity ordering** (HTML `#dom-htmloptionscollection-add`, verbatim): step 1 `element` is an ancestor of select ‚Üí **HierarchyRequestError**; step 2 `before` is an element not a descendant of select ‚Üí **NotFoundError**; step 3 `element == before` ‚Üí return (no-op); step 5 reference resolution; step 7 **pre-insert** `element` into parent before reference. Order matters (ancestor-check BEFORE descendant-check BEFORE no-op).
4. **Record shape & count** (inherited from B1/B1.2a, via `apply_*`): `add` of an already-parented option = **move = 2 records** (source-removal ¬ß4.5 adopt NOT suppressed + dest); fresh/orphan option = **1 record**. `remove(index)` = **1 record** (or 0 if out-of-range/empty ‚Äî spec "return"). `length`-setter is **asymmetric, matching the spec algorithms**: grow by *k* = **ONE coalesced record** (the ¬ß2.6.4.3 "append new option elements to a select" sub-algorithm builds a `DocumentFragment`, appends *k* options to it, then appends the fragment to the select ‚Äî one ¬ß4.2.3 fragment-insertion ‚Üí one childList record on the select with addedNodes = all *k*; the fragment-emptying record targets the unobserved transient fragment); truncate by *k* = **k remove records** ("remove the last *k* nodes from their parent nodes" ‚Äî the spec loops per-node). No new `apply_*` primitive (grow reuses `apply_append_child`'s fragment-expansion path). **AS-BUILT deviation from an earlier draft of ¬ß5.1**: the draft said grow = *k* append records ("spec loops per-node") ‚Äî that is correct for truncate but WRONG for grow, where the spec uses a fragment-append = one coalesced record. The implementation follows the spec (1 record for grow).
5. **`length` batch element allocation + leak discipline** (like insertAdjacentText): grow creates *k* bare `<option>` with the select's owner document (`create_element_with_owner`), each appended via `apply_append_child` (records). Allocate inside the loop AFTER the cap check (cap ‚Üí return before any alloc). Truncate removes the **last** *k* nodes (spec) via `apply_remove_child`.
6. **`remove(index)` bounds** (HTML "remove an option ... given index"): count 0 ‚Üí return; index `< 0` or `>= count` ‚Üí return (no-op, no record); else remove `index`th option. Out-of-range is a silent no-op, NOT an error.
7. **`select.remove()` no-arg = `ChildNode.remove`** (HTML `#dom-select-remove`): route to the existing `remove` handler via `invoke_dom_api("remove", select, &[])` ‚Üí records (#393); delete the inline duplicate (inv-1.3).
8. **Shared `select`‚Üî`options` algorithm**: `select.add`/`select.remove(idx)` "act like" the options-collection namesakes (`#dom-select-add`/`#dom-select-remove`) ‚Üí **ONE handler serves both** receivers (the receiver brand-check resolves to the same `<select>` entity / its options collection).
9. **Dual-runtime / leak-guard** (identical to #393/#399 inv. 7): handlers `push_notify_record` for all runtimes; the VM drains at `invoke_dom_api` Phase 2.5; boa/wasm never drain ‚Üí discarded by `SessionCore::flush` leak-guard. boa/wasm behaviour unchanged (records produced, unobserved); only the VM newly delivers.
10. **Index‚Üínode resolution uses the live options collection** (`CollectionFilter::Options`): both the `before` integer case (inv-2) and `remove(index)` resolve via `elidex_dom_api::LiveCollection` ‚Äî engine-independent, so it belongs in the handler (the VM `resolve_options_index` helper folds into the handler).

## ¬ß4 Design decision: new `element/select.rs` dom-api handler family (mirror #399)

The ideal (One-issue-one-way + layering-correct) resolution, mirroring #393/#399: **make the VM mirror the convergence.** Add a **new engine-independent handler module** `crates/dom/elidex-dom-api/src/element/select.rs` (NEW) (NOT `tree.rs`, which is at 992 lines ‚Äî a new module avoids crossing 1000 and keeps the select family cohesive) with handlers `OptionsAdd` / `OptionsRemove` / `OptionsSetLength` (registered in `registry.rs`), each producing records via the existing `apply_insert_before`/`apply_append_child`/`apply_remove_child` primitives. The VM natives (`html_select_proto.rs` + `html_options_collection.rs`) become **thin marshalling dispatchers**: brand-check receiver, resolve the WebIDL unions (element brand + `before` union + `ToInt32`/`ToUint32`), then `invoke_dom_api`.

**Rejected ‚Äî canonical home = `elidex-form`:** `elidex-form` owns **constraint validation + selectedness/value** (attribute state), not childList tree mutation. Putting tree ops there mixes two concerns; the B1.2b family canon is "childList algorithm ‚Üí `elidex-dom-api` handler ‚Üí `apply_*`". `elidex-form` stays the home for *selectedness* ‚Äî but note (plan-review Agent 2, verified) selectedness is a **pure derive-on-read query** (`elidex_form::select_selected_index`/`select_get_value` read the live `selected` attribute off the Options live-collection; `FormControlState.options` is a *creation-time* snapshot consumed only by interaction paths, already stale-by-design under any scripted mutation, current or post-convergence). There is **no cached selectedness component and no recompute hook**, so an option add/remove needs **no reconciliation** ‚Äî correctness is by construction (read-on-access), NOT a relink-driven recompute. (This corrects an earlier draft that mis-likened it to a `ConsumerDispatcher` recompute.)

**Rejected ‚Äî keep VM natives + teach each to push records:** perpetuates the double-impl + layering violation + leaves the optgroup/error-type/cap/no-arg-duplicate defects unfixed in the shared engine-indep path; rejected on philosophy.

This collapses all defects at once: records for free (Phase 2.5 drain), deletes the VM algorithm, restores layering, fixes the optgroup-union/error-type (handler tag-guard returns TypeError) + the no-arg `ChildNode.remove` duplicate, and folds `resolve_options_index` into the handler (single index‚Üínode home via the options live-collection).

**Crate-API / layering mapping:** VM marshalling `crates/script/elidex-js/src/vm/host/html_select_proto.rs` + `html_options_collection.rs` (thinned to dispatch) ‚Üí engine-independent algorithm home `crates/dom/elidex-dom-api/src/element/select.rs` (NEW) (`OptionsAdd`/`OptionsRemove`/`OptionsSetLength` + the index‚Üínode resolver) ‚Üí record-producing primitives `crates/script/elidex-script-session/src/mutation/mod.rs` (`apply_insert_before`/`apply_append_child`/`apply_remove_child`). No new crate, no new primitive. `select.remove()` no-arg ‚Üí existing `remove` handler.

## ¬ß5 Implementation

### 5.1 NEW dom-api handlers (`crates/dom/elidex-dom-api/src/element/select.rs` (NEW))
- **`OptionsAdd` (method `"options.add"`):** args = `[element_ref, before]` where `before` is an ObjectRef (element) | Number (index) | Null. Resolve `element` entity (handler defensive tag-guard `option`/`optgroup` ‚Üí TypeError on mismatch). HTML `#dom-htmloptionscollection-add` steps: (1) `element` ancestor-or-self of select ‚Üí HierarchyRequestError; (2) `before` is an element not a descendant of select ‚Üí NotFoundError; (3) `element == before` ‚Üí return element (no-op); (5) reference = `before` (element) OR `index`th option (Number, via options `LiveCollection`) OR null; (6-7) parent = reference.parent or select; route through `apply_insert_before(dom, parent, element, reference)` / `apply_append_child(dom, parent, element)` when reference is null; push all records (move = 2). Empty record list (non-fragment) = genuine failure ‚Üí HierarchyRequestError.
- **`OptionsRemove` (method `"options.remove"`):** arg = `[index:Number]`. Resolve `index`th option via options `LiveCollection`; count-0 / out-of-range ‚Üí `Ok(Undefined)` (no-op, no record); else `apply_remove_child(dom, parent, option)` + push record.
- **`OptionsSetLength` (method `"options.length.set"`):** arg = `[value:Number]` (VM `ToUint32`). Over-cap (`value > MAX_OPTIONS` = 10,000) ‚Üí no-op return (before any alloc). `current` = options count. grow (`value > current`): per ¬ß2.6.4.3 "append new option elements to a select given count", build a `DocumentFragment` (owner = select's node document), append `value-current` bare `<option>`s to it (raw, unobserved), then `apply_append_child(dom, select, fragment)` ‚Üí **one** coalesced record (push all returned records ‚Äî the destination record on select carries all added options; the fragment-emptying record targets the unobserved fragment), then free the now-empty transient fragment (`destroy_entity`, B1.2b leak discipline). truncate (`value < current`): remove the **last** `current-value` options (rev) via `apply_remove_child` + push each (per-node = *k* records).
- Shared helper `options_index_to_node(select, index, dom) -> Option<Entity>` (folds VM `resolve_options_index`).

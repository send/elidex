> **Terminology.** "move" in this memo = the **legacy non-atomic reinsertion** path ‚Äî `appendChild`/
> `insertBefore`/`replaceChild` applied to an already-parented node, which runs adopt‚Üíremove‚Üíinsert (DOM ¬ß4.5

## ¬ß0. Decisions this memo commits to (lens-collapsed; ¬ß7 for the gate to falsify)

1. **A move emits TWO records, not one (¬ß2).** Per spec: `appendChild`/`insertBefore` of a parented node runs
   `adopt` (¬ß4.5 step 2 "remove node", default `suppressObservers=false`) ‚Üí ¬ß4.2.3 remove step 16 queues a
   **removal record on the source parent**, then insert step 8 queues the **insertion record on the
   destination parent**. Same-parent reorder included (adopt removes from the same parent first). This is the
   spec-correct shape Codex PR379 R2/R3 pointed at; B1's single-dest-record reading was rejected as wrong, and
   "emit nothing" (B1's deferral) is now replaced by the correct two-record emission.

2. **Siblings captured at the right instants (¬ß2.2).** Source-removal record's `previousSibling`/`nextSibling`
   = the node's siblings in the **old** parent, captured **before** the move. Destination-insertion record's
   siblings = the node's siblings in the **new** parent, captured **after** the relink (this also fixes B1's
   R1 degenerate `previousSibling==self` case by construction ‚Äî post-apply the node is never its own sibling).

3. **Records produced at the engine-independent algorithm seam; `vm/host/` stays marshalling-only (Layering).**
   The two records are built in `elidex-script-session::apply_*` (or a move-aware builder) and pushed onto
   `session.notify_records`; the existing `dom_bridge` drain + `queue_mutation_record` deliver them unchanged.
   No `vm/host/` algorithm.

4. **Reuse B1's delivery mechanism verbatim.** `queue_mutation_record` already handles N independent records
   targeting different parents (each does its own ¬ß4.3.2 ancestor walk via `registry.notify`). No delivery,
   microtask, or registry change. Only the producer (the `record_deferred_to_b1_2` guard ‚Üí move-record
   emission) changes.

5. **Fragment + non-move scope unchanged.** Fresh single-node inserts keep B1's behavior. Fragments stay
   deferred (apply-layer fix is a separate slice ‚Äî scope note above). The `record_deferred_to_b1_2` guard is
   **narrowed** from "move OR fragment ‚Üí defer" to "fragment ‚Üí defer". After narrowing, the guard is
   **orthogonal to move dispatch** ‚Äî it no longer carries move-ness; the move/fresh decision is the builder's
   `old_parent.is_some()` (¬ß2.3), not this guard.

## ¬ß2. Design ‚Äî the two-record move

### 2.1 Order + targets (¬ß4.5 + ¬ß4.2.3)
A move (`node` already parented) = remove-from-source **then** insert-into-dest:
1. **Source-removal record** ‚Äî `{type: childList, target: oldParent, removedNodes:[node], previousSibling:
   <node's old prev>, nextSibling: <node's old next>}` (adopt ¬ß4.5 step 2 ‚Üí remove ¬ß4.2.3 step 16).
2. **Destination-insertion record** ‚Äî `{type: childList, target: newParent, addedNodes:[node],
   previousSibling/nextSibling: <node's NEW siblings>}` (insert ¬ß4.2.3 step 8).
Pushed in that order onto `session.notify_records`; delivered in order by the existing microtask.
Same-parent reorder: `oldParent == newParent` but still two records (removal then insertion) ‚Äî spec-faithful.

### 2.2 Sibling capture timing ‚Äî **node-relative, dest = post-write (uniform for fresh+move)**

The move-aware builder, given `(parent, node)`:
1. **Before** the EcsDom write ‚Äî *move source context only* (the write destroys it): `old_parent =
   get_parent(node)` (this `is_some()` is also the **move-vs-fresh SoT** ‚Äî see ¬ß2.3); on a move also
   `old_prev = prev_exposed_sibling(node)`, `old_next = next_exposed_sibling(node)` for the source-removal
   record.
2. Apply via `EcsDom::append_child` / `insert_before` (`detach_with_hook` + relink).
3. **After** the write ‚Äî *dest siblings, captured **node-relative**: `prev_exposed_sibling(node)` /
   `next_exposed_sibling(node)`*. This is correct for **both** fresh and move (append ‚Üí prev = new
   last-but-one, next = None; insertBefore ‚Üí prev = node's new prev, next = refChild) and fixes B1's R1
   degenerate `previousSibling == node` by construction (post-apply the node is never its own sibling).
4. Build + return the record list (1 fresh / 2 move).

‚öÝ **Capture-shape change from the existing helpers (the refactor is NOT a return-type widening).** The
current `apply_append_child` captures the dest prev **pre-write, parent-relative**
(`children_iter_rev(parent).next()`, `mutation/mod.rs:230`) and `apply_insert_before` captures it **pre-write,
ref-relative** (`prev_exposed_sibling(ref_child)`, `:254`). For a **fresh** node these pre-write captures
coincide with the post-write node-relative value (the node is not yet in the tree), which is why B1 was
correct; for a **move** they snapshot the *stale* pre-detach chain (‚Üí self-sibling). The builder therefore
**removes the pre-write dest capture entirely** and uses the uniform post-write node-relative capture above ‚Äî
One-issue-one-way (one capture rule for both cases), not a parallel move-only branch.

### 2.3 Builder shape + move-vs-fresh dispatch signal
`apply_append_child` / `apply_insert_before` currently return `Option<MutationRecord>` (one). Chosen shape:
**extend the existing `apply_*` to return a record list** (`SmallVec<[MutationRecord; 2]>` / `Vec`; one record
source, fresh = 1-elem, move = 2-elem) ‚Äî One-issue-one-way, the flush path (`apply_mutation`) and the boa-flush
path inherit correct move records for free. (Rejected: a dedicated `apply_move_*` ‚Äî it forks the producer into
two sites, the decision-tax shape CLAUDE.md warns against.)

**Move-vs-fresh dispatch is decided *inside the builder*, by `old_parent = get_parent(node)` captured
pre-write (¬ß2.2 step 1): `Some` ‚Üí move (2 records), `None` ‚Üí fresh (1 record).** This is the move-ness SoT.
The handler does **not** dispatch on move-ness ‚Äî after ¬ß4 narrows `record_deferred_to_b1_2` to **fragment-only**
(¬ß0.5 decision 5), that guard reports fragment-ness, not move-ness, and is orthogonal to the move/fresh split.
The handler's only job is: if the (fragment-only) guard says defer, withhold; otherwise push the whole
returned list. (A prior draft said "the handler computes `is_move` from the narrowed guard" ‚Äî that was wrong:
the narrowed guard is `false` for every move, so branching the handler on it would silently drop all move
records. Corrected here.)

### 2.4 replaceChild ‚Äî full record set (resolves ¬ß7 Q2, derived from `#concept-node-replace`)

**replace does NOT use ¬ß2.2's append/insertBefore capture rule** for its coalesced record ‚Äî replace has its own
spec capture discipline (steps 7‚Äì9, all read **pre-removal, oldChild-relative**). ¬ß2.2's uniform post-write
node-relative rule applies to `apply_append_child` / `apply_insert_before` only.

For `replaceChild(newChild, oldChild)` with **`newChild` already parented** (a move into the replace slot),
B1.2a emits **two** records (B1 deferred the whole replace record for any move; B1.2a un-defers it):

1. **Source-removal record** on `newChild`'s OLD parent ‚Äî fires when `newChild` is already parented (a real
   move). **Self-replace `replaceChild(X,X)` is NOT a move and produces no record at all**: `EcsDom::replace_child`
   rejects `new == old` (`tree/mutation.rs:139` ‚Üí `false`) and the VM `ReplaceChild` handler additionally
   early-returns it as a browser-parity no-op (`element/tree.rs`), so `apply_replace_child` returns an empty list.
   (This is a **pre-existing elidex deviation** from spec step 14, which would queue a coalesced
   `{added:[X],removed:[X]}` record ‚Äî the no-op predates B1.2a and is **out of B1.2a scope**; B1.2a does not change
   it. Consequently the source-record context needs **no `!= old_child` gate** ‚Äî the chokepoint rejection is the
   single source of truth.) Step 13 inserts `newChild` with
   `suppressObservers=true`, but the insert's internal **adopt** (`#concept-node-insert` step 7.1) calls "remove
   node" on `newChild` from its old parent with the **default** `suppressObservers=false` (suppression is NOT
   threaded into adopt's remove) ‚Üí a removal record fires on `newChild`'s old parent. Shape: `{removed:[newChild],
   previousSibling/nextSibling = newChild's OLD exposed siblings}`, captured **pre-write** (the forthcoming
   `dom.replace_child` ‚Üí `detach_with_hook(new_child, parent)` at `tree/mutation.rs:164` destroys this ‚Äî so
   `apply_replace_child` must read `get_parent(new_child)` + `prev/next_exposed_sibling(new_child)` **before** it,
   exactly as ¬ß2.2 step 1 does for append/insert). **This record is NEW in B1.2a** ‚Äî `apply_replace_child`
   currently captures nothing about `new_child`'s old position.
2. **Coalesced replace record** on `parent` (step 14): `{added:[newChild], removed:[oldChild],
   previousSibling, nextSibling = referenceChild}`. Capture per spec, **pre-removal, oldChild-relative**:
   - `previousSibling` = step 9 = oldChild's previous sibling. **No step-9 adjustment exists** ‚Üí this MAY legitimately
     equal `newChild` (e.g. `[A,B,C].replaceChild(A,B)` ‚Üí `previousSibling = A = newChild`). Spec-faithful ‚Äî the
     existing `prev_exposed_sibling(old_child)` pre-write capture is **correct; do NOT "fix" it** to post-write.
   - `nextSibling` = `referenceChild` = step 7 (oldChild's next sibling) **with the step-8 adjustment: if that next
     sibling IS `newChild` (newChild was oldChild's next sibling, e.g. `[A,B,C].replaceChild(C,B)` ‚Üí next(B)=C=newChild),
     set referenceChild to `newChild`'s next sibling instead.** Step 8 fires only on a **move** (a fresh newChild can't be
     oldChild's sibling), so it was unobservable in B1 (move-replace deferred) and **first becomes live in B1.2a**.
     The existing `next_exposed_sibling(old_child)` capture lacks this adjustment ‚Üí B1.2a must add it.

**Same-parent move** (`newChild`'s old parent == `parent`): both records target `parent`; order = source-removal
(from step 13's adopt) **then** coalesced (step 14 queues after). Two independent ¬ß4.3.2 records, same target ‚Äî no
coalescing between them.

‚öÝ **Same-parent source-removal siblings must exclude `oldChild` (spec step ordering vs. EcsDom primitive
ordering).** Per spec, `oldChild` is removed at **step 11**, *before* `newChild`'s adopt at **step 13** ‚Äî so at
adopt time `newChild`'s siblings no longer include `oldChild`. Example `[A,B,C].replaceChild(C,B)`: step 11
removes B ‚Üí tree `[A,C]`; step 13's adopt-removal of C sees `previousSibling = A` (NOT B). But the EcsDom
primitive `replace_child` detaches `new_child` at `tree/mutation.rs:164` **before** detaching `old_child` at
`:186` (reverse order), so a naive pre-write `prev_exposed_sibling(new_child)` would capture **B**. Therefore the
source-removal record's siblings = `newChild`'s exposed prev/next **stepping over `oldChild`** (if a captured
sibling == `oldChild`, use `oldChild`'s sibling on that side). For a **cross-parent** move `oldChild` is not in
`newChild`'s parent, so the exclusion is a no-op ‚Äî uniform rule, correct for both.

When `newChild` is **fresh** (not parented), B1 behavior is unchanged: one coalesced record, step-8 never fires,
no source record.

### 2.5 Component data-flow (read / write balance)

Every read the new code performs and the relink write-path that populates it ‚Äî Sub-check 2b ledger:

| Read | When | Anchor | Populated by |
|---|---|---|---|
| `get_parent(node)` (move SoT + source target) | pre-write | node | existing `Relations` parent component (relink-maintained) |
| `prev_exposed_sibling(node)` / `next_exposed_sibling(node)` (append/insert source-removal siblings) | pre-write | node | existing exposed-sibling helpers (skip `ShadowRoot` per ¬ß4.8) |
| `prev_exposed_sibling(node)` / `next_exposed_sibling(node)` (append/insert dest siblings) | **post-write** | node | same helpers, read after `link_node` + `rev_version` complete |
| `get_parent(new_child)` (replace move SoT ‚Äî record (1) fires when `is_some()`; self-replace rejected by `EcsDom::replace_child`, ¬ß2.4-1) | **pre-write** | new_child | existing `Relations` parent component |
| `prev/next_exposed_sibling(new_child)` **stepping over `old_child`** (replace source-removal siblings, ¬ß2.4-1) | **pre-write** | new_child | same; MUST precede `dom.replace_child` (`detach_with_hook` at `tree/mutation.rs:164`); old_child-exclusion = spec step-11-before-13 ordering |
| `prev_exposed_sibling(old_child)` / `next_exposed_sibling(old_child)` (replace coalesced record, ¬ß2.4-2; +step-8 adjust on next) | pre-write | old_child | existing (B1) ‚Äî spec steps 7‚Äì9 read pre-removal oldChild-relative |

All reads consume pre-existing EcsDom relink state ‚Äî **no new write-path** is introduced (the apply primitives
`EcsDom::append_child` / `insert_before` / `replace_child` already do `detach_with_hook` + `link_node` +
`rev_version`). B1.2a changes only *when* / *relative to what* siblings are read (¬ß2.2 append/insert) and adds the
replace source-removal capture + step-8 referenceChild adjustment (¬ß2.4).

## ¬ß5. Testing / AC
1. Same-parent reorder (`parent.appendChild(parent.firstChild)`): exactly 2 childList records (removal then
   insertion) on `parent`, siblings correct, no `previousSibling==node`.
2. Cross-parent move: source observer gets removal record (target=oldParent, removedNodes=[node]); dest
   observer gets insertion record (target=newParent, addedNodes=[node]).

> **Terminology.** "move" in this memo = the **legacy non-atomic reinsertion** path — `appendChild`/
> `insertBefore`/`replaceChild` applied to an already-parented node, which runs adopt→remove→insert (DOM §4.5
> step 2 + §4.2.3). It is **NOT** the spec's atomic "move a node" algorithm (DOM §4.2.3 `#move`, used by
> `moveBefore()` `#dom-parentnode-movebefore`), which deliberately does *not* remove-then-insert and queues different records. `moveBefore`
> atomic-move is out of scope (its own future slice).

> **What B1.2a builds.** B1 covers `MutationObserver` childList records for **fresh single-node** inserts and
> **defers moves** (`record_deferred_to_b1_2`, `element/tree.rs:29`) — a node that already has a parent is
> applied but its record withheld. B1.2a **completes the move case**: `appendChild`/`insertBefore`/
> `replaceChild` of an already-parented node emits the **two** WHATWG DOM records a move owes — a
> **source-parent removal** record (from `adopt`) + a **destination-parent insertion** record — with correct
> pre/post siblings, instead of dropping them. It reuses B1's mechanism verbatim (`queue_mutation_record` +
> §4.3 microtask + `MutationObserverRegistry`); only the record **production** at the three insert handlers
> changes.

> **Explicitly NOT in B1.2a (separate slices).**
> - **DocumentFragment expansion** — `EcsDom::append_child` (`tree/mutation.rs:22`) does **not** expand a
>   fragment: it would link the fragment *entity* as a child (an **apply-layer** bug, latent because B1 defers
>   the fragment record). Fixing it needs an `elidex-ecs` apply change (expand the fragment's children) +
>   multi-node record — a different layer/scope. **Stays deferred** (the B1 guard keeps emitting no malformed
>   record). → own slice (B1.2-fragment).
> - **direct tree ops** (ParentNode/ChildNode/insertAdjacent/select) → B1.2b. **replace-all coalescing** →
>   B1.2c. **Range / transient observers** → B1.2d. **CE-reaction timing across moves** = Mechanism A
>   (`ConsumerDispatcher`), untouched by B1.2a (MO-record-only).
>
> These five (B1.2-fragment / B1.2b / B1.2c / B1.2d / Mechanism A) are **sibling implementation slices of the
> active B1.2 umbrella program** (refining the B1 plan §5 "Follow-on slices" table), **not** open-ended
> `#11-*` defer slots — so the defer-ledger 3-element form (Why / Re-evaluation trigger / date) is N/A; each
> is a tracked next-PR slice under the umbrella, each with its own `/elidex-plan-review`.

> **Runtime note.** As with B1, the VM MutationObserver path is production-dormant (the shell runs boa) until
> S5; B1.2a is validated by VM tests.

## §0. Decisions this memo commits to (lens-collapsed; §7 for the gate to falsify)

1. **A move emits TWO records, not one (§2).** Per spec: `appendChild`/`insertBefore` of a parented node runs
   `adopt` (§4.5 step 2 "remove node", default `suppressObservers=false`) → §4.2.3 remove step 16 queues a
   **removal record on the source parent**, then insert step 8 queues the **insertion record on the
   destination parent**. Same-parent reorder included (adopt removes from the same parent first). This is the
   spec-correct shape Codex PR379 R2/R3 pointed at; B1's single-dest-record reading was rejected as wrong, and
   "emit nothing" (B1's deferral) is now replaced by the correct two-record emission.

2. **Siblings captured at the right instants (§2.2).** Source-removal record's `previousSibling`/`nextSibling`
   = the node's siblings in the **old** parent, captured **before** the move. Destination-insertion record's
   siblings = captured **pre-adopt** per DOM §4.2.3 insert step 6 (append → parent's last child; insertBefore →
   refChild's prev). For a no-position-change move this `previousSibling` is the moved node **itself** — a
   spec-mandated self-sibling (§2.2; Codex PR384 R1 corrected an earlier post-write draft).

3. **Records produced at the engine-independent algorithm seam; `vm/host/` stays marshalling-only (Layering).**
   The two records are built in `elidex-script-session::apply_*` (or a move-aware builder) and pushed onto
   `session.notify_records`; the existing `dom_bridge` drain + `queue_mutation_record` deliver them unchanged.
   No `vm/host/` algorithm.

4. **Reuse B1's delivery mechanism verbatim.** `queue_mutation_record` already handles N independent records
   targeting different parents (each does its own §4.3.2 ancestor walk via `registry.notify`). No delivery,
   microtask, or registry change. Only the producer (the `record_deferred_to_b1_2` guard → move-record
   emission) changes.

5. **Fragment + non-move scope unchanged.** Fresh single-node inserts keep B1's behavior. Fragments stay
   deferred (apply-layer fix is a separate slice — scope note above). The `record_deferred_to_b1_2` guard is
   **narrowed** from "move OR fragment → defer" to "fragment → defer". After narrowing, the guard is
   **orthogonal to move dispatch** — it no longer carries move-ness; the move/fresh decision is the builder's
   `old_parent.is_some()` (§2.3), not this guard.

## §2. Design — the two-record move

### 2.1 Order + targets (§4.5 + §4.2.3)
A move (`node` already parented) = remove-from-source **then** insert-into-dest:
1. **Source-removal record** — `{type: childList, target: oldParent, removedNodes:[node], previousSibling:
   <node's old prev>, nextSibling: <node's old next>}` (adopt §4.5 step 2 → remove §4.2.3 step 16).
2. **Destination-insertion record** — `{type: childList, target: newParent, addedNodes:[node],
   previousSibling/nextSibling: <node's NEW siblings>}` (insert §4.2.3 step 8).
Pushed in that order onto `session.notify_records`; delivered in order by the existing microtask.
Same-parent reorder: `oldParent == newParent` but still two records (removal then insertion) — spec-faithful.

### 2.2 Sibling capture timing — **dest = PRE-adopt, per DOM §4.2.3 insert step 6** (Codex PR384 R1 correction)

> **⚠ Reversal of this memo's original design.** An earlier draft captured the *destination* siblings
> **post-write, node-relative** to "fix" a self-sibling `previousSibling == node`. That was **wrong**: DOM
> §4.2.3 insert **step 6** ("Let previousSibling be child's previous sibling or parent's last child if child is
> null") is evaluated **before** the adopt at **step 7.1**, so for a no-position-change move
> (`appendChild(<current last child>)`, `insertBefore(node, node.nextSibling)`) the spec record's
> `previousSibling` **is the moved node itself** — a spec-mandated self-sibling, not a bug. Codex caught the
> divergence; B1.2a reverts to pre-adopt capture (= B1's original approach).

The move-aware builder, given `(parent, node)`:
1. **Before** the EcsDom write — *both* the move source context AND the destination siblings (the write
   destroys both):
   - source (move only): `old_parent = get_parent(node)` (its `is_some()` is the move-vs-fresh SoT, §2.3); plus
     `old_prev = prev_exposed_sibling(node)`, `old_next = next_exposed_sibling(node)` for the source-removal record.
   - destination (step 6): `appendChild` → `previousSibling` = parent's exposed last child
     (`children_iter_rev(parent).next()`, skips `ShadowRoot`), `nextSibling` = null (append reference child is
     null); `insertBefore` → `previousSibling` = `prev_exposed_sibling(ref_child)`, `nextSibling` = `ref_child`.
2. Apply via `EcsDom::append_child` / `insert_before` (`detach_with_hook` + relink).
3. Build + return the record list (1 fresh / 2 move).

This is exactly B1's original dest capture (pre-write, parent-/ref-relative) — correct for fresh AND move,
including the spec self-sibling for the degenerate no-move case. B1.2a adds **only** the source-removal record
(step 1 source); it does **not** change the dest capture.

### 2.3 Builder shape + move-vs-fresh dispatch signal
`apply_append_child` / `apply_insert_before` currently return `Option<MutationRecord>` (one). Chosen shape:
**extend the existing `apply_*` to return a record list** (`SmallVec<[MutationRecord; 2]>` / `Vec`; one record
source, fresh = 1-elem, move = 2-elem) — One-issue-one-way, the flush path (`apply_mutation`) and the boa-flush
path inherit correct move records for free. (Rejected: a dedicated `apply_move_*` — it forks the producer into
two sites, the decision-tax shape CLAUDE.md warns against.)

**Move-vs-fresh dispatch is decided *inside the builder*, by `old_parent = get_parent(node)` captured
pre-write (§2.2 step 1): `Some` → move (2 records), `None` → fresh (1 record).** This is the move-ness SoT.
The handler does **not** dispatch on move-ness — after §4 narrows `record_deferred_to_b1_2` to **fragment-only**
(§0.5 decision 5), that guard reports fragment-ness, not move-ness, and is orthogonal to the move/fresh split.
The handler's only job is: if the (fragment-only) guard says defer, withhold; otherwise push the whole
returned list. (A prior draft said "the handler computes `is_move` from the narrowed guard" — that was wrong:
the narrowed guard is `false` for every move, so branching the handler on it would silently drop all move
records. Corrected here.)

### 2.4 replaceChild — full record set (resolves §7 Q2, derived from `#concept-node-replace`)

Replace has its own spec capture discipline (steps 7–9, all read **pre-removal, oldChild-relative**) — consistent
with §2.2's pre-adopt rule for append/insertBefore (both capture the destination record's siblings *before* the
adopt/remove), but keyed on `old_child` rather than the appended node's reference.

For `replaceChild(newChild, oldChild)` with **`newChild` already parented** (a move into the replace slot),
B1.2a emits **two** records (B1 deferred the whole replace record for any move; B1.2a un-defers it):

1. **Source-removal record** on `newChild`'s OLD parent — fires when `newChild` is already parented (a real
   move). **Self-replace `replaceChild(X,X)` is NOT a move and produces no record at all**: `EcsDom::replace_child`
   rejects `new == old` (`tree/mutation.rs:139` → `false`) and the VM `ReplaceChild` handler additionally
   early-returns it as a browser-parity no-op (`element/tree.rs`), so `apply_replace_child` returns an empty list.
   (This is a **pre-existing elidex deviation** from spec step 14, which would queue a coalesced
   `{added:[X],removed:[X]}` record — the no-op predates B1.2a and is **out of B1.2a scope**; B1.2a does not change
   it. Consequently the source-record context needs **no `!= old_child` gate** — the chokepoint rejection is the
   single source of truth.) Step 13 inserts `newChild` with
   `suppressObservers=true`, but the insert's internal **adopt** (`#concept-node-insert` step 7.1) calls "remove
   node" on `newChild` from its old parent with the **default** `suppressObservers=false` (suppression is NOT
   threaded into adopt's remove) → a removal record fires on `newChild`'s old parent. Shape: `{removed:[newChild],
   previousSibling/nextSibling = newChild's OLD exposed siblings}`, captured **pre-write** (the forthcoming
   `dom.replace_child` → `detach_with_hook(new_child, parent)` at `tree/mutation.rs:164` destroys this — so
   `apply_replace_child` must read `get_parent(new_child)` + `prev/next_exposed_sibling(new_child)` **before** it,
   exactly as §2.2 step 1 does for append/insert). **This record is NEW in B1.2a** — `apply_replace_child`
   currently captures nothing about `new_child`'s old position.
2. **Coalesced replace record** on `parent` (step 14): `{added:[newChild], removed:[oldChild],
   previousSibling, nextSibling = referenceChild}`. Capture per spec, **pre-removal, oldChild-relative**:
   - `previousSibling` = step 9 = oldChild's previous sibling. **No step-9 adjustment exists** → this MAY legitimately
     equal `newChild` (e.g. `[A,B,C].replaceChild(A,B)` → `previousSibling = A = newChild`). Spec-faithful — the
     existing `prev_exposed_sibling(old_child)` pre-write capture is **correct; do NOT "fix" it** to post-write.
   - `nextSibling` = `referenceChild` = step 7 (oldChild's next sibling) **with the step-8 adjustment: if that next
     sibling IS `newChild` (newChild was oldChild's next sibling, e.g. `[A,B,C].replaceChild(C,B)` → next(B)=C=newChild),
     set referenceChild to `newChild`'s next sibling instead.** Step 8 fires only on a **move** (a fresh newChild can't be
     oldChild's sibling), so it was unobservable in B1 (move-replace deferred) and **first becomes live in B1.2a**.
     The existing `next_exposed_sibling(old_child)` capture lacks this adjustment → B1.2a must add it.

**Same-parent move** (`newChild`'s old parent == `parent`): both records target `parent`; order = source-removal
(from step 13's adopt) **then** coalesced (step 14 queues after). Two independent §4.3.2 records, same target — no
coalescing between them.

⚠ **Same-parent source-removal siblings must exclude `oldChild` (spec step ordering vs. EcsDom primitive
ordering).** Per spec, `oldChild` is removed at **step 11**, *before* `newChild`'s adopt at **step 13** — so at
adopt time `newChild`'s siblings no longer include `oldChild`. Example `[A,B,C].replaceChild(C,B)`: step 11
removes B → tree `[A,C]`; step 13's adopt-removal of C sees `previousSibling = A` (NOT B). But the EcsDom
primitive `replace_child` detaches `new_child` at `tree/mutation.rs:164` **before** detaching `old_child` at
`:186` (reverse order), so a naive pre-write `prev_exposed_sibling(new_child)` would capture **B**. Therefore the
source-removal record's siblings = `newChild`'s exposed prev/next **stepping over `oldChild`** (if a captured
sibling == `oldChild`, use `oldChild`'s sibling on that side). For a **cross-parent** move `oldChild` is not in
`newChild`'s parent, so the exclusion is a no-op — uniform rule, correct for both.

When `newChild` is **fresh** (not parented), B1 behavior is unchanged: one coalesced record, step-8 never fires,
no source record.

### 2.5 Component data-flow (read / write balance)

Every read the new code performs and the relink write-path that populates it — Sub-check 2b ledger:

| Read | When | Anchor | Populated by |
|---|---|---|---|
| `get_parent(node)` (move SoT + source target) | pre-write | node | existing `Relations` parent component (relink-maintained) |
| `prev_exposed_sibling(node)` / `next_exposed_sibling(node)` (append/insert source-removal siblings) | pre-write | node | existing exposed-sibling helpers (skip `ShadowRoot` per §4.8) |
| append dest `previousSibling` = `children_iter_rev(parent).next()` (§4.2.3 step 6 = parent's last child, may == node) | **pre-write** | parent | existing exposed last-child walk (skips `ShadowRoot`) |
| insert dest `previousSibling` = `prev_exposed_sibling(ref_child)` (§4.2.3 step 6, may == node); `nextSibling` = `ref_child` | **pre-write** | ref_child | existing exposed-sibling helper |
| `get_parent(new_child)` (replace move SoT — record (1) fires when `is_some()`; self-replace rejected by `EcsDom::replace_child`, §2.4-1) | **pre-write** | new_child | existing `Relations` parent component |
| `prev/next_exposed_sibling(new_child)` **stepping over `old_child`** (replace source-removal siblings, §2.4-1) | **pre-write** | new_child | same; MUST precede `dom.replace_child` (`detach_with_hook` at `tree/mutation.rs:164`); old_child-exclusion = spec step-11-before-13 ordering |
| `prev_exposed_sibling(old_child)` / `next_exposed_sibling(old_child)` (replace coalesced record, §2.4-2; +step-8 adjust on next) | pre-write | old_child | existing (B1) — spec steps 7–9 read pre-removal oldChild-relative |

All reads consume pre-existing EcsDom relink state — **no new write-path** is introduced (the apply primitives
`EcsDom::append_child` / `insert_before` / `replace_child` already do `detach_with_hook` + `link_node` +
`rev_version`), and **all are captured pre-write** (matching §4.2.3 insert step 6 / replace steps 7–9). B1.2a's
only new reads are the **source-removal** record's siblings (append/insert via `capture_move_source`; replace
stepping over `old_child`) + the replace step-8 referenceChild adjustment (§2.4). The dest record's capture is
unchanged from B1.

## §5. Testing / AC
1. Same-parent reorder (`parent.appendChild(parent.firstChild)`): exactly 2 childList records (removal then
   insertion) on `parent`, siblings correct, no `previousSibling==node`.
2. Cross-parent move: source observer gets removal record (target=oldParent, removedNodes=[node]); dest
   observer gets insertion record (target=newParent, addedNodes=[node]).
3. `insertBefore` move: insertion record `nextSibling==refChild`.
4. **replaceChild move**: (a) cross-parent — source-removal on newChild's old parent + coalesced record on
   parent; (b) `[A,B,C].replaceChild(C,B)` (same-parent, newChild adjacent-after) — coalesced `nextSibling` =
   C's original next (step-8 adjust), NOT C itself, **and** the source-removal record's `previousSibling` = A

> **What B1.2a builds.** B1 covers `MutationObserver` childList records for **fresh single-node** inserts and
> **defers moves** (`record_deferred_to_b1_2`, `element/tree.rs:29`) — a node that already has a parent is
> applied but its record withheld. B1.2a **completes the move case**: `appendChild`/`insertBefore`/
> `replaceChild` of an already-parented node emits the **two** WHATWG DOM records a move owes — a
> **source-parent removal** record (from `adopt`) + a **destination-parent insertion** record — with correct
> pre/post siblings, instead of dropping them. It reuses B1's mechanism verbatim (`queue_mutation_record` +
> §4.3 microtask + `MutationObserverRegistry`); only the record **production** at the three insert handlers
> changes.

> **Explicitly NOT in B1.2a (separate slices).**
> - **DocumentFragment expansion** — `EcsDom::append_child` (`tree/mutation.rs:22`) does **not** expand a
>   fragment: it would link the fragment *entity* as a child (an **apply-layer** bug, latent because B1 defers
>   the fragment record). Fixing it needs an `elidex-ecs` apply change (expand the fragment's children) +
>   multi-node record — a different layer/scope. **Stays deferred** (the B1 guard keeps emitting no malformed
>   record). → own slice (B1.2-fragment).
> - **direct tree ops** (ParentNode/ChildNode/insertAdjacent/select) → B1.2b. **replace-all coalescing** →
>   B1.2c. **Range / transient observers** → B1.2d. **CE-reaction timing across moves** = Mechanism A
>   (`ConsumerDispatcher`), untouched by B1.2a (MO-record-only).

> **Runtime note.** As with B1, the VM MutationObserver path is production-dormant (the shell runs boa) until
> S5; B1.2a is validated by VM tests.

## §0. Decisions this memo commits to (lens-collapsed; §7 for the gate to falsify)

1. **A move emits TWO records, not one (§2).** Per spec: `appendChild`/`insertBefore` of a parented node runs
   `adopt` (§4.5 step 2 "remove node", default `suppressObservers=false`) → §4.2.3 remove step 16 queues a
   **removal record on the source parent**, then insert step 8 queues the **insertion record on the
   destination parent**. Same-parent reorder included (adopt removes from the same parent first). This is the
   spec-correct shape Codex PR379 R2/R3 pointed at; B1's single-dest-record reading was rejected as wrong, and
   "emit nothing" (B1's deferral) is now replaced by the correct two-record emission.

2. **Siblings captured at the right instants (§2.2).** Source-removal record's `previousSibling`/`nextSibling`
   = the node's siblings in the **old** parent, captured **before** the move. Destination-insertion record's
   siblings = the node's siblings in the **new** parent, captured **after** the relink (this also fixes B1's
   R1 degenerate `previousSibling==self` case by construction — post-apply the node is never its own sibling).

3. **Records produced at the engine-independent algorithm seam; `vm/host/` stays marshalling-only (Layering).**
   The two records are built in `elidex-script-session::apply_*` (or a move-aware builder) and pushed onto
   `session.notify_records`; the existing `dom_bridge` drain + `queue_mutation_record` deliver them unchanged.
   No `vm/host/` algorithm.

4. **Reuse B1's delivery mechanism verbatim.** `queue_mutation_record` already handles N independent records
   targeting different parents (each does its own §4.3.2 ancestor walk via `registry.notify`). No delivery,
   microtask, or registry change. Only the producer (the `record_deferred_to_b1_2` guard → move-record
   emission) changes.

5. **Fragment + non-move scope unchanged.** Fresh single-node inserts keep B1's behavior. Fragments stay
   deferred (apply-layer fix is a separate slice — scope note above). The `record_deferred_to_b1_2` guard is
   **narrowed** from "move OR fragment → defer" to "fragment → defer" (move → emit).

## §2. Design — the two-record move

### 2.1 Order + targets (§4.5 + §4.2.3)
A move (`node` already parented) = remove-from-source **then** insert-into-dest:
1. **Source-removal record** — `{type: childList, target: oldParent, removedNodes:[node], previousSibling:
   <node's old prev>, nextSibling: <node's old next>}` (adopt §4.5 step 2 → remove §4.2.3 step 16).
2. **Destination-insertion record** — `{type: childList, target: newParent, addedNodes:[node],
   previousSibling/nextSibling: <node's NEW siblings>}` (insert §4.2.3 step 8).
Pushed in that order onto `session.notify_records`; delivered in order by the existing microtask.
Same-parent reorder: `oldParent == newParent` but still two records (removal then insertion) — spec-faithful.

### 2.2 Sibling capture timing
The move-aware builder, given `(parent, node)`:
1. **Before** the EcsDom write: `old_parent = get_parent(node)`; `old_prev = prev_exposed_sibling(node)`;
   `old_next = next_exposed_sibling(node)`.
2. Apply via `EcsDom::append_child` / `insert_before` (`detach_with_hook` + relink).
3. **After**: dest siblings from the node's now-current exposed prev/next (append → prev = new last-but-one,
   next = None; insertBefore → next = refChild, prev = its new prev).
4. Build + return both records.

### 2.3 Builder shape
`apply_append_child` / `apply_insert_before` currently return `Option<MutationRecord>` (one). Options (B1.2a
plan-review picks): (a) return `SmallVec<[MutationRecord; 2]>` / `Vec`; (b) a dedicated
`apply_move_*` returning the pair, with the handler choosing fresh-vs-move. The handler already computes
`is_move` (the narrowed guard) so it can dispatch. Lens lean: **extend the existing `apply_*` to return a
record list** (one record source, fresh = 1-elem, move = 2-elem) — One-issue-one-way, the flush path
(`apply_mutation`) gets correct move records for free too.

### 2.4 replaceChild
`apply_replace_child` already emits one coalesced record for the **old-child** removal+new-child insertion at
the dest. If `newChild` was already parented (a move into the replace slot), B1.2a adds the **source-removal
record for newChild's old parent** (its adopt). The old-child's leaving is the existing coalesced record.
(B1.2a plan-review confirms the exact record set for replace-with-moved-newChild.)

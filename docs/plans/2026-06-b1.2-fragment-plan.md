## ¬ß1 Problem (the B1 discovery, now the slice)

B1.2a left **DocumentFragment** as the sole remaining `record_deferred_to_b1_2` case
(`crates/dom/elidex-dom-api/src/element/tree.rs:28`). It is **two coupled defects**, not one:

1. **Apply-layer correctness bug (engine-level, observer-independent)**: `EcsDom::append_child` /
   `insert_before` / `replace_child` (`crates/core/elidex-ecs/src/dom/tree/mutation.rs`) are
   **single-node relink primitives**. Passing a `DocumentFragment` links the *fragment entity itself*
   as a child of `parent` ‚Äî a malformed tree (`<div>` with a `DocumentFragment` child). This is wrong
   DOM regardless of MutationObserver: `appendChild(documentFragment)` must move the fragment's
   *children* into the tree and leave the fragment empty (WHATWG DOM ¬ß4.2.3 "insert" step 1 + 7).

2. **Record gap**: the ¬ß4.3.2 childList record's `addedNodes` must be the *expanded children*, and a
   second record on the fragment (removedNodes = those children) is queued. Today the case is
   deferred (mutation applied, record withheld) so no malformed record leaks.

Because the apply layer is broken, B1.2a kept the case fully deferred. This slice fixes **both**: the
apply-layer expansion AND the record shape, and removes the last defer guard so the three core
childList methods (`appendChild` / `insertBefore` / `replaceChild`) are spec-complete for the
node-kinds DOM allows.

## ¬ß2 Coupled invariants at this corner (the design surface)

Per [[feedback_coupled-invariant-design-corner]], the intersecting axes this slice must hold
simultaneously:

1. **Layering** ‚Äî fragment *expansion* is a DOM-algorithm concern (¬ß4.2.3 step 1 "nodes = node's
   children if DocumentFragment" + step 7 "for each node in nodes"); the `EcsDom` primitive's job is
   one-node relink. So expansion belongs in the **engine-independent `apply_*` layer**
   (`elidex-script-session::mutation`), *not* baked into the ECS primitive. See ¬ß4 design decision.
2. **Multi-node record shape** ‚Äî fragment insert produces **two** records in a fixed order
   (¬ß4.2.3 step 4.2 fragment-record THEN step 8 destination-record); fragment replace produces the
   fragment-record THEN the step-14 coalesced record. The count-0 (empty fragment) early-return
   (step 3) differs between insert (no record at all) and replace (still removes oldChild ‚Üí one
   coalesced record). This is NOT the B1.2a move two-record shape ‚Äî a fragment is **not a move**
   (it has no old parent record; its detachment IS the fragment record).
3. **Per-op capture discipline** ‚Äî append/insert capture `previousSibling` per step 6
   (pre-insertion); replace captures `previousSibling`/`referenceChild` per steps 7‚Äì9 (pre-removal,
   oldChild-relative). Inherited verbatim from B1.2a's per-op split ‚Äî do **not** unify them.
4. **CE / dispatcher fan-out (mechanism A)** ‚Äî moving each child via the per-node primitive fires the
   per-node `MutationEvent` (CE `connectedCallback` per element, Range live-tracking, reconcile). This
   is correct and spec-aligned (¬ß4.2.3 step 7.7 runs insertion steps + CE reactions **per
   inclusiveDescendant**). The fragment-internal removal of each child fires `Remove` on the (detached,
   `was_connected=false`) fragment ‚Üí no `disconnectedCallback`. Spec's `suppressObservers=true` on that
   removal (step 4.1) governs **MutationObserver records only**, not the dispatcher/insertion steps.
5. **Shadow / exposed-sibling** ‚Äî sibling capture uses the existing `prev_exposed_sibling` /
   `next_exposed_sibling` / `children_iter[_rev]` (skip internal `ShadowRoot`, ¬ß4.8). Fragment children
   are ordinary; `children_iter(fragment)` yields them correctly.

## ¬ß4 Design decision: expansion lives in `apply_*`, not the ECS primitive

**Decision (anchor = ideal, layering- AND spec-structure-consistent)**: fragment expansion is
implemented in the engine-independent `apply_*` layer
(`crates/script/elidex-script-session/src/mutation/mod.rs`), looping the **existing per-node** `EcsDom`
primitive. `EcsDom::append_child` / `insert_before` / `replace_child` stay single-node relink
primitives ‚Äî **unchanged**.

Rationale:
- **Layering mandate**: DOM algorithms route through the engine-independent crate; `EcsDom::*` direct
  calls are for marshalling/relink. ¬ß4.2.3 step 7 *is literally* "for each node in nodes, ‚Ä¶ insert
  node" ‚Äî an orchestration loop over the one-node primitive. Baking fragment-awareness into the
  primitive would (a) embed spec orchestration in ECS core and (b) force a multi-node return that the
  record-building layer needs anyway.
- **Single record source (One-issue-one-way)**: VM host + deferred-flush + dom-api handler all route
  through `apply_*` (verified: ¬ß6). Putting expansion there means every runtime gets identical,
  correct records from one place. `apply_*` lives in `elidex-script-session` ‚Äî the **lower** crate
  (`elidex-dom-api` depends on it, verified `crates/dom/elidex-dom-api/Cargo.toml:16`), so it is the
  only home reachable from *both* the childList apply path and the (higher-layer) variadic dom-api
  methods ‚Üí the correct place for the canonical mechanic.
- **Correct-by-structure guard (defensive, cheap)**: add a `debug_assert!` in the three `EcsDom`
  primitives that `child`/`new_child` is **not** a `DocumentFragment` (callers must expand). Documents
  the contract + trips any future direct-caller regression in debug/test builds without baking the
  algorithm in. (Plan-review: confirm this is wanted vs. over-defensive.)

Rejected alternative ‚Äî fragment-aware `EcsDom` primitives: violates layering, needs a multi-node
return reshaping the primitive signature, and duplicates the children-collect the record layer must do
regardless.

### ¬ß4.1 Pre-existing fragment-expansion path + the One-issue-one-way convergence (F1)

A second fragment-expansion implementation **already exists and is live**:
`insert_node_expanding_fragment` (`crates/dom/elidex-dom-api/src/child_node/mod.rs:334`) +
`convert_nodes_into_node` (`:317`). It loops the per-node `EcsDom::insert_before`/`append_child` over a
fragment's children ‚Äî the *same mechanic* this slice's `expand_fragment` proposes ‚Äî but is
**record-free** (returns `Result<(), DomApiError>`, emits no `MutationRecord`). It is wired to the
variadic `ParentNode`/`ChildNode` methods (`before`/`after`/`replaceWith`/`prepend`/`append`/
`replaceChildren`) at `child_node/mutations.rs:48/59,94/97,163/175,204/207,236/238,266/283` ‚Äî which are
**MutationObserver-silent today** and are explicitly the **B1.2b** slice's scope (¬ß3 Breadth).

To avoid an unbounded strangler (CLAUDE.md "One issue, one way") while honoring the edge-dense
multi-PR split (B1.2b is a named terminal slice ‚Äî pulling it into this PR is forbidden):

- **Canonical home (going forward)** = `expand_fragment` (record-producing, this slice,
  `elidex-script-session` mutation layer). It is the lower crate + the single record source.
- **`insert_node_expanding_fragment` is the legacy record-free path** for the variadic methods. Its
  **deletion + migration onto the record-producing path is B1.2b's named deliverable**: B1.2b rewrites
  the variadic methods to emit records (its whole point ‚Äî "ParentNode/ChildNode ‚Üí notify_records"),
  which means routing their converted node through the record-producing `apply_*`/`expand_fragment`
  path and **deleting `insert_node_expanding_fragment` + `convert_nodes_into_node`'s fragment-wrapping**.
- This is a **bounded, named cross-PR convergence**, not an unbounded strangler ‚Äî the sanctioned shape
  of an edge-dense multi-PR program. This slice does **not** introduce a *new* permanent parallel home;
  it introduces the canonical one and names where the legacy one dies (¬ß9).

Not converged in *this* slice because doing so requires wiring records through the variadic methods'
own algorithm (`convert_nodes_into_node`, `viable_prev/next_sibling`, pre-insert reference
computation) = B1.2b's terminal scope; bundling it would violate the edge-dense-split discipline.

## ¬ß5 Implementation

### 5.1 New shared helper (One-issue-one-way for the expansion)

`crates/script/elidex-script-session/src/mutation/mod.rs`:

```rust
/// Expand a DocumentFragment per DOM ¬ß4.2.3 "insert" steps 1/4/7: collect the
/// fragment's children (static ‚Äî step-10 "collect all nodes before" note), emit
/// the ¬ß4.2.3 step-4.2 fragment record (addedNodes=¬´¬ª, removedNodes=children;
/// always queued ‚Äî the step "intentionally ignores suppressObservers"), then
/// move each child into the tree via `link_each` (the step-7 "for each node in
/// nodes" per-node relink, which fires the per-node insertion steps + CE
/// reactions through the EcsDom primitive). Returns `None` for an empty fragment
/// (count 0 ‚Üí step-3 early return, no records). `link_each` performs append or
/// insert-before per the caller.
fn expand_fragment(
    dom: &mut EcsDom,
    fragment: Entity,
    mut link_each: impl FnMut(&mut EcsDom, Entity),
) -> Option<(MutationRecord, Vec<Entity>)> {
    let nodes: Vec<Entity> = dom.children_iter(fragment).collect(); // step 1, static
    if nodes.is_empty() {
        return None; // step 3
    }
    let frag_record = MutationRecord {
        removed_nodes: nodes.clone(),
        ..empty_record(MutationKind::ChildList, fragment) // step 4.2
    };
    for &node in &nodes {
        link_each(dom, node); // step 7 per-node
    }
    Some((frag_record, nodes))
}

#[inline]
fn is_fragment(dom: &EcsDom, e: Entity) -> bool {
    matches!(dom.node_kind(e), Some(NodeKind::DocumentFragment))
}
```

### 5.2 `apply_append_child` ‚Äî fragment branch (prepended; existing single-node path unchanged)

```rust
if is_fragment(dom, child) {
    let prev_sibling = dom.children_iter_rev(parent).next(); // step 6, pre-move (parent's last child)
    let Some((frag_record, nodes)) =
        expand_fragment(dom, child, |d, n| { let _ = d.append_child(parent, n); })
    else { return Vec::new() }; // empty fragment ‚Üí step 3
    let dest = MutationRecord {
        added_nodes: nodes,
        previous_sibling: prev_sibling, // next_sibling = null for append
        ..empty_record(MutationKind::ChildList, parent) // step 8
    };
    return vec![frag_record, dest]; // step 4.2 record THEN step 8 record
}
```

### 5.3 `apply_insert_before` ‚Äî fragment branch

```rust
if is_fragment(dom, new_child) {
    let prev_sibling = dom.prev_exposed_sibling(ref_child); // step 6, pre-move
    let Some((frag_record, nodes)) =
        expand_fragment(dom, new_child, |d, n| { let _ = d.insert_before(parent, n, ref_child); })
    else { return Vec::new() };
    let dest = MutationRecord {
        added_nodes: nodes,
        previous_sibling: prev_sibling,
        next_sibling: Some(ref_child),
        ..empty_record(MutationKind::ChildList, parent) // step 8
    };
    return vec![frag_record, dest];
}
```
(Inserting each child before the *same* `ref_child` in tree order preserves order: [n1,n2,ref].)

### 5.4 `apply_replace_child` ‚Äî fragment-newChild branch (distinct capture, steps 7‚Äì14)

```rust
if is_fragment(dom, new_child) {
    // Deferred-flush safety: the dom-api handler checks oldChild‚ààparent, but
    // apply_mutation(Mutation::ReplaceChild) does not. Re-check here (replace step 3).
    if dom.get_parent(old_child) != Some(parent) { return Vec::new(); }
    let reference_child = dom.next_exposed_sibling(old_child); // step 7 (step 8 n/a for a fragment)
    let previous_sibling = dom.prev_exposed_sibling(old_child); // step 9, pre-removal
    if !dom.remove_child(parent, old_child) { return Vec::new(); } // step 11 (suppressObservers)
    let frag_and_nodes = expand_fragment(dom, new_child, |d, n| match reference_child {
        Some(rc) => { let _ = d.insert_before(parent, n, rc); }
        None => { let _ = d.append_child(parent, n); }
    }); // steps 12-13
    let nodes = frag_and_nodes.as_ref().map(|(_, n)| n.clone()).unwrap_or_default();
    let coalesced = MutationRecord {
        added_nodes: nodes,
        removed_nodes: vec![old_child],
        previous_sibling,
        next_sibling: reference_child,
        ..empty_record(MutationKind::ChildList, parent) // step 14
    };
    return match frag_and_nodes {
        Some((frag_record, _)) => vec![frag_record, coalesced], // step 4.2 (via step 13) THEN step 14
        None => vec![coalesced], // empty fragment: no fragment record, oldChild still removed
    };
}
```

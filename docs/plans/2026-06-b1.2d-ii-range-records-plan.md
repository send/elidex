## §0 Scope / ideal anchor

Make Range **tree-mutations** (childList) MutationObserver-visible by **decomposing them onto the canonical record-producing `apply_*` primitives** (One-issue-one-way) and delivering through the **single existing chokepoint** (`queue_mutation_record`, fed by the per-bridge-op drain). The ideal anchor is **full convergence on `apply_*`** — NOT "construct `MutationRecord`s locally inside `range/mutation.rs`".

By construction this inherits, for free and without duplicated logic:

- **move-record semantics** (B1.2a 2-record move = source-removal + destination), since `apply_append_child` / `apply_insert_before` already emit the move pair via `capture_move_source` + `move_record_list` (`mutation/mod.rs:438-458`, `526-544`);
- **fragment-expand record shape** (the §4.2.3 step-4.2 fragment record + destination record, `mutation/mod.rs:411-436`);
- **transient registered observer creation** (B1.2d-i): `notify_one` runs `add_transient_observers` whenever `record.kind == ChildList && !record.removed_nodes.is_empty()` (`mutation_observer.rs:558-560`). Any childList removal routed through the drain therefore gets transient creation automatically — §9.

**Out of scope this slice:** characterData records (§6), `cloneContents` / `surroundContents` / `createContextualFragment` (still stubs, `#11-range-full-impl`, §11).

**As-built scope addendum (pre-push gate, /elidex-review F4 — One-issue-one-way fold-in):** `Selection.deleteFromDocument` is the **only** tree-mutating Selection method and it delegates to `Range::delete_contents` (Selection API §3.3 "invoke Range's deleteContents()"). Once this slice made `delete_contents` record-producing, `Selection.deleteFromDocument` became the lone caller that **dropped** the produced records (records-produced-but-silently-dropped). Rather than defer (a whole slot for one 5-line wire), the records are threaded through `SelectionState::delete_from_document` → committed in the VM native `delete_selection_contents` via the **same** `commit_range_mutation_records` chokepoint as the Range natives. This is a delivery-completion of an existing caller of the primitive this slice built, **not** a new Selection-MO surface — so it is an in-scope generalization of the §4 chokepoint, not scope-creep. boa stays MO-silent (deletion-bound scope-out). Locked by `selection_delete_from_document_delivers_removal_records` (§10).

## §2 Ideal design — decompose onto `apply_*`

Replace the raw `EcsDom::remove_child` / `insert_before` / `append_child` calls in `range/mutation.rs` with the corresponding `apply_*` primitives from `elidex_script_session::mutation` (already a dependency: `elidex-script-session.workspace = true`; `apply_*` re-exported and called e.g. `element/tree.rs:70`). The Range methods **accumulate the returned records** and surface them to the caller (VM native).

`apply_*` return shapes (verified `mutation/mod.rs`):

| Primitive | Signature (file:line) | Returns |
|---|---|---|
| `apply_append_child` | `:407` | `Vec<MutationRecord>` (0 fail / 1 fresh / 2 move / 2 fragment) |
| `apply_insert_before` | `:466` | `Vec<MutationRecord>` (same arity) |
| `apply_remove_child` | `:550` | **`Option<MutationRecord>`** (note: not `Vec`) |

**Signature change required:** the three Range methods must surface records. Since `insert_node` is already `&self` and `delete/extract` are `&mut self`, the cleanest shape is to **return the accumulated `Vec<MutationRecord>` (alongside the existing return)**:

- `delete_contents(&mut self, dom) -> Vec<MutationRecord>`
- `extract_contents(&mut self, dom) -> (Entity, Vec<MutationRecord>)`
- `insert_node(&self, dom, node) -> Option<(Entity, usize, Vec<MutationRecord>)>` (records only meaningful on `Some`)

Internally each accumulates: `Vec` extend for `apply_append_child`/`apply_insert_before`, `Vec::extend(opt)` for `apply_remove_child`. `apply_remove_child` returning `Option` (not `Vec`) is the only call-site asymmetry to handle.

### §2.1 Coupled invariants (edge-dense enumeration — CLAUDE.md Pre-condition #3)

The `apply_*`-decomposition design simultaneously satisfies these invariants; each **pairwise intersection** is the cross-cutting detail the decomposition must hold (the §12 axes, paired):

| Invariant pair | Intersection — how the design holds both |
|---|---|
| partial-containment × move-to-fragment | the `top_level` filter (`mutation.rs:231-246`) separates fully-contained children (moved via one `apply_append_child` → childList move-record) from partially-contained boundary nodes (text-splice = characterData, deferred §6); only the former produces a record this slice. |
| move-to-fragment × observed/unobserved target | one `apply_append_child(dom, frag, child)` emits **both** the source-removal (target = old parent, **observed**) and the fragment-insertion (target = fresh fragment, **unobserved** per §4.3.2 inclusive-ancestor walk); delivery filters by target, not by the producing call (§7). |
| recursive-subrange × observed target | recursive extract removes descendants from a partially-contained **element** child that STAYS in the live tree → those `apply_remove_child` removals are **observed** (target in document), unlike the moved-out top-level children whose source-parent is the observed target (§5). |
| characterData-boundary × childList | the seam is structural: `apply_*` exists (childList → this slice) vs **no** `apply_replace_data`/`apply_split_text` producer exists (characterData → B1.3); a path that touches only text-data emits zero records here, by construction (§6). |
| drain-delivery × transient-creation | every childList **removal** record routed through the single drain (§4) triggers `add_transient_observers` by construction via the B1.2d-i `notify_one` gate (`record.kind==ChildList && !removed_nodes.is_empty()`) — no separate transient wiring in this slice (§9). |

## §4 Delivery plumbing — shared drain helper (headline decision; **plan-review must stress this**)

**Problem.** Range natives cannot flow through `invoke_dom_api` (`dom_bridge.rs:411`): they (a) return a **fragment entity** (`extractContents`), (b) **commit range-boundary state** to `LiveRangeRegistry` (`commit_range_after_mutation`, `range_proto_mutation.rs:39`), and (c) apply **§5.5 `insertNode` step 13** collapse adjustment to the *registered* range (`range_proto_mutation.rs:182-192`). None of this fits the `DomApiHandler::invoke(this, args, session, dom) -> JsValue` shape.

**Recommended approach (One-issue-one-way, no 2nd pipeline).** Extract the `invoke_dom_api` **Phase-2.5 drain** (`dom_bridge.rs:457-464`):

```rust
let notify_records = host_data.session().take_notify_records();
for record in &notify_records {
    ctx.vm.queue_mutation_record(record);
}
```

into a shared helper — proposed `drain_pending_mutation_records(host_data, vm)` (or a `VmInner` method `drain_notify_records()`) — called by **both** `invoke_dom_api` AND the three Range natives. **One delivery mechanism**, no fork.

**By-construction, not by-convention (plan-review F1 — IMP).** A range native must NOT independently `push_notify_record` then *separately remember* to drain: an un-drained scratch is **silently cleared by the per-turn leak-guard at `session.flush()` (`session.rs:115-120`)** — so a missed drain is a *silent loss* (no callback, no transient creation), not a deferred delivery. Bind produce→push→drain into **one indivisible VM-side commit step** that ALL three natives funnel through: the engine-indep method returns `Vec<MutationRecord>`, and a single helper `commit_range_mutation_records(host_data, vm, records)` performs the `push_notify_record` loop **and** the shared drain as one unit. A native therefore *cannot* push without draining — there is no per-native opt-in to the drain. (This is the structural analogue of the B1.2d-i R1 notifySet/`take_records` leak fix: delivery must not depend on a hand-written follow-up call.)

**Borrow-shape note (for plan-review):** the existing drain takes `host_data: &mut HostData` then re-borrows `ctx.vm` for `queue_mutation_record` (the `host_data` borrow must end first — see the comment at `dom_bridge.rs:459-460`). The Range natives currently borrow via `ctx.host()` → `split_dom_and_live_ranges()` (`host_data.rs:867`) which yields `(&mut EcsDom, &mut LiveRangeRegistry)` but **not** `&mut SessionCore`. Pushing into `notify_records` needs `host.session()` (`host_data.rs:1231`). The native sequencing is therefore: run engine-indep method (collect `Vec`) → push into session → commit range-boundary state → **then** call the shared drain (after the `host_data` borrow ends, mirroring Phase-2.5). Plan-review should confirm the helper signature keeps the `host_data`-borrow-then-`vm`-reborrow discipline intact and does not introduce a second `take_notify_records` race with `invoke_dom_api`.

**Considered alternative (rejected).** Route Range through `invoke_dom_api` by adding new `Mutation`/`DomApiHandler` variants for the three methods. Rejected: the fragment-return (`extractContents`) + range-state commit (`insertNode` step 13, `deleteContents` collapse) **do not fit the handler signature cleanly** — the handler returns a single `JsValue` and has no access to `LiveRangeRegistry`. Forcing them in would distort the `DomApiHandler` contract for three outliers, the opposite of One-issue-one-way. **Plan-review adjudicates** recommended-vs-alternative.

## §5 Per-method childList record decomposition

Steps below quoted from **webref-verified** DOM §5.5 algorithm prose (`webref body dom concept-range-insert | concept-range-extract`, `body dom dom-range-deletecontents`).

### `insertNode` — `#concept-range-insert` (§5.5)

> Step 12: *"Pre-insert node into parent before referenceNode."*

The current `mutation.rs:372-380` loop (`dom.insert_before` / `dom.append_child` per fanned-out node) → replace with `apply_insert_before` / `apply_append_child`, accumulating records. A `DocumentFragment` argument is already pre-fanned into `nodes` at `mutation.rs:315-320`; **but** `apply_*`'s own fragment handling (`mutation/mod.rs:411`, `:502`) re-expands a fragment passed whole. **Plan-review item:** decide whether `insert_node` passes the *original fragment* to a single `apply_*` call (reusing `apply_*`'s fragment record shape → the §4.2.3 step-4.2 fragment record + destination record) **or** keeps its pre-fanned `nodes` loop calling `apply_*` per child (which would produce one record per child, not the fragment-pair shape). The former is more faithful to the §4.3.2 record shape and avoids duplicating fan-out; the latter preserves the existing step 6/8/10-11 cycle-check and offset arithmetic that operate on the materialised `nodes` list (`mutation.rs:334-398`). Recommend: **single `apply_*` call on the fragment**, keeping the cycle pre-check (step 6) and step-13 offset read in `insert_node`; verify `apply_*`'s internal cycle reject (`mutation/mod.rs:418`, `:506`) matches the spec-step-6 reject the method already performs (`mutation.rs:331-338`) — do not double-reject or diverge.

Records produced: **1** (fresh single node) or **fragment-pair** (fragment expand) childList insertion record(s).

### `deleteContents` — `#dom-range-deletecontents` (§5.5)

> Step 4: *"Let nodesToRemove be a list of all the nodes that are contained in this, in tree order, omitting any node whose parent is also contained in this."*
> Step 10: *"For each node of nodesToRemove, in tree order: remove node."*

The `mutation.rs:97-99` (same-container child loop) and `:139-143` (cross-container fully-contained loop) `dom.remove_child` calls → `apply_remove_child` per node, collecting `Option<MutationRecord>` into the `Vec`. → one **childList removal** record per removed top-level node.

### `extractContents` — `#concept-range-extract` (§5.5)

> Step 19: *"For each contained child of containedChildren: append contained child to fragment."*

"append contained child to fragment" is a **MOVE** (the contained child is already parented in the live tree → removed from its source parent, appended to the fresh fragment). In `mutation.rs:191-195` (same-container) and `:241-246` (cross-container top-level) the current code does `dom.remove_child(src_parent, child)` **then** `dom.append_child(frag, child)` as two separate ops. → **DELETE that two-op pair and replace it with a SINGLE `apply_append_child(dom, frag, child)`** (not add `apply_*` alongside the existing `remove_child`): `apply_append_child` detects the existing parent (`capture_move_source`, `mutation/mod.rs:440`) reading it **pre-move**, and emits the **B1.2a 2-record move** (source-removal + fragment-insertion) atomically.

**Structural directive (plan-review F2 — IMP).** This is *replace-the-pair*, not *add-alongside*: if the existing `dom.remove_child(src,child)` is left in and `apply_append_child` added after it, `capture_move_source` reads a child whose parent was already cleared → the move pairing is lost (source-removal mis-targeted/dropped) or the record double-fires. The pre-move source read is a fragile ordering dependency, so **lock it with a test that asserts the move yields exactly the 2-record shape** — `kind==ChildList`, `removed_nodes==[child]`, `target == the original source parent` (not the fragment) — so an "add-alongside" mis-impl fails by construction (§10 test 3 strengthened to assert old-parent target).

Records produced per moved child: **2** (observed source-removal + unobserved fragment-insertion — §7).

**Recursive subrange (extract step 18 / 21):** partially-contained **element** children stay in the live tree; their contained descendants are removed (childList on the partial child) → those removals route through `apply_remove_child` and ARE observed (target = the partial child, which is still in the document). The current impl's recursion is the `top_level` filter + pre-order walk (`mutation.rs:231-246`); this slice keeps that structure, only swapping the primitive calls.

## §7 Move-records (extractContents) — observed vs unobserved targets

Per `#queue-a-mutation-record` (§4.3.2) **step 2**: *"Let nodes be the inclusive ancestors of target"* — a record is delivered only to observers registered on the **target's inclusive ancestors**. (Verified `webref body dom queue-a-mutation-record`.)

The destination fragment is a fresh `DocumentFragment` (`mutation.rs:158`, `create_document_fragment`) with **no registered observers** in its inclusive-ancestor chain → its insertion record is queued but **reaches no observer**. The **source-parent removal record IS observed** (target = the original parent, in the document). = the **B1.2a 2-record move pattern**: 1 observed source-removal + 1 unobserved fragment-insertion.

`deleteContents` = plain removal: **1 record per contained child** (no destination), all observed (target = source parent).

## §8 Layering (CLAUDE.md "Layering mandate")

The drain helper itself is engine-bound plumbing (it bridges `session` → `queue_mutation_record`), so it correctly lives VM-side; it carries **no DOM algorithm**.

## §9 Transient + live-range

- **Transient creation = AUTOMATIC.** Once `deleteContents`/`extractContents` removal records flow through the drain → `queue_mutation_record` → `notify_one`, the §4.2.3 "remove" step-15 transient gate fires: `if record.kind == ChildList && !record.removed_nodes.is_empty()` (`mutation_observer.rs:558`) → `add_transient_observers(dom, record.target, &record.removed_nodes)`. No new code needed — B1.2d-i delivers it.
- **Live-range fixup is ORTHOGONAL and unchanged.** `LiveRangeBridge` (`range/live.rs`) consumes the ECS `MutationEvent` hook stream (the `after_*` hooks: `after_split_text`, `after_insert`, `after_replace_data`), **not** `MutationRecord`s. The `apply_*` primitives route through the same `EcsDom` chokepoints that already fire those hooks (`apply_remove_child` → `dom.remove_child`, etc.), so live-range boundary adjustment continues to work exactly as today. **Confirm at impl time: no double-fire** — `apply_*` does NOT add a second `MutationEvent`; it reads the same `EcsDom` mutation that fires the hook once and additionally returns a `MutationRecord`. The two streams (MutationEvent for live-range, MutationRecord for observers) are disjoint by construction.

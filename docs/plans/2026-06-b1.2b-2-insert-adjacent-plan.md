## §1 Problem

`MutationObserver` does not observe `insertAdjacentElement`/`insertAdjacentText`. Four coupled defects, one root (identical shape to #393):

1. **MO-silent (the B1.2 gap).** Neither the VM natives (`element_insert_adjacent.rs` `perform_adjacent_insert` → raw `EcsDom::insert_before`/`append_child`, lines 176/187/190/198/213/218) nor the dom-api handlers (`element/tree.rs` `insert_adjacent` → raw `EcsDom` lines 389/399/405/413/427/433) produce a `MutationRecord`.
2. **Double implementation (One-issue-one-way violation).** The insert-adjacent algorithm exists twice: dom-api `InsertAdjacentElement`/`InsertAdjacentText` handlers (`element/tree.rs:313/345`, used by **boa** `element_methods.rs:57/79` + wasm) and a parallel VM native re-implementation (`element_insert_adjacent.rs` `perform_adjacent_insert` + `require_element_arg` + `parse_adjacent_position`).
3. **Layering violation (CLAUDE.md "VM host/ は engine-bound 責務のみ").** The VM re-implements the position algorithm AND the Element-kind validation (`require_element_arg`, `node_kind_inferred == Element`) — engine-independent algorithm mis-placed in VM-only.
4. **Element-kind gap reachable via boa (raw EcsDom shadow insertion).** The canonical dom-api `InsertAdjacentElement.invoke` (`element/tree.rs:322`) resolves `args[1]` from the identity map with **no kind check** → boa `host.insertAdjacentElement("beforeend", el.shadowRoot)` resolves a `ShadowRoot` entity → `insert_adjacent` → raw `EcsDom::append_child` with no `rejects_shadow_root_insertion` guard. The **VM path is currently safe** (`require_element_arg` brand-checks `NodeKind::Element`, rejecting the ShadowRoot/DocumentFragment) — but converging the VM onto the unchecked handler would *regress* the VM unless the check is relocated to the canonical layer. So the convergence MUST move Element-kind validation down into the dom-api handler (fixing the boa gap as a by-product, in the engine-independent path that the post-S5 VM inherits — NOT a boa-specific patch).

**boa proves the handler home; `appendChild` proves the record path.** boa already routes insertAdjacent* through the dom-api handlers; `appendChild`/`insertBefore` already prove VM-native → `invoke_dom_api` → registry handler → `apply_*` (record-producing) → Phase 2.5 drain (`dom_bridge.rs`).

## §2 Coupled invariants at this corner (the design surface / edge matrix)

1. **Position enum × parent-null** (DOM `#insert-adjacent`): `beforebegin`/`afterend` "If element's parent is null, return null"; `afterbegin`/`beforeend` always have `this` as parent. `parse_adjacent_position` + `position_requires_parent` (VM) / the `match` in `insert_adjacent` (dom-api) encode this. Bad `where` → **SyntaxError** ("Otherwise" branch).
2. **Element-kind validation** (insertAdjacentElement: WebIDL §3.2.15 Interface types `#js-interface`, `Element element`): non-Element arg (Text, Comment, Document, **ShadowRoot/DocumentFragment**) must be rejected, NOT inserted. Today VM-only (`require_element_arg`); relocate to the canonical handler (inv-defect 4). The relocation must preserve the VM's **`contains()`-before-kind ordering + two distinct error classes** (`element_insert_adjacent.rs:124`): a *destroyed/recycled* entity has no kind component, so a kind-first check would mis-report it as "wrong type" — the VM splits stale (`adjacent_element_detached_error`) from non-Element (`adjacent_element_arg_error`). insertAdjacentText takes a `DOMString` → always a fresh Text, no kind question.
3. **Record shape & count** (inherited from B1/B1.2a): `insertAdjacentElement` of an already-parented Element = **move = 2 records** (source-removal §4.5 adopt, NOT suppressed + dest insertion); of a fresh/orphan Element = **1 record**. `insertAdjacentText` Text is always fresh = **1 record**. `apply_append_child`/`apply_insert_before` already encode move-vs-fresh — reuse, do not re-derive.
4. **Shadow-root immovability**: a `ShadowRoot` (the node type, DOM §4.8 Interface ShadowRoot) must never be relinked — the *reject* is enforced by the §4.2 mutation algorithms / `apply_*`, not by §4.8 prose. Routing through `apply_*` gives `rejects_shadow_root_insertion` (`mutation/mod.rs:395`) by construction — defense-in-depth even though inv. 2's Element-kind check already excludes it for insertAdjacentElement.
5. **Return value & marshalling**: `insertAdjacentElement` returns the inserted Element wrapper, or **null** on the parent-null `beforebegin`/`afterend` no-op (DOM "return null"); `insertAdjacentText` returns **undefined** (void). The VM dispatcher must marshal the handler's `JsValue::ObjectRef`/null back to a VM wrapper/`JsValue::Null`/`Undefined`.
6. **Text-node allocation leak discipline** (insertAdjacentText): the Text is allocated by the handler — it MUST short-circuit the parent-null `beforebegin`/`afterend` case **before** allocating (else an unreferenced Text leaks into the ECS, no JS handle → never GC'd), and destroy the Text on a non-null-but-invalid insert failure. The VM native does this today (`element_insert_adjacent.rs:293` pre-alloc parent check + error-path `destroy_entity`); the dom-api `InsertAdjacentText.invoke` currently creates the Text **then** calls `insert_adjacent` (`element/tree.rs:363/365`) → **leaks on the parent-null no-op today**. The convergence must move this leak-careful discipline into the canonical handler (a real pre-existing handler bug surfaced by the audit).
7. **Dual-runtime / leak-guard** (identical to #393 inv. 7): the dom-api handlers, after the rework, `session.push_notify_record(...)` for all three runtimes; the VM drains at `invoke_dom_api` Phase 2.5; boa/wasm never drain → records discarded by `SessionCore::flush`'s leak-guard. boa/wasm behavior unchanged (records produced, unobserved); only the VM newly delivers.
8. **CE reactions** (HTML §4.13.6): driven by the `EcsDom` `ConsumerDispatcher` on the per-node relink inside `apply_*` — unchanged by routing.

## §4 Design decision: converge VM onto `invoke_dom_api → dom-api handler → apply_*`

The ideal (One-issue-one-way + layering-correct) resolution, mirroring #393: **make the VM mirror boa.** The dom-api `InsertAdjacentElement`/`InsertAdjacentText` handlers become the single algorithm home (record-producing + Element-kind-validating + leak-careful); the VM natives become thin marshalling dispatchers exactly like `appendChild`.

Rejected alternative (B): keep the VM natives, teach each to push records + keep `require_element_arg` in the VM. Perpetuates the double-impl + layering violation + leaves the boa kind-gap/leak unfixed; rejected on philosophy.

This collapses all four §1 defects at once: routing the VM through `invoke_dom_api` gives records for free (Phase 2.5 drain), deletes the VM duplicate algorithm, restores layering, and (via the relocated kind-check + the handler-side leak fix) closes the boa-reachable shadow/non-Element gap and the text-leak in the engine-independent path.

**Crate-API / layering mapping** (the consolidated view; details distributed across §3 spec-coverage table, §5 implementation, §6 write-site audit): VM marshalling `vm/host/element_insert_adjacent.rs` (thinned to dispatch) → engine-independent algorithm home `elidex-dom-api` `element/tree.rs` (`InsertAdjacentElement`/`InsertAdjacentText`/`insert_adjacent`) → record-producing primitives `elidex-script-session` `mutation/mod.rs` (`apply_insert_before`/`apply_append_child`/`rejects_shadow_root_insertion`). No new crate, no new primitive.

## §5 Implementation

### 5.1 dom-api handler rework (`crates/dom/elidex-dom-api/src/element/tree.rs`)
- **`insert_adjacent` helper (line 371):** replace each raw `dom.insert_before(...)`/`dom.append_child(...)` with the record-producing `apply_insert_before(dom, parent, node, ref)` / `apply_append_child(dom, parent, node)`; collect the returned `Vec<MutationRecord>` and push **every** record (match the `AppendChild` handler's `push_notify_record` loop, `element/tree.rs`) — a move (already-parented Element) returns **2** records (source-removal + dest); dropping the source-removal record (inv. 3) is the read/write-balance failure mode the §7 "2 records" test guards. Four-position branching + "Otherwise" → **SyntaxError** (`DomApiErrorKind::SyntaxError` already exists, `tree.rs:442` — no new error class needed).
  - **⚠ pre-existing spec bug to fix (3rd boa-reachable defect, found at impl-read):** the current helper **throws `HierarchyRequestError` for `beforebegin`/`afterend` when `this` has no parent** (`tree.rs:385-388/421-424`). DOM `#insert-adjacent` says "If element's parent is null, **return null**" — a silent no-op, NOT an error. The VM path is correct today (`position_requires_parent` → returns null/undefined, `element_insert_adjacent.rs:293`); the boa/dom-api path wrongly throws. The rework MUST make parent-null a no-op: change the helper signature to communicate "no-op" (e.g. `Result<Option<()>, DomApiError>` or returning an empty record list with a distinct no-op flag) so `InsertAdjacentElement.invoke` maps it to `JsValue::Null` and `InsertAdjacentText.invoke` to `JsValue::Undefined`. Add a regression test (boa-reachable parent-null no-op returns null, no throw, no record).
- **`InsertAdjacentElement.invoke` (line 322):** add Element-kind validation after entity resolution — reject non-`NodeKind::Element` (relocated from VM `require_element_arg`), **preserving the `contains()`-before-kind ordering + the two error classes** (stale `adjacent_element_detached_error` vs non-Element `adjacent_element_arg_error`, inv. 2) so a destroyed handle surfaces "detached" not "wrong type". Then route through the reworked `insert_adjacent`. Empty `apply_*` list from a non-fragment node = genuine failure (shadow-guard rejection or invalid) → map to `HierarchyRequestError` (mirror the mixin handlers).
- **`InsertAdjacentText.invoke` (line 354):** move the parent-null short-circuit **before** `dom.create_text(...)` (inv. 6 — fixes the pre-existing leak), and destroy the Text on a non-null insert failure. Then route through the reworked `insert_adjacent`.

### 5.2 VM dispatcher conversion (`crates/script/elidex-js/src/vm/host/element_insert_adjacent.rs`)
Convert `native_element_insert_adjacent_element` + `native_element_insert_adjacent_text` to thin dispatchers: receiver brand-check (`require_receiver` for `NodeKind::Element`, kept) → `invoke_dom_api(ctx, "insertAdjacentElement"|"insertAdjacentText", target, &args)`. Marshal the result (inv. 5): `JsValue::ObjectRef` → element wrapper; null → `JsValue::Null` (insertAdjacentElement) / `Undefined` (insertAdjacentText). The `where` string + `data`/`element` args pass through `invoke_dom_api` (string passthrough landed in #393; the element arg is an object handle resolved by the handler). Model: `node_proto.rs` `native_node_append_child`.

### 5.3 Deletions (One-issue-one-way)
After 5.1+5.2 the following lose all callers — re-verify callerless via grep, then delete:
- VM: `perform_adjacent_insert`, `require_element_arg`, `require_element_arg` error helpers if now unused, and `parse_adjacent_position`/`position_requires_parent` **iff** the position parse fully moves to the handler (the handler's `insert_adjacent` already lowercases + matches; confirm the SyntaxError path is handler-side before deleting the VM parser — otherwise keep the VM `where`-parse as marshalling and document why).
- Keep `create_text_with_owner` (general DOM primitive, used elsewhere).

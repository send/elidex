## §0 Scope (one sentence)

Produce the `characterData` MutationRecords that the §5.5 Range mutation methods and §4.11 `Text.splitText` queue for their text-splice steps — the last record gap B1.2d-ii dependency-deferred because no record-only characterData producer existed at the splitText-ordering seam.

**IN scope** (the deferred-to-B1.3 facets named in `range/mutation.rs` + the lock test):
1. `Range.deleteContents` text splices → `characterData` records (same-container CharacterData step 3; cross-container start step 9 + end step 11).
2. `Range.extractContents` text splices → `characterData` records (same-container step 4.4; cross-container start step 17.4 + end step 20.4).
3. `Range.insertNode` step 7 internal split → the split's `childList` (new tail) + `characterData` (head-truncate) records.
4. `Text.splitText` → its `childList` (new node insert, §4.11 step 7.1) + `characterData` (replace data, §4.11 step 8 via the §4.10 replace-data AO) records — *currently splitText produces NEITHER record* (raw `EcsDom` primitives).
5. `Selection.deleteFromDocument` inherits (1) for free — already routed through `commit_range_mutation_records` (#418); only its comment + a confirming test change.

**OUT of scope** (deferral boundaries, §8):
- `Node.normalize()` records → slot `#11-normalize-mutation-records` (distinct text-merge algorithm: `set_text_data` + `remove_child`, two record kinds).
- The `extractContents` **partial-element-ancestor deep-clone** (steps 18/21) — still the B1.2d-ii §11 simplified-cross-container carve; this slice adds *only* the characterData splice records, not the missing element clone.
- `extractContents`/`cloneContents` of a **Comment**-container Range (needs `create_comment` fragment clone) → `#11-range-comment-extract-clone` (§8). (`deleteContents` Comment IS in scope — F5.)
- Attribute records (B2). `cloneContents`/`surroundContents`/`createContextualFragment` stubs (`#11-range-*`).

## §1 First-principles ideal

The DOM spec **is** the record-producing seam: every text splice in these algorithms is the `replace data` AO (§4.10), and `replace data` step 5 *queues a `characterData` mutation record* with `oldValue` = the node's data **captured before** the splice. The two boundary steps of "split a Text node" are likewise `insert` (§4.2.3, queues `childList`) + `replace data` (queues `characterData`).

So the ideal is mechanical, no new mechanism:

> Each spec text-splice / split site emits its record through the **single canonical record builder**, and the existing `commit_range_mutation_records` / `push_notify_record`→drain chokepoint delivers it. No site hand-rolls a `MutationRecord` literal; no second delivery pipe.

Two delivery-shape sub-cases, both already canonical:
- **mutate-and-record** (deleteContents/extractContents splices): the splice mutation has no ordering coupling, so route it through the existing **`apply_replace_data`** primitive (mutate via `replace_text_data` + emit the record) — exactly as the 5 CharacterData methods (#424) do. Just *replace* the bare `dom.replace_text_data(...)` call.
- **record-only** (splitText's head-truncate): the truncate is `set_text_data`-**last** inside `split_text_at_offset`'s tuned `insert → fire_split_text → set_text_data` live-range dance (load-bearing, locked by `split_text_migrates_range_boundary_to_new_node`). We MUST NOT re-mutate (double mutation) nor reorder, so the record is **assembled from the pre-captured full data** via the same record-shape builder `apply_replace_data` uses — not a parallel literal.

**One-issue-one-way consequence**: extract one record-shape builder so the splice path (`apply_replace_data`) and the splitText hand-assembly share it. Proposed:

```rust
// elidex-script-session::mutation (pub, re-exported from crate root)
pub fn character_data_record(target: Entity, old_value: String) -> MutationRecord {
    MutationRecord { old_value: Some(old_value), ..empty_record(MutationKind::CharacterData, target) }
}
```

`apply_replace_data` then returns `character_data_record(entity, old_value)` on success; `split_text_at_offset` hand-assembles via `character_data_record(entity, original)`. This is the ideal over the anchor's lighter "promote `empty_record` → `pub`" (which would let a caller forget the `old_value`/kind coupling). **Fallback if plan-review prefers minimal surface**: promote `empty_record` → `pub` + re-export, hand-assemble inline. I recommend the dedicated builder.

The `childList` half of splitText is **not** hand-assembled — swap the raw `dom.insert_before`/`dom.append_child` (step 5) for the canonical record-producing `apply_insert_before`/`apply_append_child` (same `EcsDom::insert_before` mutation underneath, so the `MutationEvent::Insert` live-range adjust is unchanged; fresh-orphan `new_node` ⇒ exactly one fresh `childList` record).

## §2 The record-producing sites + spec order (webref-verified 2026-06-28)

| Site | Spec | Records (in queue order) | oldValue source |
|------|------|--------------------------|-----------------|
| `deleteContents` same-container CharacterData | §5.5 step 3.1 | 1 `characterData` on start node | start node full data (pre-splice) |
| `deleteContents` cross-container | §5.5 **step 9 start-trunc → step 10 removals → step 11 end-trunc** | `characterData`(start), then `childList` removals…, then `characterData`(end) | start/end node full data |
| `extractContents` same-container CharacterData | §5.5 extract step 4.4 | 1 `characterData` on start node | start node full data |
| `extractContents` cross-container | extract **step 17.4 start → step 19 moves → step 20.4 end** | `characterData`(start), `childList` moves…, `characterData`(end) | start/end node full data |
| `insertNode` Text-boundary | §5.5 insertNode **step 7 split** (before step 12 insert) | split `childList`(new tail) + split `characterData`(head-trunc), THEN step-12 insert record(s) | split node full data |
| `splitText` (parented) | §4.11 split a Text node step 7.1 insert + step 8 replace data (replace data AO itself = §4.10) | `childList`(new node) + `characterData`(head-trunc) | original node full data |
| `splitText` (orphan) | §4.11 split a Text node step 8 only (step 7 skipped, parent null) | `characterData`(head-trunc) only | original node full data |

**⚠ The one spec-fidelity bug this slice must fix**: `delete_contents` cross-container currently does start-trunc → **end-trunc** → removals (`range/mutation.rs:187-212`). Harmless while truncs were record-silent; once they emit records the **order is observable and wrong** vs §5.5 (start → removals → end). **Reorder to start-trunc → removals → end-trunc.** (extractContents order already matches spec 17/19/20 — no reorder.)

`oldValue` is carried on every record unconditionally; §4.3.2 delivery exposes it only to registrations with `characterDataOldValue:true` (existing filter, untouched).

### §2.5.1 User-input touch audit

- `Text.splitText(offset)`: offset is user-controlled — `to_uint32` coerced (VM) / `require_usize_arg` (handler), range-checked vs UTF-16 length (`OffsetOutOfBounds` → `IndexSizeError`/`RangeError`). Unchanged by this slice.
- `Range` boundaries (`setStart`/`setEnd` container + offset): user-controlled; offsets are byte-clamped (`utf16_offset_to_byte_clamped`) before splice. The `oldValue`/splice math is on already-validated boundaries.
- **Adjacent pre-existing surface (RESOLVED, F5)**: `delete_contents`/`extract_contents` gated same-container text on a `TextContent`-only check — a Range boundary landing in a **Comment** container fell to the non-text branch (latent no-op bug). Resolution: `delete_contents` broadens the guard to `get_char_data` (=CharacterData) — splices + emits the record (spec §5.5 = "CharacterData node"); `extract_contents` stays Text-only with the Comment fragment-clone deferred to `#11-range-comment-extract-clone` (§8). See E6 / §3.3.

## §3 Primitive / signature changes

### 3.1 `elidex-script-session::mutation`
- Add `pub fn character_data_record(target, old_value) -> MutationRecord` (§1). Re-export from crate root.

### 3.2 `elidex-dom-api::char_data::split_text::split_text_at_offset`
Signature: `Result<Entity, SplitTextError>` → **`Result<(Entity, Vec<MutationRecord>), SplitTextError>`**.
- Step 5 insert: raw `insert_before`/`append_child` → `apply_insert_before`/`apply_append_child`; `records.is_empty()` ⇒ the old `InsertFailed` path (rollback + destroy unchanged). Collect its `Vec` (≤1 record).
- Steps 6-7 unchanged (`fire_split_text` then `set_text_data`-last — the locked ordering).
- After step 7, push `character_data_record(entity, original)` (the §4.11 split-a-Text-node step-8 record — step 8 invokes the §4.10 "replace data" AO; `original` = full pre-split data already read at step 2). Order in the returned vec: **[childList insert?, characterData truncate]**.
- Orphan path (no parent): no insert record; still emit the characterData truncate record.
- The 8 in-file unit tests + 2 live-range tests: destructure `(new_node, _records)` / `.0`. Add a records-shape assertion to ≥1 parented + the orphan test.

### 3.3 `elidex-dom-api::range::mutation`
- `delete_contents`: replace each `dom.replace_text_data(...)` with `apply_replace_data(dom, node, off, count, "", old_full)` (capture `old_full` via `get_char_data` **before**), push the returned `Option` record. **Reorder cross-container to start → removals → end** (§2 ⚠).
  - **F5 root fix — broaden the same-container + cross-container splice guards from `TextContent`-only to `get_char_data(node).is_ok()` (= CharacterData, Text|CDATASection|Comment).** Spec §5.5 deleteContents steps 3/9/11 say "**CharacterData** node", not Text. `apply_replace_data` already dispatches Text→`replace_text_data` / Comment→`replace_comment_data` (#424), so the splice + record are trivially correct for Comment too — and this *fixes a latent pre-existing bug*: a Range wholly inside a **Comment** currently hits the same-container non-text branch (`children[start..end]`, vacuous on a childless Comment) and **silently does nothing**. Broadening makes it splice + emit the record. (Comment carries no live-range fixup event — a known elidex limitation doc'd on `apply_replace_data`; the operated range still self-collapses.)
- `extract_contents`: same `replace_text_data` → `apply_replace_data` swap at 2a/2c + same-container 4.4; existing 17/19/20 order kept. The clone-into-fragment appends stay unobserved (fragment childList, no record) — unchanged. **Keep the `NodeKind::Text` guard here** (NOT broadened to CharacterData): extract's clone-into-fragment uses `dom.create_text(...)`, so a Comment container would need a `create_comment` clone — non-trivial → **deferred to `#11-range-comment-extract-clone` (§8)**. The *splice record* on the original Comment is reachable but producing a wrong-kind (Text) fragment clone is worse than deferring the whole Comment-extract; keep extract Text-only this slice.
- `insert_node`: destructure `split_text_at_offset(...) → (tail, split_records)`; **prepend `split_records`** to the method's `records` before the step-12 insert records (spec step 7 < step 12).

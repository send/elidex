
# 28. 主要設計決定（ADR）

主要な選択とその根拠を記録したアーキテクチャ決定記録（ADR）。

| # | 決定事項 | 選択 | 根拠 |
| --- | --- | --- | --- |
| 1 | 実装言語 | Rust | メモリ安全性、fearless concurrency、所有権モデルが並列レイアウトを実現 |
| 2 | スクリプティング戦略 | 三層設計：DOM API（プラガブル、Living Standard + 互換）、ECMAScript（Boa → 自前Rustエンジン、ES2020+コア / Annex B互換）、Wasm（多言語elidex-app向けwasmtime） | HTML/DOM/JSにわたる一貫したコア+互換パターン。自前エンジンが長期的なサードパーティJSランタイム依存を解消。Wasmが多言語アプリを実現。 |
| 3 | DOMストレージモデル | ECS（Entity Component System） | キャッシュ効率、自然な並列性、ゲームエンジン（Bevy）に着想 |
| 4 | レガシー処理戦略 | コア前正規化（トランスパイラパターン） | コアをクリーンに保ち、quirksモードのコードパスがエンジンに感染するのを防止 |
| 5 | 機能拡張性 | デュアルディスパッチ：静的enumディスパッチ（コンパイル時、ゼロコスト）+ 動的トレイトオブジェクト（ランタイム拡張）、同一トレイト下で統一 | 組み込み機能はオーバーヘッドなし、拡張は同じAPIを使用。非推奨化されたコア機能はコミュニティ動的プラグインとして存続可能 |
| 6 | GPUレンダリング | Vello on wgpu（ADR #26参照） | Rustネイティブ、クロスプラットフォーム、SkiaのC++依存を回避 |
| 7 | プロセスアーキテクチャ | マルチプロセス（Chromium着想、Ladybirdクリーン） | セキュリティ分離、クリーンなIPC境界がクレート境界に一致 |
| 8 | Writing mode | 初日から抽象インライン/ブロック座標 | CJK縦書きに必要、後付け不可 |
| 9 | デュアルユースアーキテクチャ | 共有コアからelidex-browser + elidex-app | アプリランタイム（Electron/Tauri代替）が主要な価値提案 |
| 10 | 非推奨化メカニズム | 定期的クローラーサーベイによるデータ駆動 | 機能削除の客観的・定量的根拠、主観的議論を回避 |
| 11 | エラー回復戦略 | 段階的デグラデーション：strict → ルールベース → エラー（LLMレイヤーは停止中）。 | 正確さ＝速度。壊れたHTMLに比例するレイテンシコスト。**Phase 0 結果:** ルールベース回復で十分であることをサーベイが実証（[第29章](29-survey-analysis.md)）。**プロジェクト判断（2026-03-05）:** LLM関連は一旦停止。 |
| 12 | LLMフォールバック採用 | Phase 0.5サーベイに条件付き（≧2%回復不能エラー率）だが現在は一時停止 | 壊れたHTMLが稀なら不必要な複雑性を回避。**Phase 0 結果:** 回復不能エラー 0%（900サイト）— No-Go 確定（[第29章](29-survey-analysis.md)）。**プロジェクト判断（2026-03-05）:** 開発者診断を含むLLM関連は停止。 |
| 13 | ネットワーク層設計 | プラガブルResourceLoader + ミドルウェアパイプライン、コンテンツブロッキングはエンジン中立 | アプリモード向けカスタムURLスキーム、DevTools、広告ブロック、将来のプロトコル進化をコア変更なしで実現。 |
| 14 | DOM APIアーキテクチャ | DomApiHandlerプラグイントレイト（DomSpecLevel: Living/Legacy/Deprecated）。各メソッド個別にトグル可能。コンポーネント操作経由のECS統合。 | HTML/CSSと同じ非推奨化モデル。elidex-appはコンパイル時にレガシーDOMを除外。MutationObserverがECS変更検出に自然にマッピング。 |
| 15 | CSS仕様レベル分類 | CssSpecLevel enum（Standard/Aliased/NonStandard/Deprecated）。用途レベルではなくプロパティ＋値レベル。 | CSSをHTML/DOM/ESの三層パターンに整合。用途レベル検出（例：floatのレイアウト用途）は非現実的。プロパティ非推奨化はクローラーデータで駆動。 |
| 16 | Web API core/compat分離 | WebApiSpecLevel（Modern/Legacy/Deprecated）によるAPI単位分類。同期API（localStorage、XHR、document.cookie）はcompat、非同期同等物がcore。 | 同期APIがメインスレッドをブロックし、マルチプロセスアーキテクチャでブロッキングIPCを要求。elidex-appはコンパイル時に同期APIを除外。非同期ストレージAPI（elidex.storage）がアプリ開発者向けlocalStorageを置換。 |
| 17 | ScriptSessionレイヤー | すべてのScript ↔ ECSインタラクションを仲介する統一セッション。全Object Model API（DOM、CSSOM、将来のOM）に対してIdentity Map、Mutation Buffer、GC協調、ライブクエリ管理を提供。 | OMレイヤーごとのアドホックなインピーダンスミスマッチ解決を排除。ORM Unit of Workパターンに類似。変更のアトミックフラッシュ、一貫したMutationObserverレコード、すべてのOM APIにわたるオブジェクト同一性保証を単一メカニズムで実現。 |
| 18 | 三層プラガビリティモデル | エンジン層：仕様駆動（レガシーを削る）。プラットフォーム層：OS API＋トレイト契約（実装を差し替える）。ブラウザシェル：トレイト契約のみ（UIを拡張/差し替える）。 | Web標準がエンジンに明確なcore/compat境界を提供するが、ブラウザシェルには権威ある仕様がない。プラットフォーム層とシェル層ではトレイト契約がSpecLevelに代わるプラグイン根拠となる。 |
| 19 | ブラウザchrome戦略 | 段階的アプローチ：Phase 1-2はネイティブchrome（egui/iced）、Phase 3+でセルフホストオプション（HTML/CSS）。長期的に両方共存、ビルド時に選択可能。 | セルフホスティングの前にエンジンが機能的でなければならない。ネイティブchromeが初期開発中の高速イテレーションを提供。セルフホストchromeがドッグフーディングと最大限のカスタマイズを実現。 |
| 20 | プロセスモデル：段階的緩和 | Phase 1–3（Boa時代）はサイト単位分離、全Rust移行後にクラッシュ隔離のみに緩和可能。ProcessModel enumがビルド時/起動時に設定可能。IPCはトレイト抽象化（ProcessChannel / LocalChannel）。 | ブートストラップ期はBoa統合とホストバインディングのブラウザ脅威モデル向けハードニングが進行中のため、厳格な分離で影響半径を限定する。elidex-js移行とハードニング完了後は方針に応じてサイト分離をオプショナル化可能。トレイト抽象化により緩和が設定変更で完了し、書き直し不要。エンジン層と同じcore/compatパターン。 |
| 21 | 非同期ランタイム：ハイブリッドアプローチ | Renderer: カスタムelidexイベントループ + tokio reactorをI/Oバックエンドとして。Network/Browser: tokioマルチスレッド。将来の置換用AsyncRuntimeトレイト。 | ブラウザイベントループはtokioのスケジューラが提供しない精密な制御（vsync、rAF、フレーム予算）が必要。tokioはI/O集約プロセス（Network）に最適。トレイト抽象化で長期的にtokioを自前ランタイムに置換するオプションを保持、Boa → elidex-jsの移行パスに対応。 |
| 22 | ストレージバックエンド：トレイト背後のSQLite | SQLite（rusqlite）をStorageBackend初期実装とし、StorageBackend/StorageConnectionトレイトで抽象化。トレイト境界では生SQLではなくStorageOp enum。 | SQLiteはすべての主要ブラウザ（Chrome、Firefox、Safari）で実証済み。IndexedDBはKVストアでは効率的に提供できないリレーショナル操作（インデックス、カーソル、範囲クエリ）を要求。ストレージ用途としてネイティブ依存のフットプリントは許容範囲。トレイト抽象化でドメイントレイト（CookiePersistence、HistoryStore等）に影響なく将来の置換が可能。 |
| 23 | オリジン隔離：オリジン単位DBファイル | Webコンテンツストレージはオリジンごとに物理的に別のSQLiteデータベースを使用。ブラウザ所有データは集中管理のbrowser.sqliteを使用。 | クロスオリジンデータ漏洩に対する構造的保証 — データが別ファイルに存在するためクエリバグでクロスオリジンデータは公開されない。Chrome/FirefoxのIndexedDB、localStorage、Cache APIのアプローチに準拠。OriginStorageManagerがアクセスを仲介；Rendererはファイルに直接触れない（オリジン検証付きBrowser ProcessへのIPC）。 |
| 24 | コンポジタスレッド：アプローチBとB→C移行パス | Phase 1–3: Servo式メッセージパッシング（DisplayList所有権をコンポジタに移転）。Phase 4+: 共有ECS読み取り（アプローチC）へのオプショナル移行。FrameSourceトレイトが境界を抽象化。 | アプローチB（メッセージパッシング）はRustの所有権モデルと整合し、Servoの実績があり、プロセス境界を越えて動作（GPU Process分離）。アプローチC（共有ECS）はシリアライゼーションコストを排除するが同一プロセスが必要 — プロセス分離緩和と自然に連動。FrameSourceトレイトによりB→Cは設定変更であり書き直しではない。IpcChannel（ProcessChannel/LocalChannel）と同パターン。 |
| 25 | Web Workers: 1:1 OSスレッドマッピング | 各Dedicated/Shared/Service Workerが独自のOSスレッドに独自のScriptEngineとScriptSessionを持つ。プールではない。 | Workerは無期限に実行可能（ゲームループ、リアルタイム処理）。スレッドプールでは全スロットが長寿命Workerで占有されるとデッドロック。SharedArrayBufferはAtomics.wait/notify（futexセマンティクス）に真のOSスレッドが必要。1:1マッピングはすべての既存ブラウザが使用するモデル。 |
| 26 | Vello：直接依存（トレイト抽象化なし） | Velloを抽象化トレイトなしで直接使用。接触面をDisplayList→Vello Scene変換層のみに隔離。 | hyper（HttpTransport）やSQLite（StorageBackend）で使用したトレイト抽象化パターンの例外。実用的なRustネイティブGPU 2Dレンダラー代替が存在せず、トレイトはVelloのシーンAPIを実質的利点なく複製するだけ。HTTPクライアントやストレージエンジンのように複数候補が抽象化を正当化するケースと異なり、存在しない代替のための抽象化は過剰設計。上流コード（ECS、スタイル、レイアウト、ペイント、レイヤーツリー、ディスプレイリスト）すべてがVello依存ゼロ。有力な代替出現時、変換層のみ置換対象。 |
| 27 | 独立LayerTree構造（ECSコンポーネントではない） | レイヤーツリーはECS DOMから独立したデータ構造。PaintSystemがECSを読み取りLayerTreeを出力として構築。 | DOMノードとレイヤーはN:M関係：複数DOMノードが1レイヤーにスカッシュ、1ノードが複数レイヤーを生成（スクロール可能コンテナ＋コンテンツ）。ECSの1エンティティ:1コンポーネントモデルに合わない。独立LayerTreeはアプローチC（第6章）も可能に：コンポジタがArc<RwLock<LayerTree>>を読み取りDOMに触れない。Chromiumの独立cc::LayerTreeアプローチに準拠。 |
| 28 | フレームスケジューリング：FramePolicy enum | 4ポリシー：Vsync（ブラウザデフォルト、アイドル時スキップ）、Continuous（毎vsync、ゲーム）、OnDemand（dirtyフラグのみ、elidex-appデフォルト）、FixedRate(n)（タイマー駆動）。 | ブラウザとアプリランタイムはフレームレート要件が根本的に異なる。ブラウザタブはvsync同期が必要、ドキュメント系アプリはアイドル時60fps描画は電力の無駄、ゲームは最大スループット、サイネージは固定レートが必要。単一Durationベースの予算ではこれらを表現不可能。FramePolicyがタブ単位（ブラウザ）またはアプリ単位（elidex-app）でコンポジタのwake動作を制御し、すべてのユースケースを1スケジューラで統一。 |
| 29 | 統一アニメーションモデル：Web Animationsを内部表現として使用 | CSS TransitionsとCSS Animationsを作成時にWeb Animationsインスタンスに変換。1つのサンプリングエンジン、1つのタイミングモデル、1つの合成システム。 | Web Animations仕様はCSS Transitions/Animationsの統一モデルとして設計された。3つの別エンジンを維持すると補間、タイミング、合成、コンポジタ昇格のコードが3倍になる。統一モデルにより`element.getAnimations()`がオリジンに関係なくすべてのアニメーションを自然に返す。Chromeのアーキテクチャ（BlinkのCSS/WAAPI統一アニメーションモデル）に準拠。 |
| 30 | アニメーションタイミング：FrameProducer + ECSクエリAnimationEngine（C+Bハイブリッド） | FrameProducerがイベントループとレンダリングパイプライン間を協調。AnimationEngineがアニメーション状態をECSコンポーネント（ActiveAnimations）として格納し、並列ECSクエリでtick。 | 純粋委譲（C）がクリーンな責務分離を提供：イベントループがWHENを決定、FrameProducerがWHATを決定。ECSクエリ内部（B）がrayon並列サンプリング+書込（各エンティティ独立）、自動ライフサイクル管理（despawnでアニメーション除去）、StyleSystemとAnimationEngine間の疎結合（両方がECSを読み書き、互いを参照しない）を実現。ハイブリッドが両方の利点を得て、いずれの欠点もない。 |
| 31 | 画像デコーダ：トレイト抽象化、Rustクレートデフォルト | ImageDecoderトレイトにRustクレート実装（zune-jpeg、png、webp、ravif、gif）。プラットフォームネイティブデコーダ（ImageIO、WIC）が同トレイト経由で差し替え可能。DecoderStrategy enum（RustOnly / PlatformPreferred / Mixed）。 | Pure Rustが可搬性とメモリ安全性を最大化し、長期的な全Rust目標に整合。ただしプラットフォームデコーダはハードウェアアクセラレーション（特にApple SiliconでのJPEG/HEIF、dav1d経由のAVIF）を提供。トレイト抽象化によりRustクレートで開始しプロファイリングで必要な箇所にプラットフォームデコーダを差し込む — HttpTransport（第10章）やStorageBackend（第22章）と同パターン。 |
| 32 | インラインSVGはECSエンティティ、SVG-as-imageは直接Vello | インラインSVG要素をHTMLと同じECS DOMに格納（SvgGeometry、SvgTransformコンポーネント）。SVG-as-imageはECSをバイパス — 軽量シーンにパースし、Vello経由で直接オフスクリーンテクスチャにレンダリング。 | インラインSVGはDOM API（querySelector、MutationObserver、CSSカスケード）に参加する必要あり — ECSが唯一の一貫した方法。SVG-as-imageは仕様上DOMアクセスなし（JS不可、外部リソース不可）のためECSオーバーヘッドは不要。直接Velloレンダリングがシンプルで高速。DisplayItem enumをSvgPath/SvgTextで拡張し統一ペイントパイプライン — Velloの2Dベクタープリミティブがsvgパスに自然適合。 |
| 33 | OPFS SyncAccessHandleは共有メモリマップドファイル経由 | SyncAccessHandleがOPFSファイルをBrowserプロセスとWorker間の共有メモリにマッピングし同期read/writeを提供。Read/writeは直接memcpy（IPCなし）。制御操作（truncate、flush、close）のみIPCが必要。Browserプロセスがファイル単位の排他ロックを強制。 | SQLite-on-webは毎秒数千回の小さなread/write操作を要求。操作ごとのIPC往復は禁止的に遅い（呼び出しあたり~1μs+）。共有メモリがホットパスでこれを排除しつつセキュリティモデルを維持（Browserプロセスがファイル作成、ロック、メタデータを制御）。ChromiumのSyncAccessHandleアプローチに一致。 |
| 34 | メディアコーデック：ロイヤリティフリーはRust/C、特許負担ありはプラットフォーム | ロイヤリティフリーコーデック（VP8/VP9/AV1、Opus、Vorbis、FLAC、MP3）はRust/Cライブラリデコーダ（dav1d、libvpx、lewton、minimp3）を使用。特許負担コーデック（H.264、H.265、AAC）はプラットフォームデコーダ（AVFoundation、MediaFoundation、VA-API）に委譲。MediaDecoderトレイトが両パスを抽象化。Decoderプロセスがサンドボックス分離を提供。 | ImageDecoder（ADR #31）と同じトレイト抽象化パターン。OS提供デコーダで主要Web動画コーデックをサポートしつつ、コア配布の特許ライセンスコストを回避。プロセス分離がコーデックライブラリのバグを封じ込め。プラットフォームAPI経由のハードウェアアクセラレーションデコードがゼロコピーGPUテクスチャパスで最適電力効率を実現。 |
| 35 | bfcacheはRendererプロセス凍結＋退去で実現 | Back/forwardキャッシュがRendererプロセス全体を凍結してページを保存（イベントループ、タイマー、ネットワークを停止）。凍結プロセスは最大6エントリまたは合計512 MBまで保持。退去は最古の凍結プロセスを破棄。WebSocket、WebRTC、unloadリスナー、アクティブIndexedDBトランザクションがオープンのページは不適格。 | プロセスレベルの凍結が完全なページ状態（DOM、JSヒープ、アクティブ接続、Web Audioグラフ）を保存する唯一の確実な方法。WebSocket接続のような複雑な状態に対してECSシリアライゼーションは非実用的。Chromiumのアプローチに一致。メモリ上限付き退去が無制限なメモリ増加を防止。 |

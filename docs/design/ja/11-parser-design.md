
# 11. パーサー設計

## 11.1 デュアルパーサー戦略

2つのパーサーが異なるユースケースに対応する：

| パーサー | 動作 | ユースケース |
| --- | --- | --- |
| parse_strict | 不正なHTMLをコンパイラのように詳細なエラーメッセージとともにリジェクトする。未閉鎖タグをソース位置付きで報告。 | elidex-appモード。開発者はビルド/開発時にエラーを確認しマークアップを修正。最高のパフォーマンスを生む。 |
| parse_tolerant | 一般的なエラー（未閉鎖タグ、ネスト間違い）から回復する。HTML5仕様の完全なエラー回復（例：adoption agencyアルゴリズム）は実装しない。 | elidex-browserモード。マークアップエラーを含む実際のWebサイトを処理。エラー回復のプラグマティックなサブセット。 |

tolerantパーサーは、クローラーサーベイデータで判明した実際にモダンWebで必要とされるエラー回復パターンのみを実装する。これはすべての歴史的パターンに対する完全なエラー回復を要求するHTML5仕様からの意図的な逸脱である。

> **Phase 0 サーベイ結果（第29章 §29.2）:** サーベイで検出されたエラーカテゴリ（構造エラー ~48%、重複属性、予期しないトークン等）はすべてhtml5everの仕様準拠エラー回復で自動処理可能。回復不能エラーは確認されなかった。

## 11.2 CSSパーサー

CSS解析はcssparserクレートを基盤とし、プロパティ解析は登録されたCssPropertyHandlerプラグインに委譲される。不明なプロパティはサイレントにドロップされる（標準CSS動作）。elidex-compat-cssが処理するレガシープロパティは、プラグインレジストリに到達する前に標準の同等物に変換される。

## 11.3 段階的デグラデーションモデル

elidexの根本的な設計哲学は、壊れたHTMLにはコストがあるべきということである。現在のWebエコシステムは不正マークアップにペナルティゼロを課している——すべての主要ブラウザがすべてをサイレントに修正し、作者が正しいHTMLを書くインセンティブを排除している。Elidexは正確さが速度で報われる自然なパフォーマンス勾配を導入する：

| 入力品質 | レイテンシ | ハンドラ | 動作 |
| --- | --- | --- | --- |
| 有効なHTML5 | μs | elidex-core（直接） | strictパーサーで直接解析。最大パフォーマンス。これがハッピーパス。 |
| 軽微なエラー | < 1ms | ルールベース回復 | 一般的なパターン（未閉鎖タグ、単純なネスト間違い）を決定論的ルールで処理。ほぼ知覚不能なオーバーヘッド。 |
| 著しく壊れている | 100ms–500ms+ | LLMフォールバック | ルールベースパーサーが回復できない。小さなローカルLLMがインテリジェントな修復を試行。ページ読み込みが顕著に遅くなる。ユーザーにパフォーマンス警告を表示。 |
| 回復不能 | — | エラー表示 | LLM修復も失敗。崩れたレイアウトではなく、診断付きの正直なエラーページを表示。 |

> **Phase 0 サーベイ結果（第29章 §29.6）:** 900サイトのサーベイで上記Tier 3（LLMフォールバック）およびTier 4（回復不能）を必要とするケースはゼロであった。すべてのパーサーエラーはTier 2（ルールベース回復）以下で処理可能。

この勾配はWebエコシステムに正のフィードバックループを生む。elidexでテストするサイト作者は、より速いロード時間を得るために自然にクリーンなHTMLを書くようになる。TypeScriptの型システムがJavaScriptの柔軟性を禁止せずに開発者を正確さに向かわせるのと同様。

## 11.4 LLM支援エラー回復

LLMフォールバック層は、従来のブラウザパーサーにおける数百の手書きエラー回復ルールを、壊れたマークアップを推論できる小さな言語モデルに置き換える新しいアプローチである。

### 11.4.1 ランタイムアーキテクチャ

LLMはルールベースのtolerantパーサーが解決できないエラーに遭遇した場合のみ呼び出される。整形式ページではオーバーヘッドゼロを保証する：

```rust
pub fn parse_tolerant(input: &str) -> ParseResult {
    // ステージ1：ルールベース回復を試行（高速、決定論的）
    match rule_based_parse(input) {
        Ok(dom) => ParseResult::Clean(dom),
        Err(partial) if partial.is_recoverable() => {
            ParseResult::Recovered(partial.dom, partial.warnings)
        }
        Err(broken) => {
            // ステージ2：LLMフォールバック（低速、ベストエフォート）
            match llm_repair(input, &broken.diagnostics) {
                Ok(repaired_html) => {
                    let dom = strict_parse(&repaired_html);
                    ParseResult::LlmRepaired(dom, broken.diagnostics)
                }
                Err(_) => ParseResult::Failed(broken.diagnostics)
            }
        }
    }
}
```

ランタイムLLMの主要な設計制約：

**ローカル推論のみ：** モデルは軽量推論ランタイム（例：llama.cpp/candle）経由でローカル実行される。外部APIへのネットワーク呼び出しなし。プライバシーとオフライン機能は譲歩できない。

**小型モデル：** 1〜3Bパラメータ範囲のモデル。量子化（Q4/Q5）で1GB未満のメモリフットプリントを目標。タスク（HTML構造修復）が十分に狭いため、小さなファインチューニングモデルで対応可能。

**決定論的後処理：** LLMの出力はstrictパーサーで再解析される。修復されたHTMLがまだ解析できない場合、修復はリジェクトされる。LLMは直接DOMを生成しない——厳格なバリデーションを通過する候補HTMLのみを生成する。

**ユーザー可視インジケーター：** LLM修復がトリガーされた場合、可視インジケーター（例：アドレスバーの琥珀色アイコン）がAI支援回復が必要であり意図通りにレンダリングされない可能性があることをユーザーに通知する。

### 11.4.2 オフラインルール生成

ランタイム修復に加え、LLMは新しいルールベース回復パターンの生成で開発パイプラインにも役割を果たす：

```
[オフラインパイプライン - 定期的に実行]

elidex-crawler結果（壊れたHTMLコーパス）
  │
  ▼  LLM分析
パターン分類 + 修復ルール提案
  │
  ▼  人間によるレビュー
承認されたルールをelidex-parser-tolerantにマージ
  │
  ▼  次回リリース
ランタイムでLLMフォールバックが必要なページが減少
```

これは好循環を生む：LLMがランタイムで新規エラーを処理し、そのエラーパターンがテレメトリから収集され、LLMが最も一般的なパターンに対する決定論的ルールを提案し、承認されたルールが高速なルールベースパーサーに昇格する。時間とともに、ルールベースパーサーがLLMが発見したパターンを学習するにつれ、LLMフォールバックの呼び出し頻度は減少する。

### 11.4.3 エコシステムインセンティブ設計

段階的デグラデーションモデルは明確なエコシステム目標に貢献する：より良いHTMLオーサリングのインセンティブ付け。パフォーマンス勾配はアクセシビリティを損なわず正確さへの自然な圧力を提供する：

**サイト作者向け：** クリーンなHTML5のサイトはelidexで測定可能に高速。パフォーマンスメトリクス（Lighthouse相当のスコアリング）で可視化され、ユーザーに直接知覚される。

**ユーザー向け：** 完全に壊れるサイトはない。著しく不正なページでもベストエフォートでレンダリングされる。琥珀色インジケーターがなぜページが遅いかの透明性を提供する。

**エコシステム向け：** レンダリング拒否（普及の障害となる）とは異なり、コストは問題の深刻度に比例する。検索エンジンがページ速度をランキングシグナルとして使用するのと同様——壁ではなく穏やかな誘導。

### 11.4.4 アプリモード：LLM駆動開発者診断

elidex-appモードでは、strictパーサーが開発時に不正HTMLをリジェクトする。ここではLLMは異なる役割を果たす：リッチでコンテキスト認識のあるエラーメッセージの生成。

```
error[E0042]: 未閉鎖の要素
  --> ui/main.html:32:5
   |
32 |     <div class="container">
   |     ^^^^^^^^^^^^^^^^^^^^^ この<div>は閉じられていません
   |
   = help: 45行目の前に</div>を追加することを検討してください：
   |
44 |         </ul>
45 | +     </div>
46 |     </section>
   |
   = note: LLMがドキュメント構造から推論しました。
           この<div>はおそらく34行目から始まる<ul>をラップしています。
```

これは他のWebアプリケーションランタイムが提供しないElm/Rustコンパイラのような開発者体験を実現する。LLMの推論能力によりルールベースヒューリスティクスよりも大幅に優れた提案が可能であり、開発時のみ実行されるためレイテンシは問題にならない。

### 11.4.5 条件付き採用：データ駆動型意思決定

LLMフォールバック層は大きなエンジニアリング投資（モデルファインチューニング、推論ランタイム統合、メモリオーバーヘッド）を伴う。そのため、採用はサーベイデータに条件付けされる。意思決定プロセス：

```
Phase 0:   900サイト（JA 451 + EN 449） × トップページ
           → レガシー機能の普及率を計測
           → elidex-coreが省略できるものを決定

Phase 0.5: 拡張クロール（3,000〜5,000サイト × 5〜10サブページ）
           → 全ページをhtml5everで解析、エラーイベントを記録
           → 分類：ルール回復可能 vs 回復不能エラー
           → 計測：回復不能エラーのあるページの割合は？

判定ゲート：
  回復不能エラー率 < 約2%  → ルールベースのみ（LLMは投資対効果なし）
  回復不能エラー率 ≥ 約2%  → LLMフォールバックに投資
```

> **Phase 0 サーベイ結果（第29章）:**
> - パーサーエラー検出率: JA ~48% / EN ~44%（サイト単位）
> - 回復不能エラー率: **0%**（全エラーがhtml5everで自動回復）
> - 非推奨タグ使用率: JA 4.4% / EN 2.4%（< 5%）
> - Presentational hints（width/height）: 60%以上のサイトで使用
> - 判定ゲート結果: 0% < 2% → **No-Go 確定**（Phase 0.5スキップ）

Phase 0.5はPhase 0の結果分析後にのみトリガーされる。初期サーベイでモダンWebが圧倒的に整形式HTML5であることが判明すれば、LLM層は不要かもしれない——不必要な複雑性の回避はelidex哲学と合致する。

> **Phase 0 サーベイ結果（第29章 §29.6）:** サーベイにより「モダンWebは圧倒的に整形式HTML5」であることが確認された。LLMランタイムフォールバック層は不要と判断。

ランタイムの判断に関係なく、elidex-appモード向けのLLM駆動開発者診断（11.4.4節）は独立して価値があり、サーベイデータを待たずに進行すべきである。

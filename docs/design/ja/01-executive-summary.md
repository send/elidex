
# 1. エグゼクティブサマリー

## 1.1 Elidexとは

Elidexはレガシー後方互換性を意図的に排除して最大パフォーマンスを達成するRust製実験的ブラウザエンジン。名前は"elide"（省略する）に由来し、モダンブラウザエンジンを遅く肥大化させている蓄積された技術的負債の除去という核心哲学を反映。

プロジェクトはデュアルパーパス：モダンWebサイトを処理する**Webブラウザ**と、ElectronやTauriと競合する**軽量アプリケーションランタイム**（elidex-app）。レガシーサポートを組み込みではなくプラガブルにすることで、クリーンで高速なコアを維持しつつ実世界のWebコンテンツにも対応。

## 1.2 コア設計原則

**HTML5のみのコア。** レンダリングエンジンは有効なHTML5のみを受理。レガシータグ処理、Quirksモード、エラー回復はすべてオプショナルな互換プラグインとしてコア外に存在。

**三層一貫性。** HTML、DOM API、CSSOM、ECMAScript、Web APIすべてが同一パターン：クリーンなコア＋オプショナルCompatプラグインによるレガシー処理。各層がスペックレベル（core / compat / deprecated）を宣言。

**すべてがプラガブル。** HTMLタグハンドラ、CSSプロパティ、レイアウトアルゴリズム、ネットワークミドルウェア、DOM APIメソッド、JS言語機能のすべてがコンパイル時にstatic dispatchで解決される離散プラグイン。

**非推奨化を前提に設計。** 自動Webサーベイの使用データに基づき、旧式機能の定期的除去をアーキテクチャが明示的にサポート。

**データ指向設計。** DOM内部はキャッシュ効率と並列性のためにECS（Entity Component System）を使用。ゲームエンジンアーキテクチャ（Bevy）に着想。

**セッション仲介スクリプティング。** 統一ScriptSession層がオブジェクト指向スクリプト世界とデータ指向ECS世界を仲介。データベースORMのUnit of Workパターンに類似。

**デフォルトで並列。** スタイル解決とレイアウトはRustの所有権モデルで並列化。Servoの実証済みアプローチに倣う。

## 1.3 アーキテクチャ概観

マルチプロセスアーキテクチャ：Browser Process（信頼済み、状態管理）とサンドボックス化されたRenderer Process（サイトごとに1つ、非信頼）。Browser Processではtokio非同期ランタイムがI/Oを駆動、Renderer Processは決定論的イベントループを使用。rayonスレッドプールが並列CPU作業（スタイル、レイアウト）を処理。

```
┌─────────────────────────────────────────────────────────┐
│ Browser Process（信頼済み）                              │
│  NavigationController · PermissionManager · BlobStore   │
│  CookieStore · StorageManager                           │
└──────────────────┬──────────────────────────────────────┘
                   │ IPC (serde + 共有メモリ)
     ┌─────────┬───┼───────┬─────────────┐
     ▼         ▼   ▼       ▼             ▼
┌────────┐ ┌────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐
│Renderer│ │Renderer│ │ Network │ │  GPU    │ │ Utility │
│Process │ │Process │ │ Process │ │ Process │ │ Process │
│(サイトA)│ │(サイトB)│ │         │ │ (Vello) │ │(メディア)│
└────────┘ └────────┘ └─────────┘ └─────────┘ └─────────┘
```

各Renderer Process内：

```
HTML → パーサー → ECS DOM → スタイル → レイアウト → ペイント(DisplayList) → コンポジター → GPU (Vello/wgpu)
                    ↑                                                         ↑
              ScriptSession                                            アニメーション/スクロール
             (SpiderMonkey /                                          (コンポジタースレッド、
              elidex-js)                                               オフメインスレッド)
```

## 1.4 主要技術選択

| 決定 | 選択 | 根拠 |
| --- | --- | --- |
| 言語 | Rust | メモリ安全性、fearless concurrency、所有権モデルが並列レイアウトを可能に |
| DOMストレージ | ECS (hecs) | キャッシュライン効率、自然な並列性、ツリーオブオブジェクトより~40%少ないメモリ |
| GPUレンダリング | Vello + wgpu | モダン2Dベクター GPUレンダラー。Trait抽象化 — 上流コードにVello依存なし |
| JSエンジン | SpiderMonkey (Phase 1–3)、自作elidex-js (Phase 4+) | ブートストラップに実証済みエンジン、長期的にRustネイティブエンジン |
| CSSパース | cssparser + lightningcss | 実証済みMozillaクレート、プラグインプロパティレジストリで拡張 |
| フォントシェーピング | rustybuzz | 純粋RustのHarfBuzzポート、ファーストクラスCJK・BiDi |
| アクセシビリティ | AccessKit | クロスプラットフォームa11y (NSAccessibility, UI Automation, AT-SPI2) |
| メディアコーデック | ロイヤリティフリーはRust/C、特許負担はプラットフォーム | 特許コスト回避、H.264/AACはOSデコーダー経由 |

## 1.5 ドキュメント構成

本設計ドキュメントは8パート・28章で構成：

**Part I — 概要**（Ch. 1–4）：エグゼクティブサマリー、アーキテクチャ概要、ロードマップ、リスク分析。

**Part II — コアアーキテクチャ**（Ch. 5–8）：プロセスモデルと非同期ランタイム、スレッドモデル、プラグインシステム（core/compatパターン）、セキュリティモデルとパーミッション。

**Part III — コンテンツパイプライン**（Ch. 9–14）：ナビゲーションとページライフサイクル、ネットワークアーキテクチャ、HTMLパーサー、DOMとCSSOM、ScriptSession、スクリプトエンジンとWeb API。

**Part IV — レンダリング**（Ch. 15–19）：レンダリングパイプライン（GPU、コンポジター、レイヤー）、テキスト＆フォントパイプライン、アニメーション＆スクロール、画像デコードパイプライン、SVGレンダリング。

**Part V — メディア＆データ**（Ch. 20–22）：メディアパイプライン（音声/映像、Web Audio、MSE/EME）、File API＆ストリーム、ストレージ＆キャッシュ。

**Part VI — プラットフォーム＆UI**（Ch. 23–25）：プラットフォーム抽象化層、ブラウザシェル、アクセシビリティ。

**Part VII — elidex-app**（Ch. 26）：エンベディングAPIとデュアルユース設計（ブラウザ vs. アプリランタイム）。

**Part VIII — 品質＆付録**（Ch. 27–28）：テスト戦略、アーキテクチャ決定記録（35 ADR）。


# 28. Key Design Decisions

This section records architectural decisions and their rationale for future reference.

| # | Decision | Choice | Rationale |
| --- | --- | --- | --- |
| 1 | Implementation language | Rust | Memory safety, fearless concurrency, ownership model enables parallel layout |
| 2 | Scripting strategy | Three-layer design: DOM API (pluggable, Living Standard + compat), ECMAScript (Boa → self-built Rust engine, ES2020+ core / Annex B compat), Wasm (wasmtime for multi-language elidex-app) | Consistent core+compat pattern across HTML/DOM/JS. Self-built engine removes long-term dependency on third-party JS runtimes. Wasm enables multi-language apps. |
| 3 | DOM storage model | ECS (Entity Component System) | Cache efficiency, natural parallelism, inspired by game engines (Bevy) |
| 4 | Legacy handling strategy | Pre-core normalization (transpiler pattern) | Keeps core clean, avoids quirks mode code paths infecting the engine |
| 5 | Feature extensibility | Dual dispatch: static enum dispatch (compile-time, zero-cost) + dynamic trait objects (runtime extensions), unified under same traits | Built-in features pay no overhead; extensions use same API. Deprecated core features can survive as community dynamic plugins |
| 6 | GPU rendering | Vello on wgpu (see ADR #26) | Rust-native, cross-platform, avoids Skia C++ dependency |
| 7 | Process architecture | Multi-process (Chromium-inspired, Ladybird-clean) | Security isolation, clean IPC boundaries match crate boundaries |
| 8 | Writing mode | Abstract inline/block coordinates from day one | Required for CJK vertical text, cannot be retrofitted |
| 9 | Dual-use architecture | elidex-browser + elidex-app from shared core | App runtime (Electron/Tauri alternative) is a major value proposition |
| 10 | Deprecation mechanism | Data-driven via periodic crawler surveys | Objective, quantitative basis for removing features; avoids subjective debate |
| 11 | Error recovery strategy | Graduated degradation: strict → rule-based → error (LLM layer paused). | Correctness = speed. Broken HTML incurs proportional latency cost. **Phase 0 Result:** Survey demonstrated that rule-based recovery is sufficient ([Ch. 29](29-survey-analysis.md)). **Project decision (2026-03-05):** LLM-related scope is paused for now. |
| 12 | LLM fallback adoption | Conditional on Phase 0.5 survey (≥2% unrecoverable error rate), but currently paused | Avoid unnecessary complexity if broken HTML is rare. **Phase 0 Result:** Unrecoverable errors 0% (900 sites) — No-Go confirmed ([Ch. 29](29-survey-analysis.md)). **Project decision (2026-03-05):** LLM-related scope, including developer diagnostics, is paused. |
| 13 | Network layer design | Pluggable ResourceLoader + middleware pipeline; engine-neutral on content blocking | Enables custom URL schemes for app mode, DevTools, ad blocking, and future protocol evolution without modifying core. |
| 14 | DOM API architecture | DomApiHandler plugin trait with DomSpecLevel (Living/Legacy/Deprecated). Each method individually toggleable. ECS integration via component ops. | Same deprecation model as HTML/CSS. elidex-app excludes legacy DOM at compile time. MutationObserver maps naturally to ECS change detection. |
| 15 | CSS spec level classification | CssSpecLevel enum (Standard/Aliased/NonStandard/Deprecated). Property+value level, not usage-level. | Aligns CSS with HTML/DOM/ES three-tier pattern. Usage-level detection (e.g., float for layout) is impractical; property deprecation driven by crawler data instead. |
| 16 | Web API core/compat split | Per-API classification via WebApiSpecLevel (Modern/Legacy/Deprecated). Sync APIs (localStorage, XHR, document.cookie) are compat; async equivalents are core. | Sync APIs block main thread and require blocking IPC in multi-process architecture. elidex-app excludes sync APIs at compile time. Async storage API (elidex.storage) replaces localStorage for app developers. |
| 17 | ScriptSession layer | Unified session mediating all Script ↔ ECS interactions. Provides Identity Map, Mutation Buffer, GC coordination, live query management for all Object Model APIs (DOM, CSSOM, future OMs). | Eliminates ad-hoc impedance mismatch solutions per OM layer. Analogous to ORM Unit of Work pattern. Ensures atomic flush of mutations, consistent MutationObserver records, and object identity guarantees across all OM APIs through a single mechanism. |
| 18 | Three-tier pluggability model | Engine layer: spec-driven (trim legacy). Platform layer: OS API + trait contract (swap implementations). Browser shell: trait contract only (extend/replace UI). | Web standards provide clear core/compat boundaries for the engine, but browser shell has no authoritative spec. Trait contracts replace SpecLevel as the plugin basis for platform and shell layers. |
| 19 | Browser chrome strategy | Phased approach: native chrome (egui/iced) for Phase 1-2, self-hosted option (HTML/CSS) from Phase 3+. Both coexist long-term, selectable at build time. | Engine must be functional before self-hosting is possible. Native chrome provides fast iteration during early development. Self-hosted chrome enables dogfooding and maximum customization. |
| 20 | Process model: staged relaxation | Site-based isolation in Phase 1–3 (Boa era), relaxable to crash-isolation-only after full Rust migration. ProcessModel enum configurable at build/startup. IPC abstracted behind trait (ProcessChannel / LocalChannel). | During bootstrap, Boa integration and host bindings are still being hardened for browser threat models, so stricter isolation limits blast radius. After elidex-js migration and hardening, site isolation can be made optional by policy. Trait abstraction ensures relaxation is configuration change, not rewrite. Same core/compat pattern as engine layer. |
| 21 | Async runtime: hybrid approach | Renderer: custom elidex event loop + tokio reactor as I/O backend. Network/Browser: tokio multi-thread. AsyncRuntime trait for future replacement. | Browser event loop needs precise control (vsync, rAF, frame budget) that tokio's scheduler doesn't provide. tokio is optimal for I/O-heavy processes (Network). Trait abstraction preserves option to replace tokio with self-built runtime long-term, mirroring Boa → elidex-js migration path. |
| 22 | Storage backend: SQLite behind trait | SQLite (rusqlite) as initial StorageBackend implementation, abstracted behind StorageBackend/StorageConnection traits. StorageOp enum instead of raw SQL in trait boundary. | SQLite is proven in every major browser (Chrome, Firefox, Safari). IndexedDB requires relational operations (indexes, cursors, range queries) that KV stores cannot efficiently provide. Native dependency footprint is acceptable for storage, and trait abstraction enables future replacement without affecting domain traits (CookiePersistence, HistoryStore, etc.). |
| 23 | Origin isolation: per-origin database files | Web-content storage uses physically separate SQLite databases per origin. Browser-owned data uses centralized browser.sqlite. | Structural guarantee against cross-origin data leakage — query bugs cannot expose cross-origin data because data lives in separate files. Matches Chrome/Firefox approach for IndexedDB, localStorage, Cache API. OriginStorageManager mediates access; Renderers never touch files directly (IPC to Browser Process with origin verification). |
| 24 | Compositor thread: Approach B with B→C migration path | Phase 1–3: Servo-style message passing (DisplayList ownership transfer to compositor). Phase 4+: optional migration to shared ECS reads (Approach C). FrameSource trait abstracts the boundary. | Approach B (message passing) aligns with Rust ownership, has Servo precedent, works across process boundaries (GPU Process separation). Approach C (shared ECS) eliminates serialization cost but requires same-process — naturally aligns with process isolation relaxation. FrameSource trait makes B→C a configuration change, not rewrite. Same pattern as IpcChannel (ProcessChannel/LocalChannel). |
| 25 | Web Workers: 1:1 OS thread mapping | Each Dedicated/Shared/Service Worker gets its own OS thread with own ScriptEngine and ScriptSession. Not pooled. | Workers can run indefinitely (game loops, real-time processing). Thread pool would deadlock if all slots occupied by long-lived Workers. SharedArrayBuffer requires true OS threads for Atomics.wait/notify (futex semantics). 1:1 mapping is what all existing browsers use. |
| 26 | Vello: direct dependency (no trait abstraction) | Vello used directly without abstraction trait. Contact surface isolated to DisplayList→Vello Scene conversion layer only. | Exception to the trait abstraction pattern used for hyper (HttpTransport) and SQLite (StorageBackend). No viable Rust-native GPU 2D renderer alternative exists — a trait would replicate Vello's scene API with no practical benefit. Unlike HTTP clients or storage engines where multiple candidates justify abstraction, abstracting for non-existent alternatives is over-engineering. All upstream code (ECS, style, layout, paint, layer tree, display list) has zero Vello dependency. If a credible alternative emerges, only the conversion layer needs replacement. |
| 27 | Independent LayerTree structure (not ECS components) | Layer tree is a standalone data structure, separate from ECS DOM. PaintSystem reads ECS and constructs LayerTree as output. | DOM nodes and layers have N:M relationships: multiple DOM nodes squash into one layer, one node can produce multiple layers (scrollable container + content). This does not fit ECS's 1 entity : 1 component model. Independent LayerTree also enables Approach C (Ch. 6): compositor reads Arc<RwLock<LayerTree>> without touching DOM. Matches Chromium's separate cc::LayerTree approach. |
| 28 | Frame scheduling: FramePolicy enum | Four policies: Vsync (browser default, skip if idle), Continuous (every vsync, games), OnDemand (dirty-flag only, elidex-app default), FixedRate(n) (timer-driven). | Browser and app runtimes have fundamentally different frame rate requirements. Browser tabs need vsync sync; document-style apps waste power rendering at 60fps when idle; games need maximum throughput; signage needs fixed rates. Single Duration-based budget cannot express these. FramePolicy controls compositor wake behavior per-tab (browser) or per-app (elidex-app), unifying all use cases in one scheduler. |
| 29 | Unified animation model: Web Animations as internal representation | CSS Transitions and CSS Animations are translated to Web Animations instances at creation time. One sampling engine, one timing model, one composition system. | Web Animations spec was designed as the unified model for CSS Transitions/Animations. Maintaining three separate engines would mean triple the code for interpolation, timing, composition, and compositor promotion. Unified model ensures `element.getAnimations()` naturally returns all animations regardless of origin. Matches Chrome's architecture (Blink unified CSS/WAAPI animation model). |
| 30 | Animation timing: FrameProducer with ECS-query AnimationEngine (C+B hybrid) | FrameProducer coordinates between event loop and rendering pipeline. AnimationEngine stores animation state as ECS components (ActiveAnimations) and uses parallel ECS queries for tick. | Pure delegation (C) provides clean responsibility separation: event loop decides WHEN, FrameProducer decides WHAT. ECS-query internals (B) enable rayon parallel sampling+writing (each entity independent), automatic lifecycle management (despawn removes animations), and loose coupling between StyleSystem and AnimationEngine (both read/write ECS, neither references the other). Hybrid gets benefits of both without drawbacks of either. |
| 31 | Image decoders: trait-abstracted, Rust-crate default | ImageDecoder trait with Rust-crate implementations (zune-jpeg, png, webp, ravif, gif). Platform-native decoders (ImageIO, WIC) substitutable via same trait. DecoderStrategy enum (RustOnly / PlatformPreferred / Mixed). | Pure Rust maximizes portability and memory safety, aligning with long-term all-Rust goal. However, platform decoders offer hardware acceleration (especially JPEG/HEIF on Apple Silicon, AVIF via dav1d). Trait abstraction lets elidex start with Rust crates and swap in platform decoders where profiling shows need — same pattern as HttpTransport (Ch. 10) and StorageBackend (Ch. 22). |
| 32 | Inline SVG as ECS entities, SVG-as-image via direct Vello | Inline SVG elements stored in same ECS DOM as HTML (SvgGeometry, SvgTransform components). SVG-as-image bypasses ECS — parsed to lightweight scene, rendered directly via Vello to offscreen texture. | Inline SVG must participate in DOM API (querySelector, MutationObserver, CSS cascade) — ECS is the only consistent way. SVG-as-image has no DOM access per spec (no JS, no external resources), so ECS overhead is unnecessary; direct Vello rendering is simpler and faster. DisplayItem enum extended with SvgPath/SvgText for unified paint pipeline — Vello's 2D vector primitives are a natural match for SVG paths. |
| 33 | OPFS SyncAccessHandle via shared memory mapped file | SyncAccessHandle provides synchronous read/write by mapping OPFS files into shared memory between Browser Process and Worker. Read/write are direct memcpy (no IPC). Only control operations (truncate, flush, close) require IPC. Exclusive per-file lock enforced by Browser Process. | SQLite-on-web requires thousands of small read/write operations per second. IPC round-trip per operation would be prohibitively slow (~1μs+ per call). Shared memory eliminates this for the hot path while maintaining the security model (Browser Process controls file creation, locking, and metadata). Matches Chromium's approach to SyncAccessHandle. |
| 34 | Media codecs: Rust/C for royalty-free, platform for patent-encumbered | Royalty-free codecs (VP8/VP9/AV1, Opus, Vorbis, FLAC, MP3) use Rust/C library decoders (dav1d, libvpx, lewton, minimp3). Patent-encumbered codecs (H.264, H.265, AAC) delegate to platform decoders (AVFoundation, MediaFoundation, VA-API). MediaDecoder trait abstracts both paths. Decoder Process provides sandboxed isolation. | Same trait-abstraction pattern as ImageDecoder (ADR #31). Avoids patent licensing costs for core distribution while supporting the dominant web video codecs through OS-provided decoders. Process isolation contains codec library bugs. Hardware-accelerated decode via platform APIs yields zero-copy GPU texture path for optimal power efficiency. |
| 35 | bfcache via Renderer Process freeze with eviction | Back/forward cache preserves pages by freezing the entire Renderer Process (suspend event loop, timers, network). Frozen processes are held up to 6 entries or 512 MB total. Eviction destroys oldest frozen process. Pages with open WebSocket, WebRTC, unload listeners, or active IndexedDB transactions are ineligible. | Process-level freeze is the only reliable way to preserve full page state (DOM, JS heap, active connections, Web Audio graph). ECS serialization would be impractical for complex state like WebSocket connections. Matches Chromium's approach. Memory-bounded eviction prevents unbounded memory growth. |
| 36 | MSRV policy: phased adoption | During development: no MSRV (latest stable only). Post-stabilization: adopt N-2 (current stable minus 2 releases). `rust-version` field and MSRV CI job added after stabilization. | elidex is an application, not a library — no external crate consumers. Cost of maintaining MSRV during development (clippy workarounds, tracking dependency MSRV bumps) does not justify the return. Heavy dependencies (wasmtime, boa, egui) frequently bump their own MSRV, making a fixed MSRV effectively non-functional. Post-stabilization, N-2 ensures builds with Debian stable/Ubuntu LTS packaged rustc. |

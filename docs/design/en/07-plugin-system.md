
# 7. Plugin System

## 7.1 Unified Plugin Framework

Every feature in elidex—HTML tag handling, CSS properties, layout algorithms, network middleware, JS API bindings—is implemented as a plugin conforming to a common trait. The key design challenge is supporting two dispatch modes through a single trait definition:

| Mode | Mechanism | Use Case |
| --- | --- | --- |
| Static | Enum dispatch, resolved at compile time. The compiler inlines match arms and eliminates vtable indirection. Zero runtime cost. | Built-in features selected via Cargo feature flags. Core HTML5 tags, CSS properties, layout algorithms. This is the hot path where nanoseconds matter. |
| Dynamic | Trait object dispatch (Box<dyn Trait>), registered at runtime. Small overhead per call (~few ns vtable lookup). | End-user extensions: ad blockers, DevTools middleware, custom CSS properties, experimental features. Added without recompiling the engine. |

Plugin authors write their implementation against the same trait regardless of dispatch mode. The only difference is how the plugin is registered: statically (via Cargo feature) or dynamically (via extension API).

## 7.2 Plugin Traits

Core trait definitions used across all layers:

```rust
pub trait CssPropertyHandler: Send + Sync {
    fn property_name(&self) -> &str;
    fn spec_level(&self) -> SpecLevel;
    fn parse(&self, value: &str) -> Result<CssValue, ParseError>;
    fn resolve(&self, value: &CssValue, ctx: &StyleContext) -> ComputedValue;
    fn affects_layout(&self) -> bool;
    fn deprecated_by(&self) -> Option<DeprecationInfo> { None }
}
pub trait HtmlElementHandler: Send + Sync {
    fn tag_name(&self) -> &str;
    fn spec_level(&self) -> SpecLevel;
    fn default_style(&self) -> &[CssRule];
    fn create_element(&self, attrs: &Attributes) -> ElementData;
    fn parse_behavior(&self) -> ParseBehavior { ParseBehavior::Normal }
    fn accessibility_role(&self) -> Option<AccessibilityRole> { None }
}
pub trait LayoutModel: Send + Sync {
    fn name(&self) -> &str;
    fn spec_level(&self) -> SpecLevel;
    fn layout(&self, node: &LayoutNode, children: &[LayoutNode],
              constraints: &Constraints) -> LayoutResult;
}
pub trait NetworkMiddleware: Send + Sync {
    fn name(&self) -> &str;
    fn on_request(&self, req: &mut Request) -> MiddlewareAction { MiddlewareAction::Continue }
    fn on_response(&self, req: &Request, res: &mut Response) -> MiddlewareAction { MiddlewareAction::Continue }
}
```

## 7.3 Dual Dispatch Architecture

The same trait is dispatched via two mechanisms that compose into a single pipeline:

### 7.3.1 Static Dispatch (Compile-Time)

Built-in plugins are collected into an enum whose variants are gated by Cargo feature flags. The compiler optimizes the match into a jump table, eliminating all indirection:

```rust
// Auto-generated by #[elidex_plugin] proc macro
enum CssPropertyDispatch {
    Display(DisplayHandler),
    Margin(MarginHandler),
    Padding(PaddingHandler),
    #[cfg(feature = "css-flexbox")]
    FlexDirection(FlexDirectionHandler),
    #[cfg(feature = "css-grid")]
    GridTemplateColumns(GridTemplateColumnsHandler),
    // ... expands/contracts with feature flags
}

// Compiler inlines each arm → zero-cost dispatch
impl CssPropertyHandler for CssPropertyDispatch {
    fn parse(&self, value: &str) -> Result<CssValue, ParseError> {
        match self {
            Self::Display(h) => h.parse(value),
            Self::Margin(h) => h.parse(value),
            Self::Padding(h) => h.parse(value),
            #[cfg(feature = "css-flexbox")]
            Self::FlexDirection(h) => h.parse(value),
            ...
        }
    }
}
```

### 7.3.2 Dynamic Dispatch (Runtime)

Extensions and user-provided plugins implement the same trait but are registered at runtime as trait objects:

```rust
// Extension author writes the same trait implementation
struct MyCustomProperty { ... }
impl CssPropertyHandler for MyCustomProperty { ... }

// Registered dynamically (no recompilation needed)
browser.register_extension(Box::new(MyCustomProperty::new()));
```

### 7.3.3 Unified Pipeline Execution

At runtime, the registry resolves lookups by checking the static dispatch table first, then falling through to dynamic extensions:

```rust
pub struct PluginRegistry<T: ?Sized> {
    static_lookup: HashMap<&'static str, StaticDispatch>,  // compile-time
    dynamic_lookup: HashMap<String, Box<dyn T>>,           // runtime
}

impl<T: ?Sized> PluginRegistry<T> {
    pub fn resolve(&self, name: &str) -> Option<&dyn T> {
        // Static first (zero-cost path for built-in features)
        if let Some(handler) = self.static_lookup.get(name) {
            return Some(handler.as_ref());
        }
        // Dynamic fallback (extension path)
        self.dynamic_lookup.get(name).map(\|b\| b.as_ref())
    }
}
```

Execution flow for every pipeline stage:

```
Input
  │
  │  Static stages (enum dispatch, inlined by compiler)
  │  ┌─ CssPropertyDispatch::Display      ← zero cost
  │  ├─ CssPropertyDispatch::Margin       ← zero cost
  │  ├─ CssPropertyDispatch::FlexDir...   ← zero cost
  │  └─ ...
  │
  │  Dynamic stages (trait object, vtable lookup)
  │  ┌─ Box<dyn CssPropertyHandler>       ← ~few ns overhead
  │  └─ Box<dyn CssPropertyHandler>       ← ~few ns overhead
  │
  ▼
Output
```

## 7.4 Proc Macro for Plugin Authors

To eliminate the boilerplate of enum dispatch generation, the elidex-plugin crate provides a proc macro that auto-generates the static dispatch enum, the match implementation, and the registry wiring:

```rust
#[elidex_plugin(dispatch = "static")]
pub trait CssPropertyHandler: Send + Sync {
    fn property_name(&self) -> &str;
    fn parse(&self, value: &str) -> Result<CssValue, ParseError>;
    fn resolve(&self, value: &CssValue, ctx: &StyleContext) -> ComputedValue;
}

// The macro generates:
// 1. CssPropertyHandlerDispatch enum (with #[cfg(feature)] gates)
// 2. impl CssPropertyHandler for CssPropertyHandlerDispatch (match arms)
// 3. PluginRegistry<dyn CssPropertyHandler> integration
```

Plugin authors never interact with the dispatch mechanism directly. They implement a trait; the framework handles the rest.

## 7.5 Applicability Across All Layers

The dual-dispatch pattern applies uniformly to every extensible layer in elidex:

| Layer | Static (Compile-Time) | Dynamic (Runtime Extension) |
| --- | --- | --- |
| HTML Tags | Built-in HTML5 element handlers (div, span, a, img, ...) | Custom Elements, experimental tags |
| CSS Properties | Standard CSS properties (display, margin, flex, grid, ...) | User-defined custom properties, experimental CSS |
| Layout Algorithms | Block, Flex, Grid, Table layout | Experimental layout models |
| Network | HTTP loader, cache, file:// loader | Ad blockers, DevTools logger, privacy filters, API mocking |
| DOM API | Living Standard methods (querySelector, MutationObserver, ...) | Legacy DOM shims (live collections, document.write, attachEvent) |
| ECMAScript | ES2020+ core (let/const, class, async/await, modules) | Annex B semantics, var quirks, sloppy eval |
| Parser Recovery | Rule-based error recovery patterns | LLM fallback (extension point in design, currently paused) |

This uniformity means a single elidex-plugin crate, a single proc macro, and a single mental model for all contributors and extension authors, regardless of which layer they are working on.

## 7.6 Spec Levels and Feature Flags

Each plugin declares its specification level, enabling compile-time and runtime filtering. To maintain the three-layer consistency principle, every extensible layer has its own SpecLevel enum with the same core/compat/deprecated pattern:

```rust
// HTML tags — per-tag classification
pub enum HtmlSpecLevel {
    Html5,          // Living Standard: <div>, <span>, <a>, <img>, <video>, ...
    Legacy,         // Deprecated tags: <center>, <font>, <marquee>, ...
    Deprecated,     // Scheduled for removal (usage < 1%)
}

// DOM API — per-method classification (defined in Chapter 12)
pub enum DomSpecLevel {
    Living,         // DOM Living Standard: querySelector, addEventListener, ...
    Legacy,         // Legacy: getElementsByClassName (live), attachEvent, ...
    Deprecated,     // Dangerous: document.write, document.all, ...
}

// ECMAScript — per-feature classification (defined in Chapter 14)
pub enum EsSpecLevel {
    Modern,          // ES2020+: let/const, class, async/await, modules, ...
    LegacySemantics, // var hoisting, sloppy mode, ...
    AnnexB,          // HTML comments in JS, __proto__, ...
}

// CSS — per-property/value classification
pub enum CssSpecLevel {
    Standard,     // Standard property with standard value: display: flex, margin, ...
    Aliased,      // Old name for a standard property: word-wrap → overflow-wrap
    NonStandard,  // Vendor-prefixed or non-standard: zoom, -webkit-appearance, ...
    Deprecated,   // Was standard, now scheduled for removal (usage < 1%)
}

// Web API — per-API classification (defined in Chapter 14)
pub enum WebApiSpecLevel {
    Modern,       // Modern async APIs: Fetch, CookieStore, IndexedDB, ...
    Legacy,       // Sync/blocking APIs: XMLHttpRequest, localStorage, document.cookie
    Deprecated,   // Removed from standards: WebSQL, ...
}
```

The CssSpecLevel deserves special attention. Unlike HTML tags where each tag is unambiguously core or legacy, CSS has three distinct categories of non-standard usage:

| CssSpecLevel | Example | Handling |
| --- | --- | --- |
| Standard | `display: flex`, `margin: 10px` | Core. Processed directly by CssPropertyHandler plugins. |
| Aliased | `word-wrap: break-word` | Compat. elidex-compat-css normalizes to `overflow-wrap: break-word` before reaching core. |
| NonStandard | `zoom: 1.5`, `-webkit-appearance: none` | Compat. elidex-compat-css transforms to standard equivalents (`transform: scale(1.5)`, `appearance: none`). |
| Deprecated | Usage-driven (< 1% in crawler data) | Same deprecation lifecycle as all other layers. |

> **Phase 0 Survey Result (Ch. 29 §29.3):** CssSpecLevel classification validated by survey data:
> - `-webkit-appearance`: EN 17.6% (NonStandard — confirms high-priority compat requirement)
> - `word-wrap`: EN 13.8% (Aliased — confirms need for `overflow-wrap` normalization)
> - `-webkit-box-*`: EN 12–14% (Aliased — confirms legacy flexbox syntax compat requirement)

Note that usage-level legacy detection (e.g., `float` used for page layout vs. text wrapping) is deliberately not attempted. Distinguishing semantic intent from syntax alone is impractical. Instead, property-level deprecation is driven by crawler usage data through the standard deprecation policy (Section 7.7).

Cargo feature flags control which plugins are compiled into the static dispatch enum:

```
[features]
default = []                          # HTML5-only, minimal
level-2026 = ["html5-base", "css-flexbox", "css-grid", ...] 
level-2028 = ["level-2026", "css-anchor-positioning", ...]
browser-minimal = ["level-2026", "elidex-parser-tolerant"]
browser-full = ["browser-minimal", "elidex-compat-tags", ...]
```

Features not selected at compile time are excluded from the static dispatch enum entirely—they do not exist in the binary. This is the mechanism by which elidex-app achieves its minimal footprint: unused web features add zero bytes and zero overhead.

## 7.7 Deprecation Policy

The ability to remove features is fundamental to elidex’s long-term health. The deprecation lifecycle is:

| Stage | Trigger | Action |
| --- | --- | --- |
| Active | Web usage > 1% | Fully supported, included in browser-full |
| Deprecated | Web usage drops below 1% (measured by elidex-crawler) | Console warnings in browser mode; compile-time warnings in app mode; excluded from new level-YYYY feature sets |
| Removed | 2 major versions after deprecation | Plugin crate deleted from repository; Cargo feature flag removed. If users still need it, they can maintain it as a dynamic extension. |

The elidex-crawler tool is run periodically against the survey site list to produce quantitative usage data that drives deprecation decisions, eliminating subjective debate.

> **Phase 0 Survey Result (Ch. 29 §29.2):** Survey data validation of the 1% threshold:
> - `<font>`: JA 2.0% (Legacy — remains above threshold)
> - `<center>`: JA 1.6% (Legacy — remains above threshold)
> - `document.all`: 0% (Deprecated — candidate for first removal cycle)
> - `<blink>`: EN 0.2% (Deprecated — below threshold)

Note that removed static plugins can survive as community-maintained dynamic extensions. The dual dispatch model means deprecation from the core does not mean the feature becomes impossible to use—it just moves from the zero-cost static path to the dynamic extension path, which is an appropriate trade-off for rarely-used features.
